## Implementation Summary

- Added `InspectorProjectionService` in `backend/app/services/projections/inspector.py`.
- Added `GET /api/stages/{stageRunId}/inspector` in `backend/app/api/routes/query.py`.
- The route returns the existing strict `StageInspectorProjection` schema with unified `ErrorResponse` handling and OpenAPI coverage.
- The service reads the target `StageRunModel` and owning `PipelineRunModel`, validates that the owning `Session` and `Project` remain visible, and then assembles the Inspector from runtime rows, stage-scoped events, tool confirmation rows, approval rows, and `StageArtifactModel` records.
- Projection sections are grouped as `identity`, `input`, `process`, `output`, `artifacts`, and `metrics`.
- `solution_design` stages expose `SolutionDesignArtifact.implementation_plan` and stable downstream task ids; non-`solution_design` stages do not expose top-level or nested implementation-plan payloads.
- Provider retry, backoff, circuit-breaker status, failure reason, and stable trace refs are preserved in the stage Inspector.
- Tool confirmation process refs are exposed at the stage level without implementing Q3.4a detail projection semantics.
- Raw graph identifiers are scrubbed from artifact-derived content, including both `graph_thread_ref` and `graph_thread_id`.
- `provider_retry_trace_refs` only include true retry-related refs, not successful first-attempt provider-call traces or retry-looking artifact labels from non-retry calls.
