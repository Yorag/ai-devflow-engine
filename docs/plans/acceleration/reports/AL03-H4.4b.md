## Implementation Summary

- `ToolConfirmationRequestModel` now persists four runtime-owned deny fields:
  - `planned_deny_followup_action`
  - `planned_deny_followup_summary`
  - `deny_followup_action`
  - `deny_followup_summary`
- `ToolConfirmationService.create_request()` records the planned deny follow-up source while the request is still pending.
- `ToolConfirmationService.deny()` now:
  - validates the planned deny follow-up source
  - restricts final follow-up action values to `continue_current_stage`, `run_failed`, `awaiting_run_control`
  - writes final `deny_followup_action` / `deny_followup_summary` only for the denied state
  - forwards the stable denied follow-up source through the runtime resume payload for downstream runtime consumers
  - preserves rollback behavior so audit or deny failures do not leave a half-written denied request
- Focused service and API tests now cover:
  - `continue_current_stage`
  - `run_failed`
  - `awaiting_run_control`
  - missing planned follow-up source
  - paused deny conflict
  - terminal deny conflict
