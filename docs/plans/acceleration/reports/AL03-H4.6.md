## Implementation Summary

- `RunLifecycleService` now owns terminate command validation, runtime termination orchestration, current session/run/stage terminal mutation, approval refresh, tool-confirmation terminal cancellation integration, and terminal `system_status` emission.
- `TerminalStatusProjector` centralizes terminal `system_status` payload construction so terminated and failed terminal feed semantics converge on one helper instead of being hand-built in multiple paths.
- Pending approvals remain `pending` in runtime storage but their latest top-level `approval_request` feed entry is refreshed to `is_actionable = false` with a terminated disabled reason.
- Pending tool confirmations are cancelled through the existing H4.4a service path, and H4.6 appends only the refreshed top-level cancelled `tool_confirmation` projection without synthesizing allow/deny result semantics.
- The API exposes a dedicated terminate route, a strict `RunTerminateRequest` schema, a `RunCommandResponse`, and `ErrorResponse` mappings for `404`, `409`, `422`, and `500`.
- Route wiring preserves body-less POST support while documenting a clean OpenAPI request-body component for terminate without a nullable `anyOf` wrapper.

- Service-layer review initially found that H4.6 reimplemented terminal tool-confirmation cancellation semantics locally instead of reusing the H4.4a service path; the slice was corrected to delegate to `ToolConfirmationService.cancel_for_terminal_run(..., commit=False)` and re-approved.

- No H4.7 rerun semantics, no new `control_item` semantics, no central tracking writes, and no frontend changes were introduced.

Code quality / regression review:

- Service-layer review accepted the terminal mutation scope, rollback behavior, unified terminal `system_status` projection helper, and terminate-time approval/tool-confirmation refresh behavior after the cancellation reuse fix landed.
- API-layer review initially found an OpenAPI regression: the terminate body schema rendered with a nullable `anyOf` shape because the route mixed an optional body annotation with manual request-body overrides. The route was corrected to use `Body(default_factory=RunTerminateRequest)`, the manual override was removed, and an explicit no-`anyOf` assertion was added to the API test.
- Residual non-blocking risk remains that terminate's optional-body route style is not identical to pause/resume, and the OpenAPI test does not assert `requestBody.required` explicitly. Current tests cover the important runtime behavior and schema shape, so this is not a blocker for H4.6.
