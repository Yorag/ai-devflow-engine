## Implementation Summary

- Added `InspectorProjectionService.get_tool_confirmation_detail()` and tool-confirmation-specific read-only helpers in `backend/app/services/projections/inspector.py`.
- The projection now reads `ToolConfirmationRequestModel`, requires matching `RunControlRecord(control_type=tool_confirmation)` as internal linkage, filters same-stage artifacts through explicit confirmation linkage rules, reads matching `tool_confirmation` event payloads from `EventStore`, and attaches same-stage `stage_node` context.
- The projection returns the strict `ToolConfirmationInspectorProjection` schema with grouped `identity`, `input`, `process`, `output`, `artifacts`, and `metrics` sections.
- Pending, allowed, and denied outcomes preserve tool-confirmation semantics without falling back to approval or control-item structures.
- Decision resolution is normalized to `allowed`, `denied`, or `None` only; unsupported runtime or artifact decision values are ignored so the projection never violates its own schema.
- `follow_up_result`, `tool_result_ref`, `audit_refs`, `side_effect_refs`, `alternative_path_summary`, and explicit process linkage fields are exposed from persisted runtime rows, filtered artifact data, and matching event payloads.
- Raw `graph_thread_ref` and `graph_thread_id` values are scrubbed from nested snapshots before projection.
- Added `GET /api/tool-confirmations/{toolConfirmationId}` in `backend/app/api/routes/query.py` with unified `ErrorResponse` handling and OpenAPI coverage.
