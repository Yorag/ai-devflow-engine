## Implementation Summary

- Added `backend/app/observability/log_query.py` with a focused `LogQueryService` that:
  - validates visible run/stage ownership through control/runtime stores,
  - reads `RunLogEntryModel` rows from `log.db`,
  - supports `level`, `category`, `source`, `since`, `until`, `cursor`, and `limit`,
  - pages by stable `created_at + log_id` ordering and cursor encoding,
  - enforces default/max query limits from canonical current `PlatformRuntimeSettings.log_policy`,
  - returns stable `not_found`, `log_query_invalid`, and `config_snapshot_unavailable` errors.
- Extended `backend/app/schemas/observability.py` with `RunLogQueryResponse` and reconciled the query/schema ceiling with current platform hard limits.
- Added focused service and schema coverage in:
  - `backend/tests/observability/test_log_query_service.py`
  - `backend/tests/schemas/test_observability_schemas.py`
- Added read-only API routes in `backend/app/api/routes/query.py`:
  - `GET /api/runs/{runId}/logs`
  - `GET /api/stages/{stageRunId}/logs`
- Kept route behavior thin by delegating query logic to `LogQueryService`.
- Added a route-local `APIRoute` wrapper so malformed query parameters still return `log_query_invalid` while OpenAPI preserves `limit` as `integer|null`.
- Added API/OpenAPI coverage in `backend/tests/api/test_query_api.py` for:
  - paginated run logs,
  - filtered run logs,
  - stage scoping,
  - invalid and malformed `limit`,
  - missing run/stage targets,
  - missing canonical runtime settings,
  - OpenAPI route/parameter/response schema presence.
