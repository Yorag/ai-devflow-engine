## Implementation Summary

- Added `deny_followup_action` and `deny_followup_summary` to the top-level `ToolConfirmationFeedEntry` schema.
- Enforced the stable contract in schema validation:
  - when `decision == denied`, both follow-up fields are required
  - when `decision != denied`, both follow-up fields must be `null`
- Hydrated workspace projection `tool_confirmation` entries from persisted H4.4b runtime source so denied read surfaces expose the top-level follow-up contract.
- Hydrated run timeline `tool_confirmation` entries from the same runtime source.
- Hydrated SSE `tool_confirmation` payloads before encoding and validated the final hydrated payload through `ToolConfirmationFeedEntry.model_validate(...).model_dump(...)`.
- Kept read surfaces from inferring denied follow-up meaning through `alternative_path_summary`, terminal run state, or Inspector-only semantics.
- Applied the user-approved minimal cross-lane producer fix so denied `ToolConfirmationService` results populate the same top-level follow-up fields and do not fail schema validation or return `500` through the API path.
