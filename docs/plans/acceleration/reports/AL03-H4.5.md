## Implementation Summary

- `RunLifecycleService` now owns pause/resume command validation, recovery-checkpoint persistence, same-thread resume validation, paused/resumed event emission, and paused approval/tool-confirmation feed refresh behavior.
- Pause changes only `PipelineRun.status` and `SessionModel.status` to `paused`; `StageRun.status` is preserved so resume restores the pre-pause waiting/running contract directly.
- Resume restores the persisted pre-pause `RunStatus` and derived `SessionStatus` from the latest validated recovery-checkpoint artifact after confirming checkpoint/run/thread/stage metadata consistency.
- Run logs now use the same redaction/log-summary path as other runtime command services and emit H4.5-specific accepted/completed/rejected/failed messages for both pause and resume.
- The API exposes dedicated pause/resume routes, strict request/response schemas, OpenAPI request-body components, and `ErrorResponse` mappings for `404`, `409`, `422`, and `500`.
- Route dependency wiring prefers `app.state.h45_runtime_port`, `app.state.h45_checkpoint_port`, and `app.state.h45_audit_service`, while falling back to existing `h41_*` runtime/checkpoint defaults.
- Body-less `POST` pause/resume requests are accepted while preserving OpenAPI schema refs for `RunPauseRequest` and `RunResumeRequest`.
