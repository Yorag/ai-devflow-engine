## Implementation Summary

- `ApprovalService.approve()` and `reject()` now validate approval/session/run/stage consistency, reject paused or non-waiting submissions, enforce non-blank reject reasons, and persist `ApprovalDecisionModel` only after the H4.4 command gate passes.
- `code_review_approval` approve now keeps the readiness block in front of decision creation and snapshot preparation, refreshes the blocked `approval_request` projection with delivery readiness fields, and leaves the approval pending when blocked.
- Reject now emits rollback `RunControlRecordModel` plus `control_item`, including the user reject reason and explicit follow-up progression text.
- Runtime log payloads for acceptance, rejection, blocking, and failure now include `result_status`, and run-log records carry `duration_ms`.
- The approvals API now exposes `POST /api/approvals/{approvalId}/approve` and `POST /api/approvals/{approvalId}/reject` with strict request/response schemas and OpenAPI coverage.
- The real API wiring now builds `DeliverySnapshotService` on the same control/runtime sessions as `ApprovalService` and disables snapshot auto-commit, so snapshot preparation participates in the same rollback boundary as the approval command.
- `DeliverySnapshotServiceError` is now mapped back through `ApprovalService` as a stable domain `ApprovalServiceError` instead of collapsing into `internal_error`.
