## Implementation Summary

- Added `InspectorProjectionService.get_control_item_detail()` and related read-only helpers in `backend/app/services/projections/inspector.py`.
- Added `GET /api/control-records/{controlRecordId}` in `backend/app/api/routes/query.py`.
- The route returns the existing strict `ControlItemInspectorProjection` schema with unified `ErrorResponse` handling and OpenAPI coverage.
- The service reads the target `RunControlRecordModel`, validates that the owning `PipelineRun`, `Session`, and `Project` remain visible, and assembles grouped `identity`, `input`, `process`, `output`, `artifacts`, and `metrics` sections.
- Control-item detail reads only explicitly related `control_item_trace` artifacts through `process.control_record_id`, which prevents unrelated same-stage artifacts and unrelated same-stage control traces from overriding the requested detail.
- `tool_confirmation` control records return the same stable 404 as missing or hidden control items and remain reserved for Q3.4a.
- Clarification-derived fields are exposed only for `clarification_wait` controls and are not leaked into retry control items even if `payload_ref` collides with a clarification id.
- Malformed `stage_node` event payloads are skipped defensively so they do not turn control-item detail into a 500 when runtime rows and artifacts remain usable.
- Raw `graph_thread_ref` and `graph_thread_id` values are scrubbed from artifact-derived snapshots.
