## Implementation Summary

- Added `TimelineProjectionService` in `backend/app/services/projections/timeline.py`.
- Added `GET /api/runs/{runId}/timeline` in `backend/app/api/routes/query.py`.
- The route returns the existing strict `RunTimelineProjection` schema.
- The service reads the target `PipelineRun` from `runtime.db`, validates that its owning `Session` and `Project` remain visible in `control.db`, and replays top-level Narrative Feed entries from `EventStore.list_for_session()`.
- Timeline replay filters by `DomainEvent.run_id` before parsing feed payloads, so off-target malformed payloads cannot break the target run timeline.
- `entries` are sorted by `occurred_at` while preserving event replay order for same timestamps.
- `approval_result` replay updates the matching `approval_request` actionability while keeping the result as its own top-level entry.
- `tool_confirmation` remains a top-level `tool_confirmation`; Provider retry data remains inside `stage_node.items[]` as `provider_call`.
- `current_stage_type` does not leak a stage from another run if `PipelineRun.current_stage_run_id` is stale or corrupt.
- Missing run, removed session, and removed project return stable 404 semantics with `Run timeline was not found.`.
