## Source Trace Conflict Gate Result

Initial pre-implementation review found no blocking conflict. The post-review V6.4 global route test exposed a blocking source-trace conflict on the old baseline: `docs/specs/function-one-backend-engine-design-v1.md` section 9.3 lists `GET /api/runs/{runId}` as a required query API that returns a run status summary and `current_stage_run_id`, but the old FastAPI OpenAPI output did not contain `/api/runs/{runId}`.

V6.4 is a global coverage regression. It validates the routes and schemas already emitted by the current service and records the coverage boundary. It must not become the first implementation point for a missing route, missing request schema, missing response schema, missing error response, or missing SSE payload model owned by another lane. If the RED test reveals a missing current route or a missing schema that cannot be fixed inside `backend/tests/api/test_openapi_contract.py`, `docs/api/function-one-openapi-notes.md`, or `README.md`, the implementer must stop and report an owner conflict instead of modifying `backend/app/api/routes/*`, schema modules, service modules, projections, events, runtime, or frontend files.

The current SSE route documents `text/event-stream` as a string response and the event stream query/path parameters. V6.4 also verifies that the event payload projection schemas used by workspace/timeline/feed contracts are present in OpenAPI components. It does not rewrite the SSE route response into a new event envelope schema.

Q3.2a resolved the blocker on the latest baseline by adding `GET /api/runs/{runId}` and `RunStatusSummaryProjection`. V6.4 now asserts that route's path, method, `runId` parameter, success schema, and main error responses without modifying production API code.

## API/OpenAPI Checklist

V6.4 does not change an API route. The new regression must assert these generated OpenAPI facts:

- `GET /api/openapi.json` returns HTTP 200 JSON.
- `GET /api/docs` returns HTTP 200 HTML that references the OpenAPI URL.
- The OpenAPI document contains every Function One core path and method listed in `EXPECTED_CORE_ROUTE_METHODS`.
- Tool confirmation command and detail routes expose request body schemas, response schemas, and main error responses.
- Run log, stage log, and audit log routes expose query parameters, response schemas, and main error responses.
- The SSE route exposes `sessionId`, `after`, and `limit` parameters; returns a `text/event-stream` response; and exposes the main event/feed payload component schemas used by the public projections.
- The `session_status_changed` SSE event payload shape is tracked as `session_id`, `status`, `current_run_id`, and `current_stage_type`; `status` is backed by `SessionStatus`, and `current_stage_type` is backed by `StageType` when present.
- The document includes key component schemas for core route groups.

def _schema_ref(route: dict[str, Any], status_code: str) -> str:
    return route["responses"][status_code]["content"]["application/json"]["schema"][

    return route["requestBody"]["content"]["application/json"]["schema"]["$ref"]
