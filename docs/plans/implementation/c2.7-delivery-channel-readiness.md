## Scope

In scope:
- Compute readiness for the current Project's default `DeliveryChannel`.
- `demo_delivery` validates as ready without Git fields or credentials.
- `git_auto_delivery` validates as not ready when required Git fields are blank or absent.
- `git_auto_delivery` validates as not ready when `credential_ref` is absent, invalid, not an allowed `env:<NAME>` reference, or points to a missing or unusable environment credential.
- Allowed `env:<NAME>` credential references resolve to `credential_status = ready` only when the referenced environment variable exists and is non-empty after trimming. C2.7 checks availability, but it never logs, returns, audits, or otherwise exposes the credential value.
- Persist the latest validation result on the control-plane `DeliveryChannelModel`: `credential_status`, `readiness_status`, `readiness_message`, `last_validated_at`, and `updated_at`.
- Return validation response fields: `readiness_status`, `readiness_message`, `credential_status`, `validated_fields`, `validated_at`.
- Write a sanitized service JSONL log for validation attempts.
- Write audit records for successful validation and rejected validation attempts.
- Expose route and schema coverage in `/api/openapi.json`.

Out of scope:
- Configuration package import/export.
- PlatformRuntimeSettings.
- Delivery snapshot creation or runtime snapshot mutation.
- Final code-review approval blocking.
- Real Git hosting calls, remote repository probing, branch existence checks, or external service writes.
- Session/template behavior.
- Frontend UI changes.
- Database migrations and dependency changes.
- Git write actions.

`compute_readiness(channel)` behavior:
- For `delivery_mode = demo_delivery`:
  - `credential_status = ready`
  - `readiness_status = ready`
  - `readiness_message = "demo_delivery is ready."`
  - `validated_fields = ("delivery_mode",)`
- For `delivery_mode = git_auto_delivery`:
  - Check these fields: `scm_provider_type`, `repository_identifier`, `default_branch`, `code_review_request_type`, `credential_ref`.
  - Missing or blank Git target fields return `readiness_status = unconfigured`, `credential_status = unbound`, and `readiness_message = "git_auto_delivery requires <sorted missing fields>"`.
  - Missing or blank `credential_ref` returns `credential_status = unbound`, `readiness_status = unconfigured`, and `readiness_message = "git_auto_delivery requires credential_ref"`.
  - Invalid or disallowed credential reference returns `credential_status = invalid`, `readiness_status = invalid`, and `readiness_message = "DeliveryChannel credential_ref must use an allowed env: credential reference."`
  - Safe `env:<NAME>` reference whose environment variable is unset returns `credential_status = unbound`, `readiness_status = unconfigured`, and `readiness_message = "DeliveryChannel credential_ref does not resolve to an available credential."`
  - Safe `env:<NAME>` reference whose environment variable exists but is empty or whitespace returns `credential_status = invalid`, `readiness_status = invalid`, and `readiness_message = "DeliveryChannel credential_ref resolves to an empty credential."`
  - Safe `env:<NAME>` reference whose environment variable exists and is non-empty after trimming returns `credential_status = ready`, `readiness_status = ready`, and `readiness_message = "git_auto_delivery is ready."`
  - `validated_fields` includes each field inspected in deterministic order.

`resolve_credential_status(credential_ref)` behavior:
- `None` or blank string returns `CredentialStatus.UNBOUND`.
- Invalid or disallowed reference returns `CredentialStatus.INVALID`.
- Safe `env:<NAME>` reference using the C2.6 allowed prefix and ASCII name rules calls the injected credential resolver with `<NAME>`.
- Resolver returns `None`: `CredentialStatus.UNBOUND`.
- Resolver returns an empty or whitespace-only string: `CredentialStatus.INVALID`.
- Resolver returns a non-empty string: `CredentialStatus.READY`.
- It never logs, returns, audits, or persists the resolver value.

`validate_project_channel(project_id, trace_context)` behavior:
- Requires `audit_service` and `log_writer` before mutating state.
- Loads a visible Project and its default DeliveryChannel using the same not-found semantics as C2.6.
- Computes readiness from the latest current Project configuration.
- Sets `channel.credential_status`, `channel.readiness_status`, `channel.readiness_message`, `channel.last_validated_at`, and `channel.updated_at` to the validation timestamp.
- Writes a sanitized service JSONL log for every successful validation command attempt.
- Writes an audit record with action `delivery_channel.validate` and result `succeeded` when the validation command completes, even when the computed readiness is `unconfigured` or `invalid`; those states are the command output, not command failure.
- If an unexpected validation failure happens after the channel is resolved while the audit ledger is still available, write `delivery_channel.validate.failed` before raising. If the audit ledger itself fails, propagate that failure and roll back the control write.
- Commits the control write only after log and audit writes complete.
- If audit or log writing fails, rolls back the control write and propagates the failure.
- If Project or DeliveryChannel is missing, writes `delivery_channel.validate.rejected` audit before raising the existing `not_found` service error.
- Does not read or write any runtime `DeliveryChannelSnapshotModel` row.

Expected messages:
- Demo ready: `demo_delivery is ready.`
- Git missing fields: `git_auto_delivery requires <field names>`
- Missing env credential: `DeliveryChannel credential_ref does not resolve to an available credential.`
- Empty env credential: `DeliveryChannel credential_ref resolves to an empty credential.`
- Invalid credential ref: `DeliveryChannel credential_ref must use an allowed env: credential reference.`
- Git ready: `git_auto_delivery is ready.`

The C2.6 save error string `DeliveryChannel credential_ref must use an env: credential reference.` remains unchanged for PUT validation. The C2.7 readiness path additionally verifies that a safe reference resolves to a usable environment credential, but the actual credential value remains secret.

## Log & Audit Integration

- Runtime log category: service-level JSONL log with `LogCategory.DELIVERY`, source `services.delivery_channels`, message `DeliveryChannel readiness validation result computed.` This log records the computed readiness result before final commit; it must not imply the control command has committed successfully.
- Audit actions:
  - Success: `delivery_channel.validate`
  - Rejected: `delivery_channel.validate.rejected`
  - Failed: `delivery_channel.validate.failed` when validation itself fails and the audit ledger remains available.
- Actor: `AuditActorType.USER`, `actor_id="api-user"`.
- Target: `target_type="delivery_channel"`, `target_id` is the resolved channel id, or `project:<project_id>` when no channel can be resolved.
- Trace fields: route calls pass `get_trace_context()` into the service. JSONL and audit records inherit `request_id`, `trace_id`, `correlation_id`, `span_id`, and `parent_span_id`.
- Redaction: runtime log and audit metadata include project id, delivery channel id, delivery mode, readiness status, credential status, readiness message, validated fields, and `validated_at`. Credential references are stored as safe `env:<NAME>` references or `[blocked:credential_ref]`; raw credential values, authorization headers, cookies, API keys, private keys, prompt text, and run snapshot refs are not included.
- Audit failure behavior: audit failure raises and rolls back the control write. Tests prove that a failed audit does not persist validation status changes.
- Log failure behavior: service JSONL write failure writes `delivery_channel.validate.failed` if the audit ledger is available, then raises and rolls back the control write. If the later success audit write fails after the JSONL result-computed record is written, the control write still rolls back, and the JSONL record remains a neutral computed-result observation rather than a success/commit record. Ordinary run logs are not involved because C2.7 is a control-plane command.
- Product truth: audit/log records are observability facts only; `DeliveryChannelModel` remains the current Project readiness truth. No historical run snapshot is mutated.

class RecordingAuditService:
    def __init__(self) -> None:
        self.records: list[dict[str, Any]] = []

    def record_command_result(self, **kwargs: Any) -> object:
        self.records.append({"method": "record_command_result", **kwargs})
        return object()

    def record_rejected_command(self, **kwargs: Any) -> object:
        self.records.append(
            {"method": "record_rejected_command", "result": AuditResult.REJECTED, **kwargs}
        )
        return object()


class FailingAuditService:
    def record_command_result(self, **kwargs: Any) -> object:
        raise RuntimeError("audit ledger unavailable")

    def record_rejected_command(self, **kwargs: Any) -> object:
        raise RuntimeError("audit ledger unavailable")


class RecordingLogWriter:
    def __init__(self) -> None:
        self.records: list[Any] = []

    def write(self, record: Any) -> JsonlWriteResult:
        self.records.append(record)
        return JsonlWriteResult(
            log_id=record.log_id or "log-delivery-readiness",
            log_file_ref="logs/app.jsonl",
            line_offset=0,
            line_number=1,
            log_file_generation="app",
            created_at=record.created_at or LATER,
        )


class FailingLogWriter:
    def write(self, record: Any) -> JsonlWriteResult:
        raise OSError("log path unavailable")

        span_id="span-delivery-readiness",
        parent_span_id=None,

    manager = DatabaseManager(

        _database_urls={
            role: f"sqlite:///{(tmp_path / f'{role.value}.db').as_posix()}"
