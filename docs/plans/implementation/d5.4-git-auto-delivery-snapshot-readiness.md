# D5.4 Git Auto Delivery Snapshot Readiness Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:test-driven-development` for every production, behavior, test, or refactor change. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be supplied precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the AL04 D5.4 `git_auto_delivery` adapter orchestration that reads the frozen delivery snapshot, calls the five delivery tools through `ToolRegistry.execute()`, and returns a delivery adapter result with branch, commit, PR/MR, process, audit, and failure-step evidence.

**Architecture:** Add `backend/app/delivery/git_auto.py` as a deterministic adapter over the existing D4.1 delivery adapter contract and D5.1-D5.3 SCM tools. The adapter receives a `ToolRegistry`, a per-tool `ToolExecutionContext` factory, fixture/workspace repository settings, and an optional confirmation resolver; every delivery step is executed through `ToolRegistry.execute()` with a child span, never by calling `ScmDeliveryAdapter`, Git CLI, or a remote client directly. The adapter consumes only the current run's frozen `delivery_channel_snapshot_ref`; it does not read project-level latest `DeliveryChannel`, does not freeze snapshots, and does not add runtime, API, frontend, or coordination-state behavior in this slice.

**Tech Stack:** Python 3.11+, Pydantic v2 delivery models, existing `ToolRegistry`, existing D5.1-D5.3 delivery tools, W5.0c fixture Git repository and mock remote client, pytest via `uv run`.

```python

```

`deliver()` behavior:

1. Reject non-`git_auto_delivery` input with `ValueError("GitAutoDeliveryAdapter only supports git_auto_delivery")`.
2. Execute `read_delivery_snapshot` through the registry with input `{"run_id": delivery_input.run_id}`.
3. Pass the returned `delivery_channel_snapshot` into `assert_snapshot_ready()`.
4. Execute, in order, `prepare_branch`, `create_commit`, `push_branch`, and `create_code_review_request` through the registry.
5. For every step, create a child trace with `span_id=f"git-auto-delivery-{step_name}"`, inherited `trace_id`, inherited `correlation_id`, same `run_id`, and same `stage_run_id`.
6. If a tool returns `WAITING_CONFIRMATION` and `confirmation_resolver` is present, call the resolver and then execute the returned confirmed request through the registry. If no resolver returns a request, return a `blocked` `DeliveryAdapterResult`.
7. If a tool returns `BLOCKED`, return a `blocked` `DeliveryAdapterResult`.
8. If a tool returns `FAILED`, return a `failed` `DeliveryAdapterResult`.
9. On success, call `build_delivery_record()` to return a succeeded `DeliveryAdapterResult` with branch, commit, code review URL, process refs, and audit refs from the successful `ToolResult` objects.

Tool request payloads:

```python
read_delivery_snapshot = {"run_id": delivery_input.run_id}

prepare_branch = {
    "repository_path": str(repository_path),
    "branch_name": branch_name,
    "base_branch": snapshot["default_branch"],
    "delivery_record_id": pending_delivery_record_id,
}

create_commit = {
    "repository_path": str(repository_path),
    "commit_message": commit_message,
    "delivery_record_id": pending_delivery_record_id,
}

push_branch = {
    "repository_path": str(repository_path),
    "remote_name": remote_name,
    "branch_name": branch_name,
    "delivery_record_id": pending_delivery_record_id,
}

create_code_review_request = {
    "repository_identifier": snapshot["repository_identifier"],
    "source_branch": branch_name,
    "target_branch": snapshot["default_branch"],
    "title": review_title,
    "body": review_body,
    "code_review_request_type": snapshot["code_review_request_type"],
    "delivery_record_id": pending_delivery_record_id,
}
```

`assert_snapshot_ready()` behavior:

```python
def assert_snapshot_ready(
    self,
    *,
    delivery_input: DeliveryAdapterInput,
    snapshot: Mapping[str, object],
) -> Mapping[str, object]:
```

It must validate only the frozen snapshot payload returned by `read_delivery_snapshot`:
- `delivery_channel_snapshot_ref == delivery_input.delivery_channel_snapshot_ref`
- `delivery_mode == "git_auto_delivery"`
- `scm_provider_type`, `repository_identifier`, `default_branch`, `code_review_request_type`, and `credential_ref` are non-empty strings
- `credential_status == "ready"`
- `readiness_status == "ready"`

On validation failure, raise `GitAutoDeliveryError(error_code="delivery_snapshot_not_ready" or "delivery_snapshot_missing", safe_message=..., safe_details={...})`. Do not query project-level latest configuration.

`build_delivery_record()` behavior:

```python
def build_delivery_record(
    self,
    *,
    delivery_input: DeliveryAdapterInput,
    snapshot: Mapping[str, object],
    step_results: Mapping[str, ToolResult],
) -> DeliveryAdapterResult:
```

It must return a succeeded `DeliveryAdapterResult` with:
- `delivery_mode=DeliveryMode.GIT_AUTO_DELIVERY`
- `result_ref=f"git-auto-delivery-result:{run_id}"`
- `process_ref=f"git-auto-delivery-process:{run_id}"`
- `branch_name` from `prepare_branch.output_payload["branch_name"]`
- `commit_sha` from `create_commit.output_payload["commit_sha"]`
- `code_review_url` from `create_code_review_request.output_payload["code_review_url"]`
- `audit_refs` from each successful `ToolResult.audit_ref.audit_id` when present
- `log_summary_refs=[]` because `ToolRegistry` run-log recorders do not return stable log IDs in the current contract
- the original delivery stage trace context

Failure result helper:
- For failed tool results, use `DeliveryAdapterError(error_code=<tool error code>, safe_message=<tool safe message>, safe_details={"failed_step": step_name, "tool_name": result.tool_name, "tool_call_id": result.call_id, "tool_confirmation_ref": result.tool_confirmation_ref, "tool_audit_id": result.audit_ref.audit_id if present, **result.error.safe_details})`.
- Use status `"blocked"` for `WAITING_CONFIRMATION` and `BLOCKED`; use `"failed"` for `FAILED`.
- Preserve `process_ref=f"git-auto-delivery-process:{run_id}"` on failure so logs/audits can locate the attempted process.

## Log And Audit Integration

- Runtime log category and tool outcome logging are provided by the existing `ToolExecutionContext.run_log_recorder.record_tool_result(...)` path. D5.4 tests must assert log records for each executed tool.
- Gate audit intent is provided by `ToolExecutionContext.audit_recorder.record_tool_intent(...)` because all five tools are audit-required or side-effecting.
- Concrete Git and remote audit records are written by D5.2/D5.3 `ScmDeliveryAdapter` helpers. D5.4 must collect audit IDs from registry `ToolResult.audit_ref` and include them in the adapter result.
- `request_id`, `trace_id`, and `correlation_id` are inherited from the delivery stage trace. Each step gets a unique child `span_id`; `parent_span_id` must equal the delivery stage span id.
- Sensitive details must remain inside existing `ToolError` and redaction policies. D5.4 must not add raw Git stdout/stderr, tokens, authorization headers, cookies, or credential values to adapter errors.
- Logs and audit refs do not replace `DeliveryRecord`, Narrative Feed, Inspector, or product state truth.
