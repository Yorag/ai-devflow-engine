**Goal:** Persist `StageRun` lifecycle records with stable graph-contract references, runtime-state transitions, and run-log summaries so formal business stages can be queried and replayed without inferring stage identity from raw graph state.

**Architecture:** Extend the existing `runtime.db` `StageRunModel` to carry `graph_node_key` and `stage_contract_ref`, then add a focused `StageRunRepository` plus `StageRunService` that own start, waiting, and completion transitions. The service stays inside AL01 ownership boundaries: it updates only runtime rows and emits sanitized run-log summaries with inherited `TraceContext`; it does not change API routes, graph persistence, event payload schemas, or central orchestration state outside `StageRun`/`PipelineRun`.

**Tech Stack:** Python 3.11+, SQLAlchemy ORM, Pydantic `TraceContext`, existing runtime/control enums, pytest via `uv run python -m pytest`.

**Execution Status:** Local worker result `reported`; expected committed ingest result `implemented` after a user-approved checkpoint commit.

## Design Decisions

- `StageRunModel.graph_node_key` and `StageRunModel.stage_contract_ref` become required persisted columns so later runtime recovery and projections do not infer them from ephemeral graph state.
- `StageRunRepository` owns low-level row creation/loading/persistence errors; repository failures surface as slice-specific repository errors rather than raw SQLAlchemy exceptions.
- `StageRunService` exposes exactly the split-plan methods:
  - `start_stage()`
  - `mark_stage_waiting()`
  - `complete_stage()`
- Failure semantics are handled by `complete_stage(..., status=StageStatus.FAILED)` rather than inventing an unplanned fourth public method.
- `mark_stage_waiting()` must only accept waiting statuses defined by the backend contract: `waiting_clarification`, `waiting_approval`, `waiting_tool_confirmation`.
- `solution_design_approval` and `code_review_approval` remain waiting transitions on the source `StageRun`; the service must not create separate approval pseudo-stages.
- Run-log integration uses sanitized summaries with inherited `TraceContext`, following existing AL01 service patterns:
  - source should be stage-service specific, e.g. `services.stages`
  - category should be `LogCategory.RUNTIME`
  - log write failures must not roll back domain state, but must be surfaced in tests as non-domain-failing behavior
- This slice updates model metadata only. Approval-gated `alembic upgrade head` remains optional verification and must not run without user approval.

## Log & Audit Integration

This slice changes runtime lifecycle behavior, so implementation must include runtime-log integration:

- Log `start`, `waiting`, and `complete` lifecycle summaries.
- Each log payload must include at minimum `run_id`, `stage_run_id`, `stage_type`, target status, attempt index, and any stable refs touched (`graph_node_key`, `stage_contract_ref`, `input_ref`, `output_ref`).
- Logs must inherit `request_id`, `trace_id`, `correlation_id`, `span_id`, `parent_span_id`, `session_id`, `run_id`, and `stage_run_id` from `TraceContext`.
- Log failures must not replace or block persisted `StageRun` truth; tests must prove domain state commits even if the log writer raises.
- No new audit log contract is required for this internal runtime service slice.
