> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task in the main agent session. Steps use checkbox (`- [ ]`) syntax for tracking. Do not use subagent handoff, Git write actions, PR creation, merge, tag, or branch cleanup for this repository slice.

**Goal:** Establish the first `runtime.db` SQLAlchemy ORM model boundary for PipelineRun, StageRun, runtime artifacts, approvals, tool confirmations, run controls, snapshots, and DeliveryRecord without creating graph, event, log, or control models.

**Architecture:** Add a focused `backend/app/db/models/runtime.py` module whose declarative base is bound to `ROLE_METADATA[DatabaseRole.RUNTIME]`. Runtime models use stable enum values from C1.1, store structured snapshot payload groups as JSON where the already approved Schema contracts define nested structures, and use `session_id` strings to reference control sessions without duplicating the Session entity. Alembic imports the runtime model module so the runtime role metadata is populated for migration operations while other database role metadata remains isolated.

**Tech Stack:** Python 3.11, SQLAlchemy 2.x, Alembic, pytest through the repo-local `uv run` environment.

## Log & Audit Integration

This slice defines runtime persistence models only. It does not implement user commands, runtime lifecycle services, model calls, tool execution, workspace writes, `bash`, Git delivery, remote delivery, configuration writes, log writers, audit writers, or safety-sensitive command failure behavior.

The runtime model boundary must still preserve future log/audit integration points:

- `PipelineRunModel.trace_id` stores the run-level trace reference required by the backend spec.
- `ToolConfirmationRequestModel.audit_log_ref` and `process_ref` store stable references to later `log.db` or process records without creating log tables in `runtime.db`.
- `RunControlRecordModel.graph_interrupt_ref`, `ApprovalRequestModel.graph_interrupt_ref`, and `ToolConfirmationRequestModel.graph_interrupt_ref` store graph interrupt references without creating graph tables in `runtime.db`.
- Tests must prove `runtime.db` does not contain `run_log_entries`, `audit_log_entries`, `log_payloads`, `domain_events`, or graph checkpoint tables.

## Scope Rules

- Do not create graph, event, log, or control models.
- Do not create `Session`, `Project`, `GraphDefinition`, `GraphThread`, `GraphCheckpoint`, `GraphInterrupt`, `DomainEvent`, `RunLogEntry`, `AuditLogEntry`, `LogPayload`, Narrative Feed projection, or Inspector projection tables in this slice.
- Do not add API routes, OpenAPI tests, services, seed data, Pydantic schema changes, frontend changes, or migration revision files.
- Do not add per-database URL/path fields to `EnvironmentSettings`, runtime models, PlatformRuntimeSettings, configuration packages, or frontend-facing settings.
- Runtime models may use JSON columns only for nested payloads already defined as structured contract groups, stable refs, process records, metrics, or schema-compatible snapshot bodies.
- Tool confirmation remains independent from approval. It must not use `ApprovalStatus`, `ApprovalType`, `ApprovalDecisionModel`, or approval foreign keys.
- `system_status` must not be stored as `RunControlRecordModel.control_type`.
