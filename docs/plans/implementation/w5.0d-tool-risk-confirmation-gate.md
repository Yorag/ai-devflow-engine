# W5.0d Tool Risk Confirmation Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:test-driven-development` for every production or test change. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be supplied precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the first real `ToolRiskClassifier` and confirmation gate so every tool action is classified as `read_only`, `low_risk_write`, `high_risk`, or `blocked`, high-risk actions create a `ToolConfirmationRequest` and return `waiting_confirmation`, blocked actions return `tool_risk_blocked`, and confirmed execution is limited to the exact approved action.

**Architecture:** Keep W5.0b's `ToolRegistry.execute()` as the only execution entry, add a new `backend/app/tools/risk.py` value-object boundary for risk assessment and confirmation grants, then extend `ToolExecutionGate` to (1) classify risk before execution, (2) create a confirmation request through the existing H4.4a service surface for high-risk actions, and (3) verify an approved confirmation grant still matches the same tool name, input summary, target resource, and risk result before executing the concrete tool. The existing AL03-owned `ToolConfirmationService` is treated as a consumed dependency, not a write target.

**Tech Stack:** Python 3.11+, Pydantic v2, dataclasses, pytest, existing `TraceContext`, `ToolProtocol`, `ToolRegistry`, `ToolExecutionGate`, `ToolConfirmationService`, and backend enum / error-code contracts.

## Key Behavior Decisions

- `ToolExecutionGate` remains the only execution entry. Risk classification happens inside that gate before concrete tool execution.
- High-risk actions do not execute immediately. The gate creates a confirmation request and returns:
  - `ToolResult.status = waiting_confirmation`
  - `ToolResult.error.error_code = tool_confirmation_required`
  - `ToolResult.tool_confirmation_ref = <tool_confirmation_id>`
- Blocked actions do not create a confirmation request. They return:
  - `ToolResult.status = blocked`
  - `ToolResult.error.error_code = tool_risk_blocked`
  - a security audit rejection when an audit recorder is available
- To enforce “allow only this same action”, W5.0d introduces a stable confirmation grant object carried by the execution request:
  - `tool_confirmation_id`
  - `confirmation_object_ref`
  - `tool_name`
  - `input_digest`
  - `target_summary`
  - `risk_level`
  - `risk_categories`
- The confirmation object ref used for high-risk request creation is derived from the current tool call and the normalized input summary. The gate recomputes the same summary before executing after allow.
- If a high-risk request is re-attempted with no grant or with a drifted grant, the gate must refuse immediate execution. This plan uses the strict behavior:
  - if there is no grant, create a fresh confirmation request
  - if there is a grant but any checked field drifts, return `tool_confirmation_required` and create a fresh confirmation request for the new action
- The existing W5.0b workspace-boundary rejection stays in place as an earlier safety gate. W5.0d still classifies path escape and platform-runtime mutation patterns as `blocked` when they can be derived from declared targets or command text before execution.

```python
from __future__ import annotations

import hashlib
import json
from typing import Any, Protocol

from pydantic import BaseModel, ConfigDict, Field

from backend.app.domain.enums import ToolRiskCategory, ToolRiskLevel
from backend.app.tools.execution_gate import ToolExecutionRequest
from backend.app.tools.protocol import ToolProtocol


class _StrictBaseModel(BaseModel):
    model_config = ConfigDict(extra="forbid")


class ToolConfirmationGrant(_StrictBaseModel):
    tool_confirmation_id: str = Field(min_length=1)
    confirmation_object_ref: str = Field(min_length=1)
    tool_name: str = Field(min_length=1)
    input_digest: str = Field(min_length=1)
    target_summary: str = Field(min_length=1)
    risk_level: ToolRiskLevel
    risk_categories: list[ToolRiskCategory] = Field(default_factory=list)


class ToolConfirmationRequestRecord(_StrictBaseModel):
    tool_confirmation_id: str = Field(min_length=1)
    confirmation_object_ref: str = Field(min_length=1)


class ToolRiskAssessment(_StrictBaseModel):
    risk_level: ToolRiskLevel
    risk_categories: list[ToolRiskCategory] = Field(default_factory=list)
    reason: str = Field(min_length=1)
    command_preview: str | None = None
    target_summary: str = Field(min_length=1)
    expected_side_effects: list[str] = Field(default_factory=list)
    alternative_path_summary: str | None = None
    input_digest: str = Field(min_length=1)
    confirmation_object_ref: str = Field(min_length=1)

    @property
    def requires_confirmation(self) -> bool:
        return self.risk_level is ToolRiskLevel.HIGH_RISK

    @property
    def is_blocked(self) -> bool:
        return self.risk_level is ToolRiskLevel.BLOCKED


class ToolConfirmationRequestPort(Protocol):
    def create_request(
        self,
        *,
        session_id: str,
        run_id: str,
        stage_run_id: str,
        confirmation_object_ref: str,
        tool_name: str,
        command_preview: str | None,
        target_summary: str,
        risk_level: ToolRiskLevel,
        risk_categories: list[ToolRiskCategory],
        reason: str,
        expected_side_effects: list[str],
        alternative_path_summary: str | None,
        trace_context,
    ) -> Any: ...
```
