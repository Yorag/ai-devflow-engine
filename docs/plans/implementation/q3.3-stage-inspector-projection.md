# Q3.3 Stage Inspector Projection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Implementer subagents must use `superpowers:test-driven-development`. Fallback to `superpowers:executing-plans` only if subagents cannot be dispatched or this task boundary cannot be passed precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the backend stage Inspector projection and `GET /api/stages/{stageRunId}/inspector` route so the frontend can inspect a stage's identity, input, process, output, artifacts, metrics, and stable references without raw graph state.

**Architecture:** Add a read-only `InspectorProjectionService` that loads a `StageRunModel`, validates its owning `PipelineRun`, visible `Session`, and visible `Project`, gathers stage artifacts, stage-scoped domain events, tool confirmations, and approval decisions, then returns the existing strict `StageInspectorProjection` schema. The service keeps raw graph state out of the projection, uses `StageArtifactModel.process` and stable references as product truth, and keeps tool confirmation details delegated to the later Q3.4a detail projection.

**Tech Stack:** Python 3.11, FastAPI, SQLAlchemy 2.x ORM sessions, Pydantic v2 schemas, pytest through `uv run python -m pytest`.
