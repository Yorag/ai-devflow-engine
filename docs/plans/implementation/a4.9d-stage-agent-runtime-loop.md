# A4.9d Stage Agent Runtime Loop Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` as the default execution skill for this plan. Use `superpowers:executing-plans` only if bounded `gpt-5.5 xhigh` subagent dispatch is unavailable or if the task boundary cannot be kept inside the allowed files. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement a formal single-stage `StageAgentRuntime` that connects Context Management, Provider Adapter, `AgentDecisionParser`, `ToolRegistry`, `StageArtifact` process records, and `StageNodeRunnerPort` results without using model free text to advance state or execute tools.

**Architecture:** Add `backend/app/runtime/stage_agent.py` as a bounded stage runner that implements `StageNodeRunnerPort`. The runtime builds a `ContextEnvelope` for every model iteration through `ContextEnvelopeBuilder`, invokes `LangChainProviderAdapter.invoke_with_retry()`, parses decisions through `AgentDecisionParser`, executes only validated tool-call decisions through `ToolRegistry.execute()`, writes process records through existing `ArtifactStore.append_process_record()`, completes submitted stage artifacts through existing `ArtifactStore.complete_stage_output()`, and returns `StageNodeResult` to LangGraph. `backend/app/runtime/nodes.py` receives a narrow fix so waiting or failed stage results are not marked as completed in LangGraph state.

**Tech Stack:** Python 3.11+, Pydantic v2, existing runtime/context/provider/tool/artifact contracts, repository-local `uv`, and pytest via `uv run pytest`.

## Log And Audit Integration

This slice touches runtime nodes, model calls, tool calls, workspace side-effect routing, and safety-sensitive failure paths.

Runtime process records:

- `stage_agent_started`
- `model_call_trace`
- `decision_trace`
- `tool_trace`
- `tool_confirmation_trace` when `ToolResult.status == WAITING_CONFIRMATION`
- `provider_retry_trace` when returned by provider adapter
- `provider_circuit_breaker_trace` when returned by provider adapter
- `structured_output_repair_trace`
- `recovery_checkpoint`
- `side_effect_reconciliation_trace` when a tool result reports failed or unknown reconciliation
- `stage_agent_failed`

Rules:

- `StageAgentRuntime` does not create a new runtime log writer surface in this slice. It relies on existing `ArtifactStore` log behavior and existing tool/provider log boundaries.
- `StageAgentRuntime` does not write audit records directly. Audit responsibility remains inside `ToolRegistry` / tool confirmation / delivery adapters.
- `ToolRegistry.execute()` receives `ToolExecutionContext` with the stage contract, runtime tool timeout limits, workspace boundary, audit recorder, run-log recorder, risk policy, and confirmation port supplied to `StageAgentRuntime`.
- A high-risk tool result must return `StageStatus.WAITING_TOOL_CONFIRMATION` with a stable `tool_confirmation_trace`; it must not execute the tool directly.
- A blocked tool result must return `StageStatus.FAILED` with structured `tool_trace` and must not create an allowable confirmation request.
- Provider retry and circuit breaker traces are copied from `ModelCallResult` into `StageArtifact.process`, not only left in logs.
- Error safe details must not include raw model text, raw prompt bodies, credentials, raw stack traces, or full tool output.

Runtime rules:

- Resolve the current stage contract only from `graph_definition.stage_contracts[stage_type.value]`.
- Before model invocation, reject incompatible model bindings:
  - tools are available and `model_binding_snapshot.capabilities.supports_tool_calling` is false
  - structured output is required and `supports_structured_output` is false
  - requested output tokens exceed `model_binding_snapshot.capabilities.max_output_tokens`
- Build a fresh context envelope and manifest every iteration through `ContextEnvelopeBuilder.build_for_stage_call()`.
- Invoke provider through `LangChainProviderAdapter.invoke_with_retry()` only.
- Parse every provider result through `AgentDecisionParser.parse_model_result()`.
- For `REQUEST_TOOL_CALL`, call `ToolRegistry.execute()` exactly once with:

```python
ToolExecutionRequest(
    tool_name=decision.tool_call.tool_name,
    call_id=decision.tool_call.call_id,
    input_payload=dict(decision.tool_call.input_payload),
    trace_context=request.invocation.trace_context,
    coordination_key=f"{request.invocation.stage_run_id}:{decision.tool_call.call_id}",
)
```

- `ToolExecutionContext.stage_contracts` must contain only the current graph definition stage contracts.
- `ToolResultStatus.SUCCEEDED` writes `tool_trace` and continues to the next iteration.
- `ToolResultStatus.WAITING_CONFIRMATION` writes `tool_trace`, `tool_confirmation_trace`, `recovery_checkpoint`, and returns `StageStatus.WAITING_TOOL_CONFIRMATION`.
- `ToolResultStatus.BLOCKED` writes `tool_trace`, `side_effect_reconciliation_trace`, `recovery_checkpoint`, and returns `StageStatus.FAILED`.
- `ToolResultStatus.FAILED` writes `tool_trace`, `recovery_checkpoint`, and returns `StageStatus.FAILED`.
- `SUBMIT_STAGE_ARTIFACT` completes output through `ArtifactStore.complete_stage_output()` and returns `StageStatus.COMPLETED`.
- `REQUEST_CLARIFICATION` returns `StageStatus.WAITING_CLARIFICATION` with `decision_trace` and `recovery_checkpoint`; creation of `ClarificationRecord` remains outside this slice.
- `REQUEST_TOOL_CONFIRMATION` returns `StageStatus.WAITING_TOOL_CONFIRMATION` with trace data; direct creation of `ToolConfirmationRequest` is handled through ToolRegistry execution-gate high-risk results.
- `REPAIR_STRUCTURED_OUTPUT` records `structured_output_repair_trace` and uses the next iteration with `model_call_type=structured_output_repair`.
- `RETRY_WITH_REVISED_PLAN` records `decision_trace` and continues while iteration limits allow.
- `FAIL_STAGE` records `stage_agent_failed` and returns `StageStatus.FAILED`.
- Iteration count is bounded by `runtime_limit_snapshot.agent_limits.max_react_iterations_per_stage`.
- Tool call count is bounded by `runtime_limit_snapshot.agent_limits.max_tool_calls_per_stage`.
- Structured repair count is bounded by `runtime_limit_snapshot.agent_limits.max_structured_output_repair_attempts`.
- Every loop iteration writes a `recovery_checkpoint` after decision handling.
