Implementer subagents must use `superpowers:test-driven-development`; reviewers run in the order spec / plan compliance first, then code quality / testing / regression. `superpowers:executing-plans` is only a fallback when subagents cannot be dispatched or the task boundary cannot be passed precisely.

**Goal:** Reconnect the existing frontend rerun affordance to the real backend projection contract `system_status.retry_action = retry:<run_id>` without changing the backend rerun command or V6.3 Playwright scenario.

**Architecture:** Keep `RerunAction` as the single frontend decision point for terminal-run rerun visibility. The component accepts only the current run's top-level `system_status` entry when its `retry_action` exactly targets that entry's `run_id`, then continues to call the existing `createRerun(sessionId)` API rather than treating the retry action string as a route, URL, or executable command. Fixture and renderer tests use backend-shaped `retry:<run_id>` payloads so V6.3 live coverage no longer depends on the old mock-only `create_rerun` marker.

**Tech Stack:** React 18, TypeScript strict mode, TanStack Query, existing API client and workspace projections, Vitest, Testing Library.

```tsx
export function resolveRerunActionState(
  entry: SystemStatusFeedEntry,
  currentRunId: string | null,
): RerunActionState {
  const isCurrentRun = Boolean(currentRunId) && entry.run_id === currentRunId;
  const targetsEntryRun = entry.retry_action === `retry:${entry.run_id}`;

  return {
    canRender:
      isCurrentRun &&
      targetsEntryRun &&
      (entry.status === "failed" || entry.status === "terminated"),
  };
}
```
