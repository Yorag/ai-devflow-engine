# R3.2a First-Run Publication Boundary Implementation Plan

> Coordination-adopted prerequisite plan for `R3.2a`. This plan stays in the working tree until the follow-up worker claim executes it.

**Goal:** Add the missing reusable multi-SQLite publication boundary that lets first-run startup stage product truth across `control.db`, `runtime.db`, `graph.db`, and `event.db` without exposing a half-started public state before the final publish point.

**Architecture:** Keep multi-SQLite responsibility split unchanged. Introduce a control-side publication gate for first-run startup. `RunLifecycleService.start_first_run()` may stage runtime, graph, and event truth before visibility publish, but workspace, timeline, and SSE readers must consult the same publication gate so unpublished startup rows remain invisible.

**Status note:** `R3.2a` is the authoritative prerequisite for final `R3.2` completion. The current `AL01-R3.2` worker branch remains a useful business-semantic blocked checkpoint and should be resumed after `R3.2a` lands.

## Scope

In scope:

- Add a reusable control-side publication boundary for first-run startup visibility.
- Stage startup runtime, graph, and event rows before they become publicly visible.
- Publish startup visibility only after the staged product set is complete.
- Keep failed or abandoned staged startup rows invisible to workspace, timeline, and SSE readers.
- Define retry or cleanup semantics for stale pending startup publications.
- Refactor `RunLifecycleService.start_first_run()` to use the new boundary.
- Add focused tests proving pending startup rows stay invisible, publish flips visibility in one public semantic, and failed startup does not leave a visible half-started run.

Out of scope:

- Approve/reject command routing changes
- Delivery snapshot execution
- Frontend work
- LangGraph runtime work
- Platform-plan, split-plan, or coordination-store final completion writes outside the claim itself

## Acceptance Baseline

- Multi-SQLite responsibility split remains intact. `R3.2a` must not collapse product truth back into a single SQLite file.
- Before the final publish point succeeds, workspace, timeline, and SSE readers must not expose a started first run or its top-level startup events as committed public truth.
- The publish point must atomically switch the control-side visible state for the started run, including `Session.current_run_id`, session running visibility, and publication state.
- A failure before publish may leave recoverable staging state, but it must not leave a publicly visible half-started run.
- Retry or cleanup semantics for stale pending startup attempts must be explicit and tested.
- `R3.2` final completion must reuse this boundary and may not weaken its source-trace requirement into “best effort”.
- Later `H4.4` final completion must reuse the same boundary class or primitive instead of inventing a second approval-specific publish mechanism.
