> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Fallback to `superpowers:executing-plans` only if subagents cannot be dispatched or the bounded file set below becomes unsafe. All subagents for this slice must use `gpt-5.5` with `xhigh` reasoning. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Seed the three Function One system templates and two builtin Provider records so draft Sessions can bind the default template.

**Architecture:** Add small control-plane services for template and provider seeding over `control.db`, backed by versioned role prompt assets in `backend/app/prompts/assets/roles`. Startup runs the seed after the default Project bootstrap, and read-only APIs expose seeded templates and providers through existing Pydantic schemas and OpenAPI. The seed writes use the existing L2.4 audit service; API list/detail responses remain product truth from control models, not audit records or local prompt files.

**Tech Stack:** Python 3.11+, FastAPI, Pydantic v2, SQLAlchemy 2.x, existing multi-SQLite `DatabaseManager`, C1.10a `PromptAssetRead`, L2.1 request trace context, L2.4 `AuditService`, and `pytest` through `uv run --no-sync`.

## Log & Audit Integration

- Audit actions:
  - `provider.seed_builtin` as one batch audit record for newly created builtin Provider records.
  - `template.seed_system` as one batch audit record for newly created system templates.
- Audit actor:
  - Use `AuditActorType.SYSTEM` with `actor_id="control-plane-seed"` because C2.2 seeding is system initialization, even when triggered during a read API request.
- Audit target:
  - `target_type="provider"` with `target_id="builtin-provider-seed"`.
  - `target_type="pipeline_template"` with `target_id="system-template-seed"`.
- Trace fields:
  - Startup inherits `_startup_trace_context()` from `backend/app/main.py`.
  - API-triggered self-healing seed calls inherit L2.1 `get_trace_context()`.
- Redaction:
  - Audit metadata may include `provider_ids`, `display_names`, `api_key_refs`, `default_model_ids`, `supported_model_counts`, `template_ids`, `template_names`, `template_source`, role IDs, and Provider IDs.
  - Audit metadata must not include real API key values or full prompt bodies.
- Failure behavior:
  - Seed services create missing control rows, flush them, write one batch audit record, then commit the control transaction.
  - If audit ledger write fails while seeding a Provider or template, rollback the uncommitted control transaction and propagate the error.
  - JSONL audit copy write failure follows the existing L2.4 degraded-copy behavior; it must not create a second configuration truth.
- Product boundary:
  - Audit records do not act as template, Provider, PromptAsset, Narrative Feed, Inspector, event, or runtime snapshot truth.

```python
def parse_front_matter(markdown: str) -> tuple[dict[str, str], str]:
    normalized = markdown.replace("\r\n", "\n")
    if not normalized.startswith("---\n"):
        raise ValueError("Prompt asset is missing YAML front matter.")
    closing_index = normalized.find("\n---\n", len("---\n"))
    if closing_index == -1:
        raise ValueError("Prompt asset front matter is not closed.")
    front_matter_text = normalized[len("---\n") : closing_index]
    body = normalized[closing_index + len("\n---\n") :].strip()
    metadata: dict[str, str] = {}
    for raw_line in front_matter_text.splitlines():
        if not raw_line.strip():
            continue
        key, separator, value = raw_line.partition(":")
        if separator != ":":
            raise ValueError(f"Invalid front matter line: {raw_line}")
        metadata[key.strip()] = value.strip().strip('"')
    return metadata, body
```
