> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:test-driven-development` for every production or test change. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be supplied precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement `POST /api/sessions/{sessionId}/runs` as the explicit rerun command so a `failed` or `terminated` active run can create a new `PipelineRun` with fresh `run_id`, `graph_thread_ref`, and `trace_id`, while preserving the old run as history, surfacing `pipeline_run_created` and session-status events, and making the new run boundary visible in workspace/timeline queries.

**Architecture:** Extend `RunLifecycleService` as the AL03 owner for external rerun semantics by adding active-session tail validation, terminal-thread confirmation through the A4.0 runtime boundary, rerun trigger metadata construction, creation of a new `PipelineRun` row plus current-stage/session pointer mutation, top-level `system_status.retry_action` wiring for terminal runs, and event-store updates for `PIPELINE_RUN_CREATED` plus `SESSION_STATUS_CHANGED`. Reuse `RunStateMachine`, `RunSummaryProjection`, `EventStore`, `RuntimeOrchestrationService.assert_thread_terminal_for_rerun()`, and the existing workspace/timeline query model; do not pull in new frontend work, new stage bootstrap behavior beyond the bounded H4.7 slice, or central coordination updates.

**Tech Stack:** Python 3.11+, FastAPI, SQLAlchemy, Pydantic v2, existing multi-SQLite `DatabaseManager`, `RuntimeOrchestrationService`, `AuditService`, `JsonlLogWriter`, `EventStore`, `pytest` through `uv run`.

## Behavior Decisions

- `POST /api/sessions/{sessionId}/runs` is the only external rerun command surface.
- The command targets the current visible session and the current active run tail referenced by `SessionModel.current_run_id`.
- Rerun is valid only when:
  - the session exists and is visible
  - `SessionModel.current_run_id` is non-null
  - the current run exists and belongs to the session
  - the current run status is `failed` or `terminated`
  - the session status matches the current run status projection
  - the old `GraphThreadRef` is terminal according to `RuntimeOrchestrationService.assert_thread_terminal_for_rerun()`
- Rerun is rejected when:
  - there is no active/current run tail
  - the current run is `running`, `paused`, `waiting_clarification`, `waiting_approval`, `waiting_tool_confirmation`, or `completed`
  - the session status and current run status disagree
  - the runtime boundary reports a non-terminal old thread
- On success, rerun creates a new `PipelineRun` with:
  - new `run_id`
  - `session_id` and `project_id` copied from the old run
  - `attempt_index = old_run.attempt_index + 1`
  - `status = running`
  - `trigger_source = retry`
  - `template_snapshot_ref`, `graph_definition_ref`, `workspace_ref`, `runtime_limit_snapshot_ref`, `provider_call_policy_snapshot_ref`, and `delivery_channel_snapshot_ref` copied from the old run
  - new `graph_thread_ref`
  - `current_stage_run_id = None`
  - new `trace_id`
  - `started_at = now`
  - `ended_at = None`
- The old run stays untouched as historical state; rerun does not mutate its status or stage rows.
- `SessionModel.current_run_id` changes to the new run id.
- `SessionModel.status` becomes `running`.
- `SessionModel.latest_stage_type` becomes `requirement_analysis` so the new run boundary is visible immediately in queries even before later stage-owned bootstrap work extends the chain.
- `system_status.retry_action` must be wired on failed/terminated tails to a stable action token `retry:<run_id>` when the run is the current session tail; historical runs remain historical but still carry their previously emitted retry token.
- Rerun does not call `resume_interrupt()` and does not reuse the old `GraphThreadRef`.
- Rerun does not create `RunControlRecord(type=retry)`.
- Rerun does not inherit uncommitted workspace edits conceptually; the persisted `workspace_ref` may stay on the same logical workspace identifier in this slice, but H4.7 must not model the new run as a resume or continuation of the old execution thread.

```python
def _load_current_session_run_tail(
    self,
    session_id: str,
) -> tuple[SessionModel, PipelineRunModel, StageRunModel]:
    session = self._require_control_session().get(SessionModel, session_id)
    if session is None or not session.is_visible:
        raise RunLifecycleServiceError(
            "Session was not found.",
            error_code=ErrorCode.NOT_FOUND,
            status_code=404,
            detail_ref=session_id,
        )
    if session.current_run_id is None:
        self._raise_validation("A rerun requires an existing current run tail.")
    run = self._runtime_session.get(PipelineRunModel, session.current_run_id)
    stage = (
        self._runtime_session.get(StageRunModel, run.current_stage_run_id)
        if run is not None and run.current_stage_run_id is not None
        else None
    )
    if run is None or stage is None:
        raise RunLifecycleServiceError(
            "Active run context was not found.",
            error_code=ErrorCode.NOT_FOUND,
            status_code=404,
            detail_ref=session_id,
        )
    if run.session_id != session.session_id:
        self._raise_validation("Run does not belong to the Session.")
    if stage.run_id != run.run_id:
        self._raise_validation("StageRun does not belong to the run.")
    return session, run, stage
```
