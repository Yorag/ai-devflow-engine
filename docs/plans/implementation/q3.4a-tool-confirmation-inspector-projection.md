> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Implementer and reviewer subagents in this slice must use `gpt-5.5`. Implementer subagents must use `superpowers:test-driven-development`. Fallback to `superpowers:executing-plans` only if subagents cannot be dispatched or this task boundary cannot be passed precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the backend tool-confirmation Inspector projection and `GET /api/tool-confirmations/{toolConfirmationId}` route so the frontend can inspect high-risk tool confirmation details with grouped raw context, process records, outputs, artifacts, and metrics, while keeping clear boundaries from approval and control-item detail semantics.

**Architecture:** Extend the existing read-only `InspectorProjectionService` with `get_tool_confirmation_detail()` plus tool-confirmation-specific section builders that start from `ToolConfirmationRequestModel`, validate the owning visible run/session/project, resolve the stage context, attach the matching `RunControlRecordModel(control_type=tool_confirmation)` as internal linkage only, read stage-scoped `StageArtifactModel` rows and filter them to confirmation-related traces, and inspect matching `tool_confirmation` feed events plus same-stage `stage_node` context. The projection must expose tool-confirmation-specific detail from persisted runtime rows, stage artifacts, and event payloads only; it must not substitute `ApprovalRequest`, `ControlItemInspectorProjection`, or write-side runtime orchestration behavior.

**Tech Stack:** Python 3.11, FastAPI, SQLAlchemy 2.x ORM sessions, Pydantic v2 schemas, pytest through `uv run python -m pytest`.
