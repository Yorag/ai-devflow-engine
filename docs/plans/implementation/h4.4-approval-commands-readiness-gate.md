# H4.4 Approval Commands And Readiness Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:test-driven-development` for every production or test change. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be supplied precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the single public approve/reject command boundary for `solution_design_approval` and `code_review_approval`, including `approval_result`, reject rollback control items, code-review delivery readiness blocking, delivery snapshot solidification, and runtime interrupt resume as one public semantic.

**Architecture:** Extend the existing H4.3 `ApprovalService` with command execution methods that own approval target validation, delivery readiness evaluation, `ApprovalDecisionModel` persistence, `approval_result` projection/event emission, and runtime resume coordination. Add a focused `ControlRecordService` for rollback control-item creation, add a dedicated approvals API route layer with request/response schemas, and keep H4.4 limited to the single approval command entry without introducing tool confirmation, delivery adapters, or new runtime schema.

**Tech Stack:** Python 3.11+, FastAPI, SQLAlchemy, Pydantic v2, existing multi-SQLite `DatabaseManager`, `RuntimeOrchestrationService`, `DeliverySnapshotService`, `AuditService`, `JsonlLogWriter`, `pytest` through `uv run`.

## Behavior Decisions

- The public API stays split by command path only: `POST /api/approvals/{approvalId}/approve` and `POST /api/approvals/{approvalId}/reject`. No secondary code-review approve endpoint is introduced.
- `Approve` request body is empty. `Reject` request body requires `reason` with non-blank content.
- The success response model returns the accepted `approval_result` projection and, for Reject only, the emitted rollback `control_item` projection. This keeps the response aligned with feed semantics and avoids inventing a second frontend-specific approval envelope.
- Delivery readiness blocking returns HTTP `409` with `ErrorResponse` and a `detail_ref` that points to the blocked `approval_id`. The service must also refresh the stored/emitted `approval_request` projection so `delivery_readiness_status`, `delivery_readiness_message`, and `open_settings_action` become non-null for frontend re-query.
- `open_settings_action` is the fixed string `open_delivery_settings` for delivery readiness blocking.
- `Approve` for `solution_design_approval` resumes toward `code_generation`; `Reject` returns to `solution_design`.
- `Approve` for `code_review_approval` resumes toward `delivery_integration`; `Reject` returns to `code_generation`.
- Runtime resume payload uses `RuntimeResumePayload(payload_ref=<decision-id>, values={"decision": "approved" | "rejected", "reason": <reason or null>, "approval_id": <approval-id>, "next_stage_type": <stage-type>})`.
- If runtime resume fails after a decision row, approval result event, or rollback record has been staged but before durable commit, the service must roll back runtime/control/event state and record failure audit/log evidence. External observers must not see a committed approval decision without a committed resume boundary result.

## Log & Audit Integration

- Runtime log category: `LogCategory.RUNTIME`.
- Runtime log source: `services.approvals` for approve/reject flow, `services.control_records` for rollback record creation if a separate source is needed.
- Required runtime log messages:
  - `Approval approve command accepted.`
  - `Approval reject command accepted.`
  - `Approval command rejected.`
  - `Approval command blocked by delivery readiness.`
  - `Approval command failed.`
  - `Rollback control record created.`
- Associated objects: `approval_id`, `decision_id`, `control_record_id`, `run_id`, `stage_run_id`, `graph_interrupt_ref`, `delivery_channel_snapshot_ref`, `approval_type`, `decision`, `next_stage_type`, and trace ids.
- Audit actions:
  - success: `approval.approve`, `approval.reject`
  - paused or validation rejection: `approval.approve.rejected`, `approval.reject.rejected`
  - readiness blocked: `approval.approve.blocked`
  - runtime/internal failure after command acceptance attempt: `approval.approve.failed`, `approval.reject.failed`
- Actor and target contract:
  - `actor_type = AuditActorType.USER`
  - `actor_id = "session-user"`
  - `target_type = "approval_request"`
  - `target_id = approval_id`
- Sensitive data handling:
  - reject reason may appear in domain state and `approval_result`, but log/audit excerpts must still pass through `RedactionPolicy`.
  - delivery readiness metadata may include safe readiness status/message and `env:<NAME>` refs, but never raw credential values.
- Required failure behavior:
  - `AuditService.require_audit_record()` failures must reject or roll back the approval command.
  - Failure-audit and failure-log attempts must be made independently when the command fails after partially staged state.
  - Logs and audit records do not replace `ApprovalDecisionModel`, `RunControlRecordModel`, `approval_result`, or snapshot state as product truth.

- Reject missing approval with `ErrorCode.NOT_FOUND`, status `404`.
- Reject `approval.status is not PENDING` with `ErrorCode.VALIDATION_ERROR`, status `409`.
- Reject paused runs with `ErrorCode.VALIDATION_ERROR`, status `409`, and `APPROVAL_PAUSED_MESSAGE`.
- For non-paused non-waiting runs, reject with `ErrorCode.VALIDATION_ERROR`, status `409`.
- Build child trace spans named `approval-approve-<approval-id>` and `approval-reject-<approval-id>`.
- Create `ApprovalDecisionModel` only after validation, paused check, and code-review readiness gate have passed.
- For `code_review_approval` + `DeliveryMode.GIT_AUTO_DELIVERY`, inspect the current project default delivery channel before decision creation. If readiness is blocked:
  - keep `ApprovalRequestModel.status = pending`
  - do not create `ApprovalDecisionModel`
  - do not emit `approval_result`
  - do not call `prepare_delivery_snapshot()`
  - do not call `resume_interrupt()`
  - raise `ApprovalServiceError` with `error_code = ErrorCode.DELIVERY_SNAPSHOT_NOT_READY`, `status_code = 409`, `detail_ref = approval_id`, and non-null `readiness_status`, `readiness_message`, `open_settings_action`
  - write blocked audit/log evidence
  - refresh the current `approval_request` projection through `build_approval_request_projection()` with populated `delivery_readiness_status`, `delivery_readiness_message`, and `open_settings_action`
- For `demo_delivery`, skip the readiness block path.
- For approve success:
  - set approval status to `approved`
  - resolve `next_stage_type = code_generation` for solution-design approvals and `delivery_integration` for code-review approvals
  - call `prepare_delivery_snapshot()` only for code-review approve, before `approval_result` event append and before runtime resume
  - append `DomainEventType.APPROVAL_APPROVED`
  - append `approval_result` payload using the mapped `SseEventType.APPROVAL_RESULT`
  - call `RuntimeOrchestrationService.resume_interrupt()` with the approval interrupt and decision payload
- For reject success:
  - set approval status to `rejected`
  - create rollback control record through `ControlRecordService.append_rollback_control_item()`
  - append `DomainEventType.APPROVAL_REJECTED`
  - append `approval_result` payload
  - call `resume_interrupt()` with rejected decision payload
- `approval_result` projection requirements:
  - `decision` is `approved` or `rejected`
  - `reason` non-null only on Reject
  - `next_stage_type` matches the semantic next business stage
  - `created_at` equals decision timestamp
- Update `ApprovalRequestModel.resolved_at` and `updated_at` when a command succeeds.
- Use `approval-decision-<uuid4 hex>` for decision ids.
- On any failure after a decision or snapshot has been staged but before durable commit, roll back runtime/control/event sessions, attempt failed audit/log recording, and re-raise `ApprovalServiceError(ErrorCode.INTERNAL_ERROR, APPROVAL_COMMAND_FAILED_MESSAGE, 500)`.
- Reuse `RedactionPolicy` and `LogPayloadSummary` for log payload summarization.
- Do not modify DB schema or add a second code-review approve entry.
