# Q3.4 Control Item Inspector Projection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Implementer and reviewer subagents in this slice must use `gpt-5.4`. Implementer subagents must use `superpowers:test-driven-development`. Fallback to `superpowers:executing-plans` only if subagents cannot be dispatched or this task boundary cannot be passed precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the backend control-item Inspector projection and `GET /api/control-records/{controlRecordId}` route so the frontend can inspect clarification, rollback, and retry control items with grouped raw context, process records, outputs, artifacts, and metrics, without collapsing them into summary-only text and without leaking tool confirmation or system-status semantics.

**Architecture:** Extend the existing read-only `InspectorProjectionService` with `get_control_item_detail()` and control-item section builders that start from `RunControlRecordModel`, validate the owning visible run/session/project, gather the source stage and run context, read related `StageArtifactModel` rows, inspect stage-scoped domain events for `control_item` and `stage_node` payloads, and attach directly related clarification records when present. The projection must enforce the current semantic boundary: `tool_confirmation` control records remain internal linkage for Q3.4a and must not be returned as `ControlItemInspectorProjection`, while `system_status` remains outside `control_item` entirely.

**Tech Stack:** Python 3.11, FastAPI, SQLAlchemy 2.x ORM sessions, Pydantic v2 schemas, pytest through `uv run python -m pytest`.
