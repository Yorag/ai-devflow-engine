# C2.7a Configuration Package Import Export Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:executing-plans` only if subagents cannot be scheduled, the task boundary cannot be precisely assigned, or review subagents are unavailable. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement project-scoped user-visible configuration package export and import through the control-plane API.

**Architecture:** Add a `ConfigurationPackageService` that exports and imports only the current Project's user-visible Provider, default DeliveryChannel, and user-template runtime configuration. The service reuses existing Provider, DeliveryChannel, and Template validation rules by constructing their write request schemas and invoking non-committing validation/apply helpers before one final transaction commit. It exposes `GET /api/projects/{projectId}/configuration-package/export` and `POST /api/projects/{projectId}/configuration-package/import` from the existing Project router with API/OpenAPI tests and log/audit coverage.

**Tech Stack:** FastAPI, Pydantic v2, SQLAlchemy 2.x, pytest, repository-local `uv run --no-sync`.

## Scope

In scope:
- Export the current visible Project scope as `ConfigurationPackageExport`.
- Include all visible formal Provider records, the current Project default DeliveryChannel, and current `user_template` runtime configuration.
- Exclude real credential values, platform hidden runtime settings, `compression_threshold_ratio`, system prompt asset bodies, runtime snapshots, historical run data, logs, audit bodies, and platform database paths.
- Import a `ConfigurationPackageImportRequest` only when `package_schema_version` is supported and `scope.scope_type = "project"` with `scope.project_id` equal to the target route Project id.
- Import Providers by reusing `ProviderWriteRequest` validation and Provider service rules without calling Provider service methods that commit internally. Built-in Provider identity is immutable, custom Provider is created when missing and patched when existing, and per-model capabilities are defaulted by the Provider service validation helper.
- Import DeliveryChannel by reusing `ProjectDeliveryChannelUpdateRequest` and DeliveryChannel service validation/apply helpers without calling `DeliveryChannelService.update_project_channel()`, because that public method commits internally.
- Import `user_template` runtime configuration by reusing `PipelineTemplateWriteRequest` and Template service validation helpers without calling `TemplateService.patch_user_template()` or `TemplateService.save_as_user_template()`, because those public methods commit internally.
- Reject `system_template` package entries with a field-level error because C2.4 and the backend spec forbid direct system template overwrite; users must import modified template runtime configuration as `user_template`.
- Return `ConfigurationPackageImportResult` with generated import `package_id`, `package_schema_version`, `summary`, `changed_objects`, and optional `field_errors`.
- Each changed object includes `config_version`, using the saved row's `updated_at.isoformat()` value after apply. This satisfies C2.7a's configuration version requirement without adding a new migration or C2.8 runtime settings version field.
- Return stable field-level errors for unsupported version, scope mismatch, invalid Provider, invalid DeliveryChannel, and invalid template runtime config. Missing Project remains the route-level/service-level unified 404 error contract, not a field-level package validation result.
- Write sanitized runtime log and audit records for successful and rejected import/export commands.
- Assert the two routes, request/response schemas, field errors, and primary error responses in `/api/openapi.json`.

Out of scope:
- C2.8 `PlatformRuntimeSettings` management.
- Runtime snapshot creation or mutation.
- Run start, Delivery Integration, final approval, or delivery snapshot gate behavior.
- Frontend UI changes or frontend client changes.
- Database migrations and dependency changes.
- Environment variable writes or credential value reads beyond existing `api_key_ref` / `credential_ref` reference validation.
- Git write actions.

## Behavior Details

Add these response schemas:

```python
class ConfigurationPackageChangedObject(_StrictBaseModel):
    object_type: Literal["provider", "delivery_channel", "pipeline_template"]
    object_id: str = Field(min_length=1)
    action: Literal["created", "updated", "unchanged"]
    config_version: str = Field(min_length=1)


class ConfigurationPackageFieldError(_StrictBaseModel):
    field: str = Field(min_length=1)
    message: str = Field(min_length=1)


class ConfigurationPackageImportResult(_StrictBaseModel):
    package_id: str = Field(min_length=1)
    package_schema_version: str = Field(min_length=1)
    summary: str = Field(min_length=1)
    changed_objects: list[ConfigurationPackageChangedObject] = Field(default_factory=list)
    field_errors: list[ConfigurationPackageFieldError] = Field(default_factory=list)
```

Use `PACKAGE_SCHEMA_VERSION = "function-one-config-v1"` for C2.7a service exports and imports. Existing schema tests that use `"function-one-v1"` only prove Pydantic shape, not the service compatibility version.

`ConfigurationPackageImportResult.package_id` is an import processing id generated by the service as `config-import-<uuid>`. It is not read from `ConfigurationPackageImportRequest`, because the current package request schema has no package id and frontend callers already treat the response field as optional operation metadata. Rejected parsed packages still receive an import processing id so field-error responses are traceable.

`ConfigurationPackageService.export_project_package(project_id, trace_context)` behavior:
- Requires `audit_service` and `log_writer`.
- Loads the visible Project by id. Missing or invisible Project raises `ConfigurationPackageServiceError(ErrorCode.NOT_FOUND, "Project was not found.", 404)` and records `configuration_package.export.rejected`.
- Calls `ProviderService.seed_builtin_providers()` before reading Providers so builtin defaults exist in formal storage.
- Exports Providers ordered by builtin order then custom creation/id order. Provider entries include references such as `api_key_ref`, not secret values.
- Exports the Project default DeliveryChannel as a single package delivery channel. It includes `credential_ref` reference only; if an unsafe legacy raw credential is present, export must use `[blocked:credential_ref]` and must not leak the raw value.
- Exports only `user_template` PipelineTemplates, ordered by creation time and id. Template entries include `template_id`, `name`, `template_source`, `stage_role_bindings`, `auto_regression_enabled`, and `max_auto_regression_retries`. They do not include `base_template_id`, `fixed_stage_sequence`, `approval_checkpoints`, `role_name`, prompt asset ids, system prompt asset bodies, stage contracts, or tool permissions. System templates are seeded platform assets and are not exported in C2.7a because their prompt bodies originate from system built-in prompt assets.
- Returns `ConfigurationPackageExport(export_id="config-export-<uuid>", exported_at=<now>, package_schema_version=PACKAGE_SCHEMA_VERSION, scope={"scope_type": "project", "project_id": project_id}, ...)`.
- Writes `configuration_package.export` audit with counts and object ids only. Metadata must not include prompt text, base URLs, raw secret values, log bodies, or audit bodies.
- Writes service JSONL with `LogCategory.API`, source `services.configuration_packages`, payload type `configuration_package_export`, and a sanitized count/object summary.

`ConfigurationPackageService.import_project_package(project_id, package, trace_context)` behavior:
- Requires `audit_service` and `log_writer`.
- Loads the visible Project. Missing/invisible Project returns a `not_found` service error and records `configuration_package.import.rejected`.
- Validates `package.package_schema_version == PACKAGE_SCHEMA_VERSION`. Unsupported versions return `ConfigurationPackageImportResult` with a `field_errors` item `{field: "package_schema_version", message: "Unsupported configuration package schema version."}` and audit action `configuration_package.import.rejected`; no configuration rows change.
- Validates `package.scope.scope_type == "project"` and `package.scope.project_id == project_id`. Scope mismatch returns `field_errors` item `{field: "scope.project_id", message: "Configuration package scope does not match the target Project."}` and no row changes.
- Performs a validate-then-apply import. The validation phase constructs every `ProviderWriteRequest`, `ProjectDeliveryChannelUpdateRequest`, and `PipelineTemplateWriteRequest`, invokes the existing non-committing validation helpers, and records planned model changes without mutating rows. If any package item fails validation, the service returns field errors and calls `session.rollback()` before any row change.
- Applies each Provider entry through transaction-aware direct model writes after validation succeeds.
  - Built-in Provider entries must preserve `provider_id`, `provider_source`, and protocol identity through existing Provider service validation.
  - Custom Provider entries patch an existing custom Provider with the same id when present. Missing custom Provider entries create a custom Provider row using the package `provider_id` when available and not already used. The changed object id is the actual saved Provider id.
- Applies the DeliveryChannel entry through transaction-aware direct model writes after validation succeeds. C2.7a accepts exactly one delivery channel entry for the target Project. More than one entry returns `{field: "delivery_channels", message: "Configuration package must contain at most one project DeliveryChannel."}` and no row changes.
- Applies PipelineTemplate entries through transaction-aware direct model writes after validation succeeds.
  - Existing user template: patch it in place.
  - Missing user template or missing custom package template id: create a new `user_template` row using the package `template_id` when available and not already used.
  - Any `template_source = system_template` package entry returns `{field: "pipeline_templates[N].template_source", message: "System templates cannot be overwritten by configuration package import."}` and no row changes.
- The import path must not call public service write methods that commit internally: `ProviderService.patch_builtin_provider_runtime_config()`, `ProviderService.patch_custom_provider()`, `ProviderService.create_custom_provider()`, `DeliveryChannelService.update_project_channel()`, `TemplateService.patch_user_template()`, or `TemplateService.save_as_user_template()`.
- On any validation error, return `ConfigurationPackageImportResult` with field errors instead of partially applying valid entries. Error fields use deterministic package paths such as `providers[0].default_model_id`, `delivery_channels[0].credential_ref`, or `pipeline_templates[0].stage_role_bindings`.
- On success, commit after runtime log and audit writes complete, and return `summary = "Imported <N> configuration objects."` where `<N>` is the count of created or updated changed objects. Unchanged objects may be listed with action `unchanged` but do not contribute to `<N>`.
- Changed object `config_version` values are the saved `updated_at.isoformat()` timestamps of Provider, DeliveryChannel, and PipelineTemplate rows after the import transaction is flushed.
- On audit or log failure, roll back the control write and propagate the failure.
- Never touches runtime snapshot tables.

Do not add a system-template overwrite helper. C2.7a must preserve the C2.4 contract that system templates can be selected or saved as user templates, but cannot be directly overwritten.

## Log & Audit Integration

- Runtime log categories:
  - Export: `LogCategory.API`, source `services.configuration_packages`, message `Configuration package exported.`
  - Import: `LogCategory.API`, source `services.configuration_packages`, message `Configuration package import processed.`
- Audit actions:
  - Success: `configuration_package.export`, `configuration_package.import`
  - Rejected: `configuration_package.export.rejected`, `configuration_package.import.rejected`
  - Failed: `configuration_package.export.failed`, `configuration_package.import.failed` when an unexpected service failure happens after target resolution and the audit ledger remains available.
- Actor: `AuditActorType.USER`, `actor_id="api-user"`.
- Target: `target_type="project"`, `target_id=<project_id>`.
- Trace fields: API routes pass `get_trace_context()` into the service. JSONL and audit records inherit `request_id`, `trace_id`, `correlation_id`, `span_id`, and `parent_span_id`.
- Redaction: service log and audit metadata include package id, project id, counts, object ids, changed object actions, and field names. They must not include raw secret values, credential values, Provider base URLs, prompt bodies, system prompt asset bodies, run snapshots, logs, audit bodies, or database paths.
- Failure behavior: audit or service-log failure raises and rolls back the control write. A failed import with field errors is a rejected command result and returns HTTP 200 with `field_errors`, because the package was parsed but not applied.
- Product truth: log/audit records are observability facts only. The formal config rows remain the source of truth. The service does not read or write runtime snapshots.
