> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. **Goal:** Implement `POST /api/runs/{runId}/terminate` so the current active run can terminate through the A4.0 runtime boundary, mark the active stage and run terminal, append a top-level terminal `system_status` entry, and leave historical approval/tool-confirmation artifacts readable but non-actionable.

**Architecture:** Extend `RunLifecycleService` as the H4.6 owner for terminate semantics by combining active-run validation, runtime-thread termination, terminal run/stage/session mutation, unified terminal `system_status` projection, pending tool-confirmation cancellation reuse, pending approval refresh, and audit/log integration. Reuse the existing `RuntimeOrchestrationService`, event store projection matrix, route wiring from H4.5, and feed-entry schemas; do not introduce new control-item types or rerun behavior.

**Tech Stack:** Python 3.11+, FastAPI, SQLAlchemy, Pydantic v2, existing multi-SQLite `DatabaseManager`, `RuntimeOrchestrationService`, `AuditService`, `JsonlLogWriter`, `EventStore`, `ToolConfirmationService`, `pytest` through `uv run`.

## Source Trace Resolution

No blocking source-trace conflict is present.

- Current specs and split-plan align that `system_status` is a top-level terminal feed entry for `failed` and `terminated` runs only.
- Existing event-matrix rules already map `RUN_FAILED` and `RUN_TERMINATED` to `system_status`; the missing implementation is service-side payload generation and terminate command orchestration.
- Workspace and timeline projections primarily replay event payloads instead of recomputing approval/tool-confirmation actionability from current run status. Therefore H4.6 must explicitly refresh pending `approval_request` events and cancel pending `tool_confirmation` requests during terminate rather than assuming projections will infer terminal disablement.

## Behavior Decisions

- `terminate` is valid only for the current active run whose `SessionModel.current_run_id` matches the target `runId`.
- `terminate` is allowed only from active, non-terminal runtime-control states:
  - `running`
  - `waiting_clarification`
  - `waiting_approval`
  - `waiting_tool_confirmation`
  - `paused`
- `terminate` rejects `completed`, `failed`, and `terminated`.
- `terminate` calls `RuntimeOrchestrationService.terminate_thread()` before committing terminal local state.
- On successful terminate:
  - `PipelineRun.status = terminated`
  - `PipelineRun.ended_at = now`
  - `PipelineRun.updated_at = now`
  - `StageRun.status = terminated` only for the current active stage
  - `StageRun.ended_at = now`
  - `StageRun.updated_at = now`
  - `SessionModel.status = terminated`
  - `SessionModel.updated_at = now`
  - `SessionModel.current_run_id` remains unchanged so current-activity/history semantics stay anchored to the terminated run until H4.7 creates a new run
- `terminate` does not delete, resolve, or otherwise rewrite existing clarification objects, approval rows, completed stage artifacts, or historical records.
- Pending approval on the terminated run stays `ApprovalRequest.status = pending`, but the latest top-level `approval_request` event must be refreshed with:
  - `is_actionable = false`
  - terminal `disabled_reason`
- Pending tool confirmation on the terminated run must be cancelled through the existing H4.4a cancellation helper:
  - `ToolConfirmationRequest.status = cancelled`
  - `user_decision = None`
  - no allow/deny result event is emitted
  - the latest top-level `tool_confirmation` event remains historical/non-actionable
- `TerminalStatusProjector.append_terminal_system_status()` is the only H4.6 path allowed to construct terminal `system_status` payloads. H4.6 terminate and existing/future run-failure paths must converge on that helper rather than hand-building payloads in multiple places.
- The H4.6 terminated `system_status` payload must:
  - use `status = terminated`
  - be appended as a top-level feed entry, not `control_item`
  - preserve a free-text reason that explains termination
  - leave `retry_action` unset in H4.6; explicit rerun command wiring belongs to H4.7

```python
@dataclass(frozen=True)
class TerminalStatusProjector:
    events: EventStore
    now: Callable[[], datetime]

    def append_terminal_system_status(
        self,
        *,
        domain_event_type: DomainEventType,
        run: PipelineRunModel,
        title: str,
        reason: str,
        trace_context: TraceContext,
        retry_action: str | None = None,
        occurred_at: datetime | None = None,
    ) -> None:
        timestamp = occurred_at or self.now()
        entry = SystemStatusFeedEntry(
            entry_id=f"entry-system-status-{run.run_id}-{domain_event_type.value.lower()}",
            run_id=run.run_id,
            occurred_at=timestamp,
            status=(
                RunStatus.TERMINATED
                if domain_event_type is DomainEventType.RUN_TERMINATED
                else RunStatus.FAILED
            ),
            title=title,
            reason=reason,
            retry_action=retry_action,
        )
        self.events.append(
            domain_event_type,
            payload={"system_status": entry.model_dump(mode="json")},
            trace_context=trace_context,
            occurred_at=timestamp,
        )
```

```python
def _assert_can_terminate(
    self,
    *,
    session: SessionModel,
    run: PipelineRunModel,
) -> None:
    if run.status not in {
        RunStatus.RUNNING,
        RunStatus.WAITING_CLARIFICATION,
        RunStatus.WAITING_APPROVAL,
        RunStatus.WAITING_TOOL_CONFIRMATION,
        RunStatus.PAUSED,
    }:
        self._raise_validation("Run cannot be terminated from its current status.")
    expected_session = (
        SessionStatus.PAUSED
        if run.status is RunStatus.PAUSED
        else RunStateMachine.project_session_status(run.status)
    )
    if session.status is not expected_session:
        self._raise_validation("Session status must match run status before terminate.")
```

```python
def mark_terminal(
    self,
    *,
    session: SessionModel,
    run: PipelineRunModel,
    stage: StageRunModel,
    terminal_status: RunStatus,
    occurred_at: datetime,
) -> None:
    run.status = terminal_status
    run.ended_at = occurred_at
    run.updated_at = occurred_at
    stage.status = StageStatus.TERMINATED
    stage.ended_at = occurred_at
    stage.updated_at = occurred_at
    session.status = SessionStatus(terminal_status.value)
    session.updated_at = occurred_at
    self._runtime_session.add_all([run, stage])
    self._require_control_session().add(session)
```

Terminate mutation flow requirements:
- Accept audit first with `action="runtime.terminate"` and rollback hook `self._rollback_all`.
- Call `self._require_runtime_orchestration().terminate_thread(...)` using:
  - `paused=True` thread-ref status only when current run status is `paused`
  - otherwise the current run status
- If run status before terminate is `waiting_tool_confirmation`, cancel pending confirmations by invoking the existing H4.4a helper/service instead of hand-mutating confirmation rows.
- If run status before terminate is `waiting_approval`, refresh the latest pending approval-request payload with `is_actionable=False` and a terminal disabled reason.
- After local terminal mutation, append `DomainEventType.RUN_TERMINATED` through `TerminalStatusProjector.append_terminal_system_status(...)`.
- Commit runtime/control/event sessions together through the existing `_commit_all()`.
- Emit H4.6-specific accepted/completed logs and reuse rejected/failed helpers for invalid-state/runtime-failure paths.

```python
def _refresh_pending_wait_entry_for_terminate(
    self,
    *,
    status_before: RunStatus,
    run: PipelineRunModel,
    trace_context: TraceContext,
    occurred_at: datetime,
) -> None:
    if status_before is RunStatus.WAITING_APPROVAL:

                trace_context=trace_context,
                occurred_at=occurred_at,
```
