**Goal:** Implement `ArtifactStore` so `StageArtifact` rows become the stable runtime container for stage input, process, output, artifact references, and metrics, while `StageRun.input_ref` and `StageRun.output_ref` point at those persisted artifacts instead of raw transient state.

**Architecture:** Reuse the approved generic `StageArtifactModel` from `C1.7` and add a focused runtime service that writes and mutates stage-scoped artifact rows inside `runtime.db`. The service owns three things only: creating the input artifact and binding `StageRun.input_ref`, appending standardized process content and metrics to the same artifact row, and completing the output artifact while binding `StageRun.output_ref`; runtime logs record redacted/truncated summaries of these writes, but the authoritative stage facts remain in `StageArtifactModel.process`, `StageArtifactModel.metrics`, and stable refs on `StageRun`.

**Tech Stack:** Python 3.11+, SQLAlchemy ORM, runtime.db models, `TraceContext`, `RedactionPolicy`, JSONL run-log writer integration, pytest via `uv run python -m pytest`.

**Execution Status:** Local worker result `reported`; expected committed ingest result `implemented` after a user-approved checkpoint commit.

## Design Decisions

- `ArtifactStore` stays inside the approved generic `StageArtifactModel` boundary:
  - `process` remains the single persisted JSON container for `input_snapshot`, appended process records, `output_snapshot`, attachment refs, and other stable references.
  - `metrics` remains the single persisted JSON container for stage metrics.
  - `payload_ref` remains the stable external reference for the artifact payload or typed output object.
- `create_stage_input()` creates the canonical input artifact row for a given stage and sets `StageRun.input_ref` to the created `artifact_id`.
- `append_process_record()` merges one named process record or record list into the existing artifact `process` payload without dropping prior persisted keys.
- `complete_stage_output()` writes the output snapshot and output stable refs into the artifact row, updates the artifact `payload_ref` when needed, and sets `StageRun.output_ref` to that same `artifact_id`.
- `attach_metric_set()` merges only the provided metric keys into the artifact `metrics` JSON without clearing previously recorded metrics.
- `get_stage_artifact()` returns the persisted `StageArtifactModel` for stable downstream reads and raises a slice-specific error if the referenced artifact is missing or the runtime store is unavailable.
- Runtime logs are summary-only:
  - artifact writes remain authoritative in `runtime.db`
  - redaction/truncation affects only logs
  - log/redaction failures must not erase or roll back already-persisted artifact truth
- Because the split-plan file list is minimal and C1.7 already approved the schema, this slice prefers no schema change and no new repository abstraction unless review proves one is required.

## Log & Audit Integration

This slice changes runtime artifact persistence, so implementation must include runtime-log integration:

- Log at least these write actions:
  - `create_stage_input`
  - `append_process_record`
  - `complete_stage_output`
  - `attach_metric_set`
- Use `LogCategory.RUNTIME`.
- Include stable metadata at minimum:
  - `run_id`
  - `stage_run_id`
  - `artifact_id`
  - `artifact_type`
  - `payload_ref`
  - action name
  - changed process keys or metric keys
- Use `RedactionPolicy.summarize_payload()` so long text and sensitive fields are summarized/truncated in logs instead of written raw.
- If redaction or run-log writing fails, keep persisted artifact changes intact and swallow the observability failure after logging locally, matching the non-fatal runtime-service pattern already used in AL01.
- No new audit log contract is required for this internal runtime storage slice.
