## Source Trace Conflict Gate

V6.3 live completion requires a browser scenario that can drive the frontend against real backend APIs into these states:

- `waiting_approval` with actionable `approval_request`
- approval `reject` with `approval_result` and rollback `control_item`
- `waiting_tool_confirmation` with actionable `tool_confirmation`
- tool allow and deny, including stable deny follow-up projection
- pause/resume while waiting at approval and tool confirmation checkpoints
- terminate and rerun from terminal `system_status`
- workspace projection and SSE updates that reflect the real persisted events

Current source facts:

- `POST /api/sessions/{sessionId}/messages` starts the first run and creates only the initial `requirement_analysis` `StageRun`; backend API regression asserts the returned session is `running` with `latest_stage_type=requirement_analysis`.
- The HTTP route layer has command routes for session messages, run pause/resume/terminate, approval approve/reject, tool confirmation allow/deny, workspace query, and SSE stream.
- The only observed path that advances a run into `solution_design` approval and `code_generation` tool-confirmation interrupts is the Python test helper `_advance_until_interrupt_or_stage_result`, which directly constructs `DeterministicRuntimeEngine` and calls `engine.run_next(...)`.
- The existing Playwright config starts only the frontend Vite server by default. It can target an externally supplied `E2E_BASE_URL`, but it does not start or bind a backend deterministic runtime advancement harness.
- The existing `e2e/tests/function-one-control-flow.spec.ts` installs route-level API fixtures and replaces `EventSource`; it is intentionally mock-ready coverage, not live backend coverage.

Gate decision: full live V6.3 implementation is blocked in this QA-E2E lane. Adding a backend route, dev-only runtime advancement endpoint, or production harness would cross the runtime/API owner boundary and change backend behavior outside QA-E2E ownership. Adding another route fixture would duplicate the prior `mock_ready` checkpoint and would not satisfy the live claim.
