# A4.4 Deterministic Terminal Control Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be precisely constrained. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add configurable completed, failed, and terminated terminal paths to the deterministic test runtime without creating DeliveryRecord or demo delivery behavior.

**Architecture:** Extend `DeterministicRuntimeEngine` with a small terminal-control configuration and terminal helpers. Configured failure or termination is evaluated after the source `StageRun` and deterministic `StageArtifact` exist but before the stage is completed, so the terminal record can cite the direct source stage and artifact. Failed and terminated terminal feed entries are appended through the existing `TerminalStatusProjector`; completed terminal state only updates run/session state and emits the existing session-status event semantics.

**Tech Stack:** Python 3.11+, SQLAlchemy sessions, Pydantic v2, pytest, existing backend runtime models, event store, `TerminalStatusProjector`, runtime orchestration boundary, and log writer.

## Log & Audit Integration

A4.4 changes runtime node behavior and terminal run lifecycle behavior.

- Runtime log source: `runtime.deterministic`.
- Runtime log category: `LogCategory.RUNTIME`.
- Terminal actions:
  - `deterministic_terminal_completed`
  - `deterministic_terminal_failed`
  - `deterministic_terminal_terminated`
- Associated objects: `session_id`, `run_id`, `stage_run_id` when available, `source_stage_type` when available, `graph_thread_id`, artifact refs, domain event refs, result ref, direct failure point, and retry clue.
- Failed and terminated logs must include enough information to locate the direct failure or termination point and construct the later rerun relationship: `terminal_reason`, `direct_failure_point`, `source_stage_type`, and `retry_action`.
- Log payloads must not include raw graph state, credentials, API keys, authorization headers, cookies, or unbounded tool/model output.
- Runtime log write failure must not erase persisted `PipelineRun`, `StageRun`, `Session`, `StageArtifact`, domain event, or `system_status` facts.
- Audit entries are not added in this slice because these are deterministic runtime fixture outcomes, not user command routes or high-impact external side effects. User-triggered terminate audit remains owned by H4.6 `RunLifecycleService`.
