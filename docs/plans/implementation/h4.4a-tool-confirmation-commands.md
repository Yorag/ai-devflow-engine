# H4.4a Tool Confirmation Commands Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:test-driven-development` for every production or test change. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be supplied precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement `ToolConfirmationRequest` creation and allow/deny command handling so high-risk tool confirmation becomes a distinct runtime permission boundary with auditable allow/deny outcomes, stable `waiting_tool_confirmation` semantics, and a single backend API surface.

**Architecture:** Add a focused `ToolConfirmationService` that owns request creation, command validation, status transitions, top-level `tool_confirmation` projection building, runtime resume coordination, terminal-run cancellation, and tool-confirmation-specific audit/log semantics. Reuse the existing runtime models, event matrix, `RuntimeOrchestrationService`, control-record persistence, and approval-route dependency style, while keeping H4.4a strictly separate from approval semantics, pause/resume commands, deterministic runtime execution, and tool-risk classification.

**Tech Stack:** Python 3.11+, FastAPI, SQLAlchemy, Pydantic v2, existing multi-SQLite `DatabaseManager`, `RuntimeOrchestrationService`, `AuditService`, `JsonlLogWriter`, `pytest` through `uv run`.

## Behavior Decisions

- `ToolConfirmationService.create_request()` is the only H4.4a write-side surface for creating `ToolConfirmationRequestModel`, `RunControlRecordModel(control_type=tool_confirmation)`, `GraphInterruptRef(type=tool_confirmation)`, and the top-level `tool_confirmation_requested` event payload.
- `ToolConfirmationService.allow()` and `deny()` are the only public command methods for user submission.
- `POST /api/tool-confirmations/{toolConfirmationId}/allow` accepts an empty request body. `POST /api/tool-confirmations/{toolConfirmationId}/deny` accepts an optional operator reason field that augments, but does not replace, the stored risk rationale.
- `ToolConfirmationFeedEntry.allow_action` and `deny_action` stay machine-readable and local to the specific confirmation id. Allow must authorize only the specific `tool_confirmation_id` and covered `confirmation_object_ref`; it must not create any wildcard semantics.
- A pending confirmation is actionable only when:
  - request status is `pending`
  - run/session/stage status are `waiting_tool_confirmation`
  - the referenced run is the current active run
  - the run is not paused, failed, terminated, or completed
- On allow:
  - `ToolConfirmationRequestModel.status = allowed`
  - `ToolConfirmationRequestModel.user_decision = allowed`
  - `responded_at` is set
  - the emitted top-level `tool_confirmation` payload includes `decision = allowed`, `responded_at`, and `is_actionable = false`
  - runtime resumes through `RuntimeOrchestrationService.resume_tool_confirmation()`
- On deny:
  - `ToolConfirmationRequestModel.status = denied`
  - `ToolConfirmationRequestModel.user_decision = denied`
  - `responded_at` is set
  - the emitted top-level `tool_confirmation` payload includes `decision = denied`, `responded_at`, and `is_actionable = false`
  - runtime also resumes through `resume_tool_confirmation()` with a structured deny payload so later runtime slices can branch to an alternative path or structured failure
- On terminal-run cancellation:
  - only pending confirmations can be cancelled
  - `status = cancelled`
  - `user_decision` remains `None`
  - no allow/deny result event is emitted
  - returned projection is historical, non-actionable, and preserves risk details
- Paused, completed, failed, and terminated runs reject allow/deny submissions with tool-confirmation-specific command errors and no mutation.
- H4.4a must not create `ApprovalRequest`, `ApprovalDecision`, `approval_result`, or rollback control-item semantics.
- `RunControlRecordModel(control_type=tool_confirmation)` is stored only as process trace and linkage; it must not surface as a top-level `control_item` feed entry.

- Runtime log category: `LogCategory.RUNTIME`

- Required runtime log messages:
  - `Tool confirmation request created.`
  - `Tool confirmation command accepted.`
  - `Tool confirmation command rejected.`
  - `Tool confirmation command failed.`
  - `Tool confirmation cancelled for terminal run.`
- Audit actions:
  - request creation: `tool_confirmation.request`
  - allow success: `tool_confirmation.allow`
  - deny success: `tool_confirmation.deny`
  - paused/history/terminal rejection: `tool_confirmation.allow.rejected` or `tool_confirmation.deny.rejected`
  - failure after staging state: `tool_confirmation.allow.failed` or `tool_confirmation.deny.failed`
  - cancellation: `tool_confirmation.cancel`
- Associated object identifiers:
  - `tool_confirmation_id`
  - `confirmation_object_ref`
  - `run_id`
  - `stage_run_id`
  - `graph_interrupt_ref`
  - `tool_name`
  - `risk_level`
  - `risk_categories`
  - trace ids
- Sensitive-data handling:
  - `command_preview`, target summaries, and operator deny reason must pass through `RedactionPolicy`
  - logs/audit may keep safe excerpts and stable refs, never raw credentials or unbounded payloads
- Failure behavior:
  - `AuditService.require_audit_record()` failures must reject or roll back request creation and allow/deny commands
  - audit/log records do not replace `ToolConfirmationRequestModel`, domain events, or feed projections as product truth

from __future__ import annotations

from pydantic import BaseModel, Field, field_validator

from backend.app.schemas.feed import ToolConfirmationFeedEntry


class ToolConfirmationAllowRequest(BaseModel):
    pass


class ToolConfirmationDenyRequest(BaseModel):
    reason: str | None = Field(default=None)

    @field_validator("reason")
    @classmethod
    def validate_reason(cls, value: str | None) -> str | None:
        if value is None:
            return value
        stripped = value.strip()
        if not stripped:
