# A4.0 Runtime Orchestration Boundary Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task when explicit subagent delegation is available. Fallback to `superpowers:executing-plans` only when subagent delegation is not explicitly available, task context cannot be safely limited, or the current worker prompt restricts execution to this session. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Define the runtime orchestration boundary and fake-able ports used by clarification, approval, tool confirmation, pause, resume, terminate, and rerun commands.

**Architecture:** Add strict domain reference models in `backend/app/domain/runtime_refs.py` and a boundary service in `backend/app/services/runtime_orchestration.py`. The service delegates all graph operations through `RuntimeCommandPort` and `CheckpointPort`, returns stable references, derives child `TraceContext` spans, and never reads or writes raw graph state.

**Tech Stack:** Python 3.11+, Pydantic v2, `typing.Protocol`, existing `TraceContext`, existing runtime/domain enums, `pytest` through `uv run`.

## Log & Audit Integration

A4.0 does not write runtime logs or audit records directly because it is a boundary and fake-able port slice. It still satisfies the trace portion of Log & Audit Integration:

- Every service method accepts `trace_context: TraceContext`.
- Every service method passes a child `TraceContext` to the port layer.
- Child spans inherit `request_id`, `trace_id`, and `correlation_id`.
- Child spans attach available object identifiers such as `run_id`, `stage_run_id`, `approval_id`, `tool_confirmation_id`, and `graph_thread_id`.
- Tests prove trace metadata is propagated to ports and not used as product truth.
- Later command slices own audit acceptance/rejection semantics and must call this boundary after their own audit gate.
