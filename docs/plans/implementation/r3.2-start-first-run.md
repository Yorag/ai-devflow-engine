**Goal:** Implement `POST /api/sessions/{sessionId}/messages` `new_requirement` semantics so the first user requirement on a draft session atomically creates the first `PipelineRun`, freezes all required run snapshots, compiles and persists the initial graph definition/thread state, appends the first user message event, and creates the initial `requirement_analysis` stage run.

**Architecture:** Extend the existing session message entrypoint to dispatch between `clarification_reply` and `new_requirement`, then add a focused first-run startup path that composes existing C2.8 runtime settings, R3.4/R3.4a/R3.4b snapshot builders, R3.5 `GraphCompiler`, `EventStore`, and `StageRunService`. Persist runtime rows in `runtime.db`, graph rows in `graph.db`, control session state in `control.db`, and user-message / pipeline-run events in `event.db`; keep logs and audit facts as observability facts only, not product truth.

## Design Decisions

- Keep `/api/sessions/{sessionId}/messages` as the single route for both supported message semantics. Dispatch in `SessionService` rather than adding a second route, because the spec fixes this entrypoint.
- Preserve the existing `clarification_reply` flow unchanged except for routing through the new dispatcher. Do not widen its contract or event semantics.
- Model the first user requirement as a `MessageFeedEntry(type=user_message)` event payload so workspace/timeline rebuild uses the same top-level feed truth as later clarification replies.
- Use the existing `RunSummaryProjection` contract for `pipeline_run_created` event payloads and for the route response.
- Add a narrow graph persistence repository under `backend/app/repositories/graph.py` for `GraphDefinitionModel` and `GraphThreadModel`. This avoids pushing graph-db concerns into `RunLifecycleService` or overloading `RuntimeSnapshotRepository`.
- `RunLifecycleService.start_first_run()` owns the orchestration boundary for first-run creation. `SessionService.start_run_from_new_requirement()` loads and validates control-plane session/template context, then calls into the run lifecycle service with already-open control/runtime/event/graph/log resources.
- The initial `workspace_ref` is a deterministic minimal legal placeholder in AL01 owner scope, for example `workspace-{run_id}`. It must only be a stable reference, not real workspace allocation; W5.1 later owns actual workspace manager creation semantics.
- The initial `graph_thread_ref` is the persisted `GraphThreadModel.graph_thread_id`. Use a deterministic bounded id `graph-thread-{run_id}` with a SHA fallback helper if needed.
- The initial `StageRun` must be `requirement_analysis`, `status=running`, `attempt_index=1`, `graph_node_key=requirement_analysis`, and `stage_contract_ref=requirement_analysis`.
- Reuse existing C2.8, R3.4, R3.4a, R3.4b, and R3.5 components. `R3.2` must not create a second graph compiler truth, infer internal model bindings from template/provider defaults, or derive runtime limits from constants or environment state.
- Error mapping must stay on the existing stable error code set:
  - non-draft or already-started session: `validation_error`, `409`
  - missing session: `not_found`, `404`
  - invalid/missing persisted runtime settings or snapshot build failures: use the originating stable config error code and mapped status
  - graph persistence or runtime persistence failure: `config_storage_unavailable`, `503`
  - unexpected orchestration failure after acceptance: `internal_error`, `500`

def test_post_session_message_new_requirement_starts_first_run_and_returns_first_user_message(
    tmp_path: Path,
) -> None:
    app = build_app(tmp_path)

    with TestClient(app) as client:
        created = create_draft_session(client)
        response = client.post(
            f"/api/sessions/{created['session_id']}/messages",
            json={
                "message_type": "new_requirement",
                "content": "Implement workspace projection startup.",
            },
            headers={
                "X-Request-ID": "req-new-requirement",
                "X-Correlation-ID": "corr-new-requirement",
            },
        )

    assert response.status_code == 200
    body = response.json()
    assert body["session"]["session_id"] == created["session_id"]
    assert body["session"]["status"] == "running"
    assert body["session"]["current_run_id"]
    assert body["session"]["latest_stage_type"] == "requirement_analysis"
    assert body["message_item"]["type"] == "user_message"
    assert body["message_item"]["author"] == "user"
    assert body["message_item"]["content"] == "Implement workspace projection startup."
    assert body["message_item"]["run_id"] == body["session"]["current_run_id"]
    assert body["message_item"]["stage_run_id"]

    run_id = body["session"]["current_run_id"]

    with app.state.database_manager.session(DatabaseRole.CONTROL) as session:
        control_session = session.get(SessionModel, created["session_id"])
        assert control_session is not None
        assert control_session.status is SessionStatus.RUNNING
        assert control_session.current_run_id == run_id
        assert control_session.latest_stage_type is StageType.REQUIREMENT_ANALYSIS

    with app.state.database_manager.session(DatabaseRole.RUNTIME) as session:
        run = session.get(PipelineRunModel, run_id)
        assert run is not None
        assert run.status is RunStatus.RUNNING
        assert run.attempt_index == 1
        assert run.graph_definition_ref
        assert run.graph_thread_ref
        assert run.workspace_ref
        assert run.runtime_limit_snapshot_ref
        assert run.provider_call_policy_snapshot_ref
        assert run.template_snapshot_ref
        stage = session.get(StageRunModel, run.current_stage_run_id)
        assert stage is not None
        assert stage.stage_type is StageType.REQUIREMENT_ANALYSIS
        assert stage.status is StageStatus.RUNNING
        assert stage.attempt_index == 1
        assert stage.graph_node_key == "requirement_analysis"

    with app.state.database_manager.session(DatabaseRole.GRAPH) as session:
        definition = session.get(GraphDefinitionModel, run.graph_definition_ref)
        thread = session.get(GraphThreadModel, run.graph_thread_ref)
        assert definition is not None
        assert definition.run_id == run_id
        assert thread is not None
        assert thread.run_id == run_id
        assert thread.graph_definition_id == definition.graph_definition_id
        assert thread.current_node_key == "requirement_analysis"
        assert thread.status == "running"

    with app.state.database_manager.session(DatabaseRole.EVENT) as session:
        event_types = [
            row.event_type.value
            for row in session.query(DomainEventModel)
            .filter(DomainEventModel.session_id == created["session_id"])
            .order_by(DomainEventModel.sequence_index.asc())
            .all()
        ]
    assert "pipeline_run_created" in event_types
    assert "session_message_appended" in event_types

    with app.state.database_manager.session(DatabaseRole.LOG) as session:
        accepted = (
            session.query(AuditLogEntryModel)
            .filter(AuditLogEntryModel.action == "session.message.new_requirement")
            .one()
        )

    assert accepted.request_id == "req-new-requirement"
    assert accepted.correlation_id == "corr-new-requirement"

def _bounded_id(prefix: str, value: str, *, limit: int = 80) -> str:
    candidate = f"{prefix}-{value}"

## Task Definition Fix Proposal

The current source-trace conflict is not only a local implementation gap. It is a task-definition gap:

- `docs/specs/function-one-backend-engine-design-v1.md` fixes persistence as multi-SQLite by responsibility and explicitly states the current version should not over-optimize for database-level distributed transactions.
- The same spec and split-plan also require `R3.2` and later `H4.4` to present a single public service-transaction semantic with no externally visible intermediate state.
- The repository currently has no dedicated reusable consistency boundary for those multi-store high-impact writes, so `R3.2` is carrying both business startup semantics and missing infrastructure prerequisites in one slice.

Recommended authoritative plan correction:

1. Add a new prerequisite task before `R3.2`:

   - **Task id:** `R3.2a`
   - **Suggested title:** `首个 run 启动多库可见性边界`
   - **Owner lane:** `AL01`

2. `R3.2a` should define and implement a reusable multi-store publication boundary for high-impact backend writes across `control.db`, `runtime.db`, `event.db`, and `graph.db`.

3. `R3.2` should then depend on `R3.2a` and focus on first-run business behavior, while keeping the existing strong public-semantic requirement rather than weakening it.

4. `H4.4` should also depend on `R3.2a` or reuse the same boundary for the code-review approval -> delivery snapshot -> runtime resume path, because it has the same “single public semantic / no intermediate state” requirement.

Recommended `R3.2a` acceptance baseline:

- It must not collapse the product back into one SQLite file; the multi-SQLite responsibility split remains unchanged.
- It must provide a single reusable boundary for high-impact multi-store writes, not a one-off `start_first_run()` special case.
- Before the final publish point succeeds, external product reads must not observe a started first run or an approved delivery transition as committed product truth.
- If a pre-publish write fails, the system must leave at most unpublished / recoverable staging state, not a visible half-started run or a visible approved-but-unpublished delivery transition.
- The boundary must define recovery or cleanup semantics for unpublished leftovers on retry or service restart.
- The boundary must be usable by both `RunLifecycleService.start_first_run()` and the future `ApprovalService.approve()` code-review delivery gate path.

Recommended queue / gate corrections:

- `docs/plans/function-one-platform/03-run-projection-and-feed.md`
  - insert `R3.2a` immediately before `R3.2`
  - keep `R3.2` focused on `new_requirement` startup semantics, but make it explicitly depend on `R3.2a`
- `docs/plans/function-one-platform-plan.md`
  - add a new row for `R3.2a`
- `docs/plans/function-one-acceleration-execution-plan.md`
  - AL01 queue becomes `... -> R3.7 -> R3.2a -> R3.2 -> R3.3 -> ...`
  - AL01 gate for `R3.2` requires `R3.2a` complete
  - AL03 gate for `H4.4` final completion requires the shared boundary from `R3.2a`
  - Shared Ownership adds a new shared entry similar to:
    - `高影响多库公开语义 / publication boundary` | owner `AL01` | consumers `AL03`, later `QA`

Recommended implementation surface for `R3.2a`:

- A new reusable backend service / helper, for example under a dedicated consistency boundary module, rather than more ad hoc `_commit_all()` branching inside `RunLifecycleService`.
- Focused tests that simulate:
  - startup failure before publish

  - retry / cleanup of unpublished leftovers
  - shared reuse by first-run startup and approval/delivery snapshot gating
