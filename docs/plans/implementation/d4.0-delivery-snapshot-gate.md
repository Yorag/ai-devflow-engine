# D4.0 Delivery Snapshot Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task-by-task. Use `superpowers:test-driven-development` for every production or test change. Fallback to `superpowers:executing-plans` only if subagents cannot be scheduled or the task boundary cannot be supplied precisely. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the backend service boundary that freezes a run-scoped delivery channel snapshot for the final `code_review_approval` approve path before entering `delivery_integration`.

**Architecture:** Add a focused `DeliverySnapshotService` that reads the current project `DeliveryChannel`, validates the D4.0 gate context, persists `DeliveryChannelSnapshotModel`, attaches the snapshot ref to `PipelineRunModel`, and exposes read/assert helpers for later delivery integration slices. Keep approval command routing, `ApprovalDecisionModel`, `approval_result`, runtime resume, and DeliveryRecord creation out of this slice; H4.4 and D5.x consume this boundary.

**Tech Stack:** Python 3.11+, SQLAlchemy, existing multi-SQLite `DatabaseManager`, existing `DeliveryChannelService`, existing `AuditService`, existing JSONL log writer protocol, `pytest` through `uv run`.

`prepare_delivery_snapshot()` behavior:
- Reject with `ErrorCode.VALIDATION_ERROR`, HTTP-like status `409`, and `DELIVERY_SNAPSHOT_GATE_CONTEXT_MESSAGE` unless `approval_type is ApprovalType.CODE_REVIEW_APPROVAL` and `target_stage_type is StageType.DELIVERY_INTEGRATION`.
- Load `PipelineRunModel` by `run_id`; missing run returns `ErrorCode.NOT_FOUND`, status `404`.
- Require `run.project_id == project_id`; mismatch returns `ErrorCode.VALIDATION_ERROR`, status `409`.
- Resolve the current project channel through `DeliveryChannelService.resolve_current_project_channel(project_id, trace_context=trace_context)`.
- If `run.delivery_channel_snapshot_ref` already points to an existing snapshot, return that snapshot without creating a second row. The idempotent return must not recopy newer project channel changes.
- For `DeliveryMode.GIT_AUTO_DELIVERY`, require `channel.readiness_status is DeliveryReadinessStatus.READY` and `channel.credential_status is CredentialStatus.READY`; otherwise reject with `ErrorCode.DELIVERY_SNAPSHOT_NOT_READY`, status `409`, and do not mutate the run.
- For `DeliveryMode.DEMO_DELIVERY`, allow snapshot creation even when SCM fields are absent. The snapshot still stores `readiness_status`, `credential_status`, `readiness_message`, and `last_validated_at` from the current channel.
- Create `DeliveryChannelSnapshotModel` with id `delivery-snapshot-<uuid4 hex>`, `schema_version = DELIVERY_SNAPSHOT_SCHEMA_VERSION`, `created_at = now`.
- Copy exactly the project channel fields listed above.
- Assign `run.delivery_channel_snapshot_ref` to the new snapshot id and update `run.updated_at`.
- Write a run log with source `services.delivery_snapshots`, category `LogCategory.DELIVERY`, message `Delivery snapshot prepared.`, and sanitized metadata.
- Write required audit with action `delivery_snapshot.prepare`, target type `pipeline_run`, target id `run_id`, result `AuditResult.SUCCEEDED`, and rollback callback that rolls back the runtime session.
- Commit only after the log and audit write complete. On error, roll back runtime state and propagate the error.

`get_snapshot_for_run()` behavior:
- Load `PipelineRunModel` by `run_id`.
- If the run is missing or `delivery_channel_snapshot_ref` is empty, raise `ErrorCode.DELIVERY_SNAPSHOT_MISSING`, status `409`.
- Return the referenced `DeliveryChannelSnapshotModel`, or raise the same missing error if the ref points to no row.

`assert_snapshot_ready_for_delivery()` behavior:
- Call `get_snapshot_for_run()`.
- Require `snapshot.readiness_status is DeliveryReadinessStatus.READY`.
- Require `snapshot.credential_status is CredentialStatus.READY`.
- If not ready, raise `ErrorCode.DELIVERY_SNAPSHOT_NOT_READY`, status `409`.
- Return the snapshot.

`DeliveryChannelService.resolve_current_project_channel()` behavior:
- Reuse the existing visible project/default channel lookup semantics.
- Return the current project default `DeliveryChannelModel`.
- Keep `get_project_channel()` behavior unchanged.

## Log & Audit Integration

- Runtime log category: `LogCategory.DELIVERY`.
- Runtime log source: `services.delivery_snapshots`.
- Runtime log messages:
  - `Delivery snapshot prepared.`
  - `Delivery snapshot preparation rejected.`
  - `Delivery snapshot preparation failed.`
- Audit action:
  - success: `delivery_snapshot.prepare`
  - rejected: `delivery_snapshot.prepare.rejected`
  - failed: `delivery_snapshot.prepare.failed`
- Associated objects: `run_id`, `project_id`, `delivery_channel_id`, `delivery_channel_snapshot_id`, `delivery_mode`, readiness fields, `approval_type`, `target_stage_type`, and trace ids.
- Sensitive field handling: metadata may include safe `env:<NAME>` credential references, but never credential values. Unsafe credential refs must be redacted through the existing redaction policy.
- Failure behavior: required audit write failure rejects or rolls back snapshot preparation. Ordinary run log failure must not silently preserve a prepared snapshot; this service rolls back and raises before claiming the snapshot is prepared.
- Product truth boundary: logs and audit records do not replace `PipelineRunModel.delivery_channel_snapshot_ref` and `DeliveryChannelSnapshotModel`.
