# A4.9 LangChain Provider Adapter Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` as the default execution skill for this plan. Use `superpowers:executing-plans` only as a fallback when the environment cannot safely dispatch bounded subagents. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement `LangChainProviderAdapter` so AL05 can create LangChain chat models from frozen run snapshots, bind registered tools only when the bound model supports tool calling, execute structured-output calls within snapshot token budgets, and return structured `ModelCallResult` metadata without introducing runtime-orchestration behavior.

**Architecture:** Add a pure adapter module under `backend/app/providers/` that composes existing `ProviderConfig`, runtime/provider-policy snapshots, prompt-render outputs, tool bindable descriptions, LangChain `ChatOpenAI`, and fake fixture models behind a stable adapter boundary. The adapter owns model construction, tool/schema binding, structured invocation, output normalization, safe log payload summarization, and `model_call_trace`-ready metadata, but it does not own retry/circuit-breaker policy, context construction, tool execution, StageArtifact persistence, or AgentDecision parsing.

**Tech Stack:** Python 3.11+, `langchain==1.2.17`, `langchain-openai==1.2.1`, Pydantic v2, repository-local `uv`, and pytest via `uv run pytest`.

## Planned Adapter Contract

`backend/app/providers/langchain_adapter.py` will define:

- `ModelCallToolRequest`
  - normalized tool-call candidate with `call_id`, `tool_name`, `input_payload`, and `schema_version`
- `ModelCallUsage`
  - normalized `input_tokens`, `output_tokens`, and `total_tokens`
- `ModelCallTraceSummary`
  - redacted/summarized input/output trace metadata plus snapshot refs and trace ids
- `ModelCallResult`
  - normalized success/failure result with:
    - `provider_snapshot_id`
    - `model_binding_snapshot_id`
    - `model_call_type`
    - `structured_output`
    - `structured_output_candidates`
    - `tool_call_requests`
    - `invalid_tool_call_candidates`
    - `provider_error_code`
    - `provider_error_message`
    - `usage`
    - `raw_response_ref`
    - `native_reasoning_ref` or summarized reasoning only when supported and present
    - `trace_summary`
- `LangChainProviderAdapterError`
  - structured adapter-level failure with stable `error_code`
- `LangChainProviderAdapter`
  - `create_chat_model()`
  - `bind_tools()`
  - `with_structured_output()`
  - `invoke_structured()`

Design rules:

- `create_chat_model()` uses only frozen `ProviderConfig` plus passed-in timeout/output-budget inputs.
- real `ChatOpenAI` construction resolves `env:` credential refs before SDK creation and returns a structured adapter error when the referenced env value is unavailable.
- `bind_tools()` accepts only `ToolBindableDescription` inputs and calls `.to_langchain_tool_schema()`.
- `with_structured_output()` chooses native structured output only when `supports_structured_output` is true; otherwise it returns a compatible unbound runnable marker for fallback invocation logic inside `invoke_structured()`.
- `invoke_structured()` returns normalized `ModelCallResult`; it does not parse business decisions and does not execute tools.
- Output-budget enforcement uses `min(config.max_output_tokens, requested_max_output_tokens)` when a requested budget is provided.
- Tool binding must fail with structured capability errors when `supports_tool_calling` is false.
- malformed OpenAI-compatible raw tool-call payloads must surface as invalid tool-call candidates rather than empty executable tool requests.
- `supports_native_reasoning == false` means `ModelCallResult` must not invent raw reasoning content even if the fake/raw response includes unrelated text.

- [x] **Step 4: Implement chat-model creation, tool binding, structured invocation, and normalization**

Implementation requirements:

- `LangChainProviderAdapter.__init__()` inputs:
  - `provider_config: ProviderConfig`
  - `provider_call_policy_snapshot: ProviderCallPolicySnapshotRead`
  - optional `redaction_policy: RedactionPolicy`
  - optional `chat_model_factory` test seam
- `create_chat_model(requested_max_output_tokens: int | None = None)`:
  - compute `effective_max_tokens = min(provider_config.max_output_tokens, requested)` when requested exists
  - instantiate `ChatOpenAI` with:
    - `model=provider_config.model_id`
    - `openai_api_base=provider_config.base_url`
    - `openai_api_key=provider_config.api_key_ref or "fixture-api-key"`
    - `request_timeout=provider_call_policy_snapshot.provider_call_policy.request_timeout_seconds`
    - `max_tokens=effective_max_tokens`
    - `model_kwargs=dict(provider_config.model_parameters)`
  - do not read live Provider service or settings
- `bind_tools(chat_model, tool_descriptions)`:
  - if `provider_config.supports_tool_calling` is false, raise `LangChainProviderAdapterError("provider_capability_unsupported", ...)`
  - convert `ToolBindableDescription` via `.to_langchain_tool_schema()`
  - call `chat_model.bind_tools(tool_schemas, strict=True)`
- `with_structured_output(chat_model, response_schema)`:
  - if `provider_config.supports_structured_output` is true, call `chat_model.with_structured_output(response_schema, include_raw=True, strict=True)`
  - if false, return the original/bound model unchanged and let `invoke_structured()` perform compatibility parsing from raw `AIMessage`
- `invoke_structured(...)`:
  - accept messages, response schema, model_call_type, tool descriptions, trace context, and optional requested output budget
  - for tool-enabled calls, bind tools before invocation
  - for structured-output-capable models, use `with_structured_output(...).invoke(messages)`
