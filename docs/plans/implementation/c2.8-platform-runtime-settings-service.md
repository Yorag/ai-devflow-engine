**Goal:** Implement the backend `PlatformRuntimeSettings` repository, service, and internal management API with versioned persistence, hard-limit validation, log/audit integration, and OpenAPI coverage.

**Architecture:** Add a narrow repository over the existing `PlatformRuntimeSettingsModel`, then layer a service that initializes defaults, merges updates, validates against `PlatformHardLimits`, increments `runtime-settings-vN` versions, and records sanitized log/audit facts. Expose only internal backend management routes at `/api/runtime-settings`; this capability does not enter normal frontend settings or configuration package flows.

**Tech Stack:** FastAPI, Pydantic v2, SQLAlchemy 2.x, pytest, repository-local `uv run --no-sync`.

## Scope

In scope:
- Create `PlatformRuntimeSettingsRepository` over `PlatformRuntimeSettingsModel`.
- Persist one current settings row with `settings_id = "platform-runtime-settings"`.
- Initialize the settings row on first read using C1.10 defaults, schema version `runtime-settings-schema-v1`, hard limit version `platform-hard-limits-v1`, and initial config version `runtime-settings-v1`.
- Return `PlatformRuntimeSettingsRead` with four settings groups, version metadata, and read-only `PlatformHardLimits`.
- Update settings through `PlatformRuntimeSettingsUpdate` using `expected_config_version`.
- Merge partial updates over the current settings row and save a new monotonic config version such as `runtime-settings-v2`.
- Validate `agent_limits`, `provider_call_policy`, `context_limits`, and `log_policy` through the existing Pydantic schemas and service hard-limit checks.
- Reject hard-limit violations with `config_hard_limit_exceeded`.
- Reject empty updates, invalid persisted payloads, and service-level invalid values with `config_invalid_value`.
- Reject stale `expected_config_version` with `config_version_conflict`.
- Map storage failures to `config_storage_unavailable`.
- Record sanitized JSONL service logs and audit records for initialization, successful updates, rejected updates, and failed storage/audit operations.
- Expose `GET /api/runtime-settings` and `PUT /api/runtime-settings` as internal backend management routes and assert OpenAPI path, request, response, and error schemas.

Out of scope:
- Frontend UI or frontend API client changes.
- `ConfigurationPackage` import/export changes.
- Environment variable changes or dependency changes.
- Runtime start snapshot creation, `RuntimeLimitSnapshotModel` writes, `ProviderCallPolicySnapshot` writes, `GraphDefinition`, LangGraph runtime, Provider adapter, Context Management, or active run mutation.
- Alembic migrations.
- Git write actions.

## Behavior Details

Constants:

```python
RUNTIME_SETTINGS_ID = "platform-runtime-settings"
RUNTIME_SETTINGS_SCHEMA_VERSION = "runtime-settings-schema-v1"
INITIAL_RUNTIME_SETTINGS_VERSION = "runtime-settings-v1"

SYSTEM_ACTOR_ID = "runtime-settings-seed"

VERSION_CONFLICT_MESSAGE = "PlatformRuntimeSettings expected_config_version does not match the current version."
STORAGE_UNAVAILABLE_MESSAGE = "PlatformRuntimeSettings storage is unavailable."
INVALID_PERSISTED_SETTINGS_MESSAGE = "PlatformRuntimeSettings stored payload is invalid."
```

Repository behavior:
- `get_current()` returns the `PlatformRuntimeSettingsModel` row with `settings_id = RUNTIME_SETTINGS_ID`, or `None`.
- `save_new_version(model)` adds the current model and flushes it. The repository does not commit; the service owns commit/rollback boundaries.
- Repository methods catch `SQLAlchemyError` and raise `RuntimeSettingsRepositoryError`.
- The repository does not know hard-limit rules, audit semantics, JSONL logging, run snapshots, or API errors.

Service read behavior:
- `ensure_initialized(trace_context)` loads the current row. If it exists, return it as `PlatformRuntimeSettingsRead`.
- If the row is missing, create a persisted default row with:
  - `settings_id = "platform-runtime-settings"`
  - `config_version = "runtime-settings-v1"`
  - `schema_version = "runtime-settings-schema-v1"`
  - `hard_limits_version = PlatformHardLimits().hard_limits_version`
  - `agent_limits = AgentRuntimeLimits().model_dump(mode="python")`
  - `provider_call_policy = ProviderCallPolicy().model_dump(mode="python")`
  - `context_limits = ContextLimits().model_dump(mode="python")`
  - `log_policy = LogPolicy().model_dump(mode="python")`
  - `created_by_actor_id = "runtime-settings-seed"`
  - `updated_by_actor_id = "runtime-settings-seed"`
  - `last_trace_id = trace_context.trace_id`
- Initialization first commits the control settings row, then writes a service JSONL record and a `runtime_settings.initialize` audit record with system actor metadata containing config version, schema version, hard limit version, changed fields, and effective scope `future_runs_and_diagnostic_queries`.
- If the control commit fails, no success observability fact is written and the service returns `config_storage_unavailable`. If post-commit JSONL or audit recording fails, the service returns `config_storage_unavailable` while the committed control row remains the configuration truth; this avoids a false success observability fact for an uncommitted control write across separate SQLite resources.
- `get_current_settings(trace_context)` delegates to `ensure_initialized()`.
- `current_version(trace_context)` returns the current `config_version`, initializing the row if needed.

Service update behavior:
- `update_settings(body, trace_context)` first ensures a row exists.
- It rejects an update where all four settings groups are `None` with `config_invalid_value`, status `422`, and audit action `runtime_settings.update.rejected`.
- It rejects stale `body.expected_config_version` with `config_version_conflict`, status `409`, and audit action `runtime_settings.update.rejected`.
- It merges each non-`None` settings group over the current row; omitted groups keep current values.
- It constructs new schema objects from the merged dictionaries so Pydantic validators remain the common validation path for backend management, initialization, and tests.
- It calls `validate_against_hard_limits()` before saving. On violation, it records a rejected audit and raises `config_hard_limit_exceeded`, status `422`.
- It increments the config version by parsing the current `runtime-settings-vN` suffix. If the suffix is not parseable, it falls back to `runtime-settings-v<unix timestamp seconds>` to preserve monotonic non-empty versioning without blocking recovery.
- It updates the same current row rather than creating a history table, because C1.6 defined one current `PlatformRuntimeSettingsModel` primary key and no migration chain for historical versions in this slice.
- It sets `updated_by_actor_id = "api-user"` and `last_trace_id = trace_context.trace_id`.
- It writes the settings row through a conditional update whose predicate includes the previous `config_version`, then commits the control row before writing the success service JSONL record and `runtime_settings.update` audit record. Metadata includes changed group names, changed field paths, old value summary, new value summary, previous version, new version, hard-limit version, and effective scope. It does not include prompt bodies, credentials, runtime snapshots, database paths, or frontend setting fields.
- If the conditional update observes a different current `config_version`, it rejects with `config_version_conflict` and does not overwrite the latest row. If the control commit fails, no success observability fact is written. If success observability recording fails after the control commit, the service returns `config_storage_unavailable` and records a failed audit when possible; the committed control row remains the source of truth.
- It never updates active or historical run snapshots. Future C3/R3 slices will read this service to create run-start snapshots.

Hard-limit validation:
- `agent_limits` fields must be less than or equal to `PlatformHardLimits().agent_limits` fields.
- `provider_call_policy` fields must be less than or equal to `PlatformHardLimits().provider_call_policy` fields.
- `context_limits` fields other than `compression_threshold_ratio` must be less than or equal to `PlatformHardLimits().context_limits` fields.
- `context_limits.compression_threshold_ratio` is validated by `ContextLimits` as greater than `0` and less than `1`; there is no separate hard-limit field for that ratio.
- `log_policy.run_log_retention_days`, `audit_log_retention_days`, `log_rotation_max_bytes`, and `log_query_max_limit` must be less than or equal to `PlatformHardLimits().log_policy` fields.
- `log_policy.log_query_default_limit` must be less than or equal to both `log_policy.log_query_max_limit` and the hard `log_query_max_limit`; the first relation is already enforced by `LogPolicy`, but the service should include it in the field path summary when it rejects a malformed persisted row.
- A hard-limit violation message must name the exact field path, for example `agent_limits.max_react_iterations_per_stage exceeds platform hard limit 50.`

Error mapping:
- `RuntimeSettingsServiceError(ErrorCode.CONFIG_INVALID_VALUE, ..., 422)`
- `RuntimeSettingsServiceError(ErrorCode.CONFIG_HARD_LIMIT_EXCEEDED, ..., 422)`
- `RuntimeSettingsServiceError(ErrorCode.CONFIG_VERSION_CONFLICT, ..., 409)`
- `RuntimeSettingsServiceError(ErrorCode.CONFIG_STORAGE_UNAVAILABLE, ..., 503)`

API behavior:
- `GET /api/runtime-settings`
  - Response model: `PlatformRuntimeSettingsRead`.
  - Initializes and persists defaults on first read.
  - Error responses: `422`, `500`, `503` with `ErrorResponse`.
- `PUT /api/runtime-settings`
  - Request model: `PlatformRuntimeSettingsUpdate`.
  - Response model: `PlatformRuntimeSettingsRead`.
  - Requires `expected_config_version`.
  - Error responses: `409`, `422`, `500`, `503` with `ErrorResponse`.
- The new route module is included from `backend/app/api/router.py`.
- This route is internal backend management only. Do not add frontend client, query hook, settings modal field, mock fixture field, or configuration package field.

## Log & Audit Integration

- Runtime log category:
  - Initialization: `LogCategory.API`, source `services.runtime_settings`, message `PlatformRuntimeSettings initialized.`
  - Update success: `LogCategory.API`, source `services.runtime_settings`, message `PlatformRuntimeSettings updated.`
  - Update rejected: `LogCategory.API`, source `services.runtime_settings`, message `PlatformRuntimeSettings update rejected.`
- Audit actions:
  - `runtime_settings.initialize`
  - `runtime_settings.update`
  - `runtime_settings.update.rejected`
  - `runtime_settings.update.failed`
- Actors:
  - Initialization: `AuditActorType.SYSTEM`, `actor_id="runtime-settings-seed"`
  - API update: `AuditActorType.USER`, `actor_id="api-user"`
- Target: `target_type="platform_runtime_settings"`, `target_id="platform-runtime-settings"`.
- Trace fields: API routes pass `get_trace_context()` into the service. JSONL and audit records inherit `request_id`, `trace_id`, `correlation_id`, `span_id`, and `parent_span_id`.
- Redaction: service log and audit metadata are passed through `RedactionPolicy.summarize_payload()` and `LogPayloadSummary.from_redacted_payload()`. Metadata includes only settings group names, numeric field paths and values, version ids, hard-limit version, and effective scope.
- Failure behavior: control storage failures map to `config_storage_unavailable` before success observability facts are written. Rejected-update JSONL or audit failures also map to `config_storage_unavailable`. Success observability failures after a committed control update map to `config_storage_unavailable` and record `runtime_settings.update.failed` when possible, while preserving the committed control row as source of truth.
- Product truth: logs and audits are observability facts only. `PlatformRuntimeSettingsModel` is the control-plane source of truth. This service does not read or write runtime snapshots.

RUNTIME_SETTINGS_ID = "platform-runtime-settings"


class RuntimeSettingsRepositoryError(RuntimeError):
    pass


class PlatformRuntimeSettingsRepository:
    def __init__(self, session: Session) -> None:
        self._session = session

    def get_current(self) -> PlatformRuntimeSettingsModel | None:
        try:
            return self._session.get(PlatformRuntimeSettingsModel, RUNTIME_SETTINGS_ID)
        except SQLAlchemyError as exc:
            raise RuntimeSettingsRepositoryError(
                "PlatformRuntimeSettings storage is unavailable."
            ) from exc

    def save_new_version(
        self,
        model: PlatformRuntimeSettingsModel,
    ) -> PlatformRuntimeSettingsModel:
        try:
            self._session.add(model)
            self._session.flush()
            return model
        except SQLAlchemyError as exc:
            raise RuntimeSettingsRepositoryError(
                "PlatformRuntimeSettings storage is unavailable."
            ) from exc
