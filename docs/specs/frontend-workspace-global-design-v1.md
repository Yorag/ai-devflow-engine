# 前端工作台全局设计 V1

## 1. 文档目标

本文档用于定义 `AI 驱动的需求交付流程引擎` 的全局前端工作台设计，作为后续前端信息架构、页面设计、交互实现、组件拆分和验收评估的统一依据。

本文档聚焦：
- 全局前端工作台的页面形态与信息架构
- 项目、会话、执行链路、人工审批之间的前端关系
- 需求输入、需求澄清、执行展示、审批操作、高风险工具确认、交付查看的统一交互方式
- 面向用户可理解的执行叙事展示方式

本文档不展开：
- 后端 API、数据模型与实时推送协议
- 未确定的组件库选型与样式工程细节
- 功能二的浏览器注入、圈选交互与页面悬浮操作

### 1.2 前端技术选型

功能一与功能二前端统一采用 `单一 SPA，多路由承载不同页面` 的实现方向。

功能一 V1 前端主栈固定为：
- `React`
- `Vite`
- `React Router`

该技术选型必须满足以下规则：
- 官网、功能页与控制台属于同一个前端工程，在同一个 SPA 中通过不同路由承载
- 当前版本不以 `SSR`、`SSG` 或 SEO 能力作为前端技术选型前提
- 前端主路由切换不得破坏控制台会话页内部的 `Session` 级连续阅读与运行回放语义

为满足控制台规格中的会话主流、实时更新与复杂本地交互，功能一 V1 前端还固定采用以下支撑层：
- `TanStack Query`
  负责 REST 快照加载、缓存与失效管理
- `Zustand`
  负责当前会话 UI 状态、本地交互状态与页内焦点状态管理
- `EventSource` 薄封装
  负责消费后端 `SSE` 事件流，并把增量事件回写到工作台状态中

这些支撑层必须满足以下边界：
- `TanStack Query` 负责服务端数据快照、查询缓存与重新拉取
- `Zustand` 负责 `Inspector` 开关、当前 run 焦点、模板脏状态、审批块本地交互态等纯前端状态
- `SSE` 增量事件消费不得绕过前端既定状态模型直接修改界面

本节定义的是仓库级前端实现策略；除本节外，本文其余章节仍只约束功能一第一阶段控制台前端。

以下细节在当前文档中仍不固定：
- 具体组件库品牌
- 具体样式工程方案
- 具体图标库与动效库

## 2. 适用范围

本设计适用于功能一第一阶段的主控制台前端。

本设计覆盖以下前端范围：
- 项目加载与项目切换
- 项目下新建需求会话
- 会话模板选择与模板配置编辑
- 需求输入与需求澄清
- 执行链路瀑布流展示
- 人工审批交互
- 高风险工具确认交互
- 交付结果查看
- 执行详情侧栏查看

本设计不覆盖以下范围：
- 独立报表后台
- 管理员系统
- 多租户与权限中心
- 功能二交互式网页改动面板

## 3. 产品定位

前端工作台应被定义为：

`一个以项目为上下文、以需求会话为单位、以执行叙事流为中心的 AI 交付工作台`

其核心目标不是展示静态阶段状态，而是让用户在单一工作台中连续看到：
- 当前处于哪个项目
- 当前正在处理哪个需求会话
- 系统如何理解需求
- 当信息不足时系统如何发起澄清
- 系统如何推进方案、实现、测试、评审与交付
- 当人工介入发生时用户应在哪里决策
- 当高风险工具动作需要确认时用户应如何理解风险并做出决定

## 4. 全局设计原则

前端工作台必须遵循以下原则：

1. `项目优先`
所有需求会话必须挂载在一个项目之下。项目是上下文感知的基础载体。

2. `会话承载需求`
每一次需求输入、需求澄清、执行推进、审批决策、高风险工具确认和交付结果都必须归属到一个会话中。

3. `中栏优先`
用户的主要阅读、输入和理解行为必须发生在中栏，左栏与右栏只承担导航和深度查看职责。

4. `叙事优先于看板`
主界面不采用仪表盘优先结构，不用全局卡片墙组织主要信息，主要信息必须沿一条自上而下增长的执行叙事流展开。

5. `澄清属于链路内部`
需求澄清不作为独立页面，不作为独立审批模式，不作为脱离主链路的特殊流程。
需求澄清发生在 `Requirement Analysis` 结点的执行过程中；当信息不足时，该结点必须在主链路中显式中断并要求用户补充信息，用户补充后该结点继续分析并决定是否进入下一个阶段。

6. `审批嵌入链路`
人工审批必须直接出现在执行链路中，不允许跳转到独立审批页面后再返回。

7. `工具确认嵌入链路`
高风险工具确认必须直接出现在执行链路中，不允许跳转到独立权限页面后再返回。工具确认不是人工审批，不使用 Approval Request，不影响两个必需人工审批检查点的固定范围。

8. `默认可读，按需深看`
主链路正文必须足以让普通用户理解系统在做什么；更结构化、更高密度的细节信息通过右侧详情栏展开。

9. `结点异构展示`
不同执行结点的显示内容必须根据其业务语义单独设计，不允许所有结点套用统一摘要模板。

10. `信息契约先于样式`
在进入视觉样式与组件封装之前，必须先定义结点正文与结点内条目的信息内容、右侧 Inspector 中的完整信息内容、量化指标的字段和展示位置。

11. `固定骨架，有限配置`
功能一 V1 的模板编辑不是自由流程编排。用户可以在固定阶段骨架上调整允许的执行配置，但不能删除、禁用、重排核心业务阶段，也不能关闭两个必需人工审批检查点。

## 5. 名词定义

1. `Project`
一个被加载到系统中的本地项目上下文，绑定一个文件夹路径，后续该项目下所有需求会话都以此路径作为上下文感知入口。

2. `Session`
项目下的一次需求会话，承载一次从需求输入到交付结果的完整执行链路。

3. `Pipeline Template`
用于创建 `PipelineRun` 的可复用流程蓝图。功能一 V1 中，模板向用户暴露的是固定阶段骨架上的有限配置项，而不是任意阶段编排器。

4. `PipelineRun`
同一 `Session` 下某一次具体执行尝试。首条需求消息会自动启动首个 `PipelineRun`；同一需求链路后续如发生重新尝试或运维重启，可以在同一会话下形成新的 `PipelineRun`。原 run 的暂停后恢复属于同一个 `PipelineRun` 的继续执行，不创建新的 run。

5. `Run Switcher`
用于在同一 `Session` 页面内定位、跳转和快速聚焦不同 `PipelineRun` 分段的前端控件。它不改变会话归属，也不把会话拆成多个独立 run 页面。

6. `Narrative Feed`
中栏中的统一主瀑布流。它按照时间顺序展示顶层用户输入、系统澄清、执行结点、审批请求、高风险工具确认、审批结果和交付结果。
它不是纯聊天记录，也不是纯流程图，而是 `对话式内容 + 执行链路内容` 共存的一条主流。
在需求尚未澄清完成时，`Requirement Analysis` 结点内部可以表现为多轮对话；在需求进入正式执行后，它表现为按阶段推进的执行链路瀑布流；这两种内容必须连续地存在于同一条主流中。

7. `Execution Node`
执行链路中的阶段级可阅读单元。结点对应一次显式业务阶段运行，例如 `Requirement Analysis`、`Solution Design`、`Code Generation`、`Test Generation & Execution`、`Code Review`、`Delivery Integration`。
结点以阶段大框形式存在于会话式瀑布流中；结点内部可以逐步显示多种异构条目，例如推理内容、关键决策、工具调用记录、diff 预览、审批相关提示与阶段结果内容。

8. `Approval Request`
当系统运行到人工检查点时，在叙事流中插入的交互型审批块。

9. `Tool Confirmation`
当系统运行到高风险工具动作前，在叙事流中插入的交互型工具确认块。它用于确认具体工具动作、命令、目标路径、预期副作用与风险说明，不用于判断阶段产物质量。

10. `Control Item`
执行主链中的控制型条目，用于表达等待澄清、回退说明与重试说明等运行时控制语义。控制型条目不是正式业务阶段结点。

11. `Inspector`
右侧按需展开的详情栏，用于展示阶段结点、可深看的控制型条目或交付结果对应的完整结构化详情。

12. `展示名` 与 `契约名`
本文档中的 `Requirement Analysis`、`Solution Design`、`Code Review` 等 Title Case 名称用于描述面向用户的阶段展示名或正文语义；`requirement_analysis`、`solution_design`、`code_review` 等 snake_case 名称用于描述前后端字段值、投影类型和接口契约。
除非特别说明，展示层文案与接口契约名不得混用为同一层级的输出要求。

## 6. 全局页面结构

全局前端工作台采用三栏结构：

- 左栏：`Project + Session Sidebar`
- 中栏：`Narrative Workspace`
- 右栏：`Inspector`

布局规则如下：
- 左栏与右栏都支持折叠
- 中栏始终是主工作区域
- 默认进入页面时必须优先聚焦中栏
- 右栏默认关闭，点击结点或交付结果后展开

布局分配如下：
- 左栏宽度：`280px - 340px`
- 中栏宽度：自适应主区域
- 右栏宽度：`360px - 480px`

## 7. 左栏设计

左栏负责 `项目切换` 与 `会话切换`，不承担主链路阅读职责。

### 7.1 结构组成

左栏必须包含以下区域：

1. `Project Switcher`
- 展示当前项目名称
- 展示当前项目路径截断文本
- 提供项目切换入口

2. `Load Project Entry`
- 提供加载本地文件夹路径的入口
- 加载完成后创建一个新的项目上下文

3. `Current Project Summary`
- 展示当前项目的基础信息
- 显示该项目下会话数量、最近活动信息与当前默认交付模式摘要

4. `New Session Entry`
- 在当前项目下创建新需求会话
- 新建后中栏立即进入该会话

5. `Session List`
- 展示当前项目下近期需求会话
- 每个会话项至少显示：标题、状态、更新时间、当前链路位置

### 7.2 项目规则

项目必须满足以下规则：
- 系统启动后默认存在一个项目，绑定平台仓库自身路径
- 用户可以继续加载其他本地文件夹作为新项目
- 左栏必须持续显示当前默认交付模式摘要，但不承担交付配置编辑入口
- 当前项目切换后，中栏应切到该项目下最近一次会话；如果不存在会话，则展示该项目的空白欢迎内容
- 项目切换不应打断其他项目已存在的会话历史

### 7.3 会话列表规则

会话列表必须满足以下规则：
- 支持显示当前项目下的近期会话
- 允许继续打开历史会话
- 当前会话应有明确选中态
- 等待人工审批的会话应在列表中有显式提示
- 已完成、执行中、暂停中、失败、已终止、等待澄清、等待审批等状态必须可区分

### 7.4 统一设置弹窗

前端控制台必须提供一个统一的 `设置` 入口，用于承载功能一 V1 用户可见的业务配置，包括项目级交付配置、模型提供商配置与项目作用域配置导入导出。

该设置入口必须满足以下规则：
- 入口位于控制台右上角或等价全局工具区，不位于左栏项目区域
- 点击后以桌面端居中的大弹窗打开
- 弹窗必须采用 `左侧导航 + 右侧内容区` 的结构
- 弹窗左上角必须提供显式关闭入口
- 窄屏场景下允许退化为全屏抽屉或等价全屏设置页，但信息结构不变

设置弹窗在 V1 至少包含三个子页面：
1. `通用配置`
2. `模型提供商`
3. `导入导出`

设置弹窗必须满足以下边界：
- 不展示或编辑环境变量、平台运行数据目录、SQLite 文件路径、CORS、日志文件路径或 `deterministic test runtime`
- 不展示或编辑后端平台硬上限、全局 ReAct 循环上限、日志保留策略、日志裁剪策略、诊断查询分页上限或 `compression_threshold_ratio`
- 不展示或编辑后端系统内置提示词资产、`prompt_id`、`prompt_version`、`runtime_instructions`、结构化输出修复提示词或 `compression_prompt`
- 不把配置导入导出包展示为启动配置文件、运行模式或本地长期配置真源
- 若后端返回配置校验错误或平台硬上限错误，前端必须展示错误原因，但不得把这些平台设置实现为普通用户可编辑表单
- 除项目级 `DeliveryChannel` 在交付快照固化前可用于当前活动 run 的交付就绪校验外，设置保存不得改变已启动 run 的执行语义；对已固化快照的 run，前端必须按后端返回的只读状态展示，不得暗示配置变更会回写历史 run

#### 7.4.1 通用配置

`通用配置` 页面用于编辑当前 `Project` 的默认 `DeliveryChannel`，不属于会话模板配置。

该页面必须满足以下规则：
- 必须显式显示当前配置作用于哪个 `Project`
- 至少支持查看和编辑 `delivery_mode`
- 当 `delivery_mode = git_auto_delivery` 时，至少支持查看和编辑托管平台类型、仓库标识、默认分支、代码评审请求类型与凭据绑定，并显示当前凭据绑定状态
- 当 `delivery_mode = git_auto_delivery` 时，前端提交的是 `credential_ref`，展示的是 `credential_status`；前端不展示真实密钥内容
- 必须提供交付配置状态展示，至少可区分 `unconfigured`、`invalid`、`ready`
- 支持独立的配置校验与保存动作
- 保存结果至少影响后续新启动的 run
- 若当前活动 run 尚未进入 `Delivery Integration`，则当前项目交付配置的最新保存结果必须可用于该 run 后续的交付就绪校验
- 当当前活动 run 已通过最终审批并固化交付快照后，后续保存不得回写该 run 或历史 run 的交付快照

#### 7.4.2 模型提供商

`模型提供商` 页面用于管理前端可见的 Provider 配置。

该页面必须满足以下规则：
- 只展示已经配置的 Provider；未配置的内置 Provider 不在初始页面显示为默认卡片
- `Add custom provider` 入口提供 `火山引擎`、`DeepSeek`、`OpenAI Completions` 三个并列接入模板，用户选择后才在页面中创建 Provider 卡片
- 同一个接入模板允许添加多个 Provider；前端不得把模板名称当成唯一 Provider 实例
- Provider 卡片展示和编辑字段限于显示名称、`base_url`、API key、`supported_model_ids`、`default_model_id`、启用状态和折叠的模型能力高级设置
- Provider 显示名称可由用户点击卡片名称编辑；`provider_id` 不展示也不作为用户可编辑字段
- 内置 Provider 的 `provider_id`、`provider_source` 和协议归属只读，不提供供应商类型修改入口
- 内置 Provider 允许编辑显示名称、连接字段、默认模型、模型列表、API key 和模型能力字段；这些字段保存到后端正式 Provider 配置对象，不作为环境变量或本地配置文件字段保存
- `custom Provider` 允许编辑显示名称、连接字段、默认模型、模型列表、API key 和模型能力字段
- 每个 Provider 卡片必须提供启用或禁用控件，并提供删除 Provider 的入口
- Provider 连接字段至少覆盖后端 API 要求的 `base_url`、`default_model_id`、`supported_model_ids` 与 API key 输入
- Provider 模型能力字段必须放在折叠的 `高级设置` 区域中，不作为新增或编辑 Provider 的首屏必填项
- `高级设置` 允许按模型编辑 `context_window_tokens`、`max_output_tokens`、`supports_tool_calling`、`supports_structured_output`、`supports_native_reasoning`
- `context_window_tokens` 用于后端上下文压缩阈值计算，`max_output_tokens` 用于模型输出预算和请求参数上限，三个 `supports_*` 字段用于后端能力校验和调用路径选择；这些字段都不属于平台运行设置
- `高级设置` 中的字段都有后端默认值；前端不得暗示用户必须填写这些字段才能保存 Provider
- 该页面管理的是 Provider 自身配置；模板绑定区只能选择当前已配置且启用的 Provider
- 模板编辑区若发现缺少所需 Provider 或 Provider 已禁用，必须明确阻止保存或启动 run，并提供跳转或唤起该页面的路径
- Provider 配置变更只影响后续 run 或尚未启动 run 的模板选择；已经启动的 run 必须继续展示其已固化的 Provider 与模型绑定快照
- Provider API key 由用户在前端直接输入并提交；读投影不得回显真实 key，前端只显示空的密码输入框并在未触碰该字段时保留后端已配置状态
- 删除 `custom Provider` 后该 Provider 从页面中移除；删除内置 Provider 后该 Provider 取消配置并从普通列表隐藏

#### 7.4.3 导入导出

`导入导出` 页面用于当前 `Project` 作用域下用户可见配置的备份、迁移和环境复制。

该页面必须满足以下规则：
- 必须显式显示导入导出作用于哪个 `Project`
- 导出动作必须调用项目作用域配置包导出接口，导出内容只包含用户可见配置和引用信息
- 导出包不得包含真实密钥、平台隐性运行设置、`compression_threshold_ratio`、系统内置提示词正文、运行快照、历史 run、日志、审计正文或平台内部数据库路径
- 导出包中的 Provider 模型能力可以包含 `context_window_tokens`、`max_output_tokens`、`supports_tool_calling`、`supports_structured_output`、`supports_native_reasoning`；这些字段缺省时由后端默认值补齐
- 导入动作必须调用项目作用域配置包导入接口，并展示后端返回的变更摘要、受影响配置对象和字段级校验错误
- 导入成功后，前端必须刷新当前项目交付配置、Provider 列表和模板列表
- 导入 Provider、DeliveryChannel 或模板运行配置只影响后续新启动 run、尚未启动 run 的模板选择，或尚未固化交付通道快照的交付就绪校验；前端不得暗示导入会修改已启动 run 或历史 run
- 导入包中的 Provider API key 不在导出结果中回显；导入时用户显式提供的 key 只提交给后端，前端不得解析或展示已保存真实值

## 8. 中栏设计

中栏是整个工作台的核心，固定由以下两部分组成：

- `Narrative Feed`
- `Composer`

中栏的整体形态必须接近 `输入框 + 瀑布流` 的工作台模式，而不是传统阶段看板。

中栏的主呈现方式必须明确为：
- 一条会话式主瀑布流
- 一条能够连续追加内容的链路流
- 一条允许多种条目类型混合出现的统一主流
- 一条由显式阶段结点顺序组成的执行序列
- 一条在同一页面内串联多个 `PipelineRun` 分段的会话级运行视图

中栏不得被实现为以下任一单一形态：
- 纯聊天消息窗口
- 纯流程图或纯阶段图
- 纯日志控制台
- 纯卡片式看板

中栏必须同时满足以下要求：
- 在需求澄清阶段提供自然的多轮对话阅读体验
- 在正式执行阶段提供清晰的链路瀑布流阅读体验
- 在同一条主流中承载推理内容、决策内容、工具调用、diff、审批和结果
- 每进入一个新阶段时，主流中必须出现一个显式的阶段结点大框
- 阶段结点大框内部的内容必须随着执行推进逐步追加显示
- 当同一会话存在多个 `PipelineRun` 时，中栏必须按时间顺序连续展示多个 run 分段
- 不同 run 之间必须存在强分界，避免被误读为同一次连续运行

### 8.1 Run Switcher

当同一 `Session` 存在多个 `PipelineRun` 时，中栏必须提供 `Run Switcher`。

`Run Switcher` 必须满足以下规则：
- 默认聚焦当前会话的最新运行；若存在正在执行中的运行，默认聚焦该运行
- 触发后应跳转到所选 run 的分界起点，或将其滚动定位到可见区域
- 不同 run 的 Narrative Feed 必须保留在同一会话页面内，不因切换动作被整体替换为单 run 页面
- `Run Switcher` 只负责定位与导航，不承担新建会话、重新运行或模板编辑职责
- run 切换后，左栏当前会话选中态保持不变
- 历史 run 必须至少能区分：运行次序、状态、启动时间、结束时间或最近更新时间
- run 分界标题必须至少能与 `Run Switcher` 列表一一对应

### 8.2 空白会话与模板配置

在用户尚未提交首条需求消息之前，中栏必须以 `空白会话` 形态承载模板选择与模板编辑能力。
该模板区属于 `Narrative Feed` 的空态内容，不构成独立于 `Narrative Feed` 和 `Composer` 之外的第三主区。

空白会话必须满足以下规则：
- 系统启动时必须提供至少三个可选 `system_template`：`Bug 修复流程`、`新功能开发流程`、`重构流程`
- 新建会话时默认预选 `新功能开发流程` 模板，供用户直接开始输入
- 用户可以在空白会话中切换 `system_template` 与 `user_template`
- 模板编辑只允许修改功能一 V1 已开放的配置项
- 模板编辑不得改变固定阶段骨架，不得删除或关闭必需审批检查点
- 模板配置区与 Composer 必须共存于同一工作台，不得跳转独立模板管理页面后才能开始会话

三个预置 `system_template` 必须共享同一固定阶段骨架。
三者的默认差异只允许体现在以下方面：
- 各必需角色槽位默认绑定的 `AgentRole`
- 各 `AgentRole` 的默认 `system_prompt`
- 各 `AgentRole` 绑定的默认内置 `Provider`
- 自动回归默认策略

前端在模板选择区域必须能清楚区分三个预置模板的适用场景：
- `Bug 修复流程`：用于已有缺陷、失败测试、报错修复
- `新功能开发流程`：用于新增业务能力、接口或页面功能
- `重构流程`：用于不改变外部行为前提下的结构整理与可维护性提升

模板编辑区在 V1 至少支持以下字段：
- 为每个阶段的必需 `AgentRole` 槽位选择具体 `AgentRole`
- 查看并编辑所选 `AgentRole` 的 `system_prompt`
- 为每个 `AgentRole` 绑定一个 `Provider`
- 配置 `自动回归是否开启`
- 配置 `最大自动回归重试次数`

模板编辑区在 V1 不支持以下字段：
- 修改 `AgentRole` 名称
- 查看、编辑或切换后端系统内置提示词资产、`prompt_id`、`prompt_version`、`runtime_instructions`、结构化输出修复提示词或 `compression_prompt`

这些可编辑字段必须被理解为 `运行配置`：
- 它们服务于后端后续阶段 Agent 的实际执行行为
- 它们不是只用于前端展示的说明性信息
- 首次运行启动后，本次 run 所绑定的配置必须冻结为快照
- `AgentRole` 在这里应被理解为模板内阶段生效配置的预设来源；前端编辑后保存的是当前模板各阶段槽位最终生效的配置，而不是修改一个会影响其他模板的共享运行对象
- `最大自动回归重试次数` 必须受后端平台硬上限约束；当用户输入超过上限或后端拒绝保存时，前端必须展示明确错误，不得自行截断后保存
- 后端平台运行上限不进入模板配置 UI；模板编辑区只展示当前模板允许用户编辑的自动回归开关和最大自动回归次数

Provider 选择区域必须满足以下规则：
- 只显示当前已配置且启用的 Provider，不把未配置或已禁用的 Provider 当作默认可选项
- 允许用户通过统一设置弹窗中的 `模型提供商` 页面新增一个或多个 `custom Provider`
- 新增后的 `custom Provider` 以用户自定义名称出现在 Provider 选择列表中
- `OpenAI Completions compatible` 只作为自定义 Provider 的接入协议存在；`OpenAI Completions` 可以作为新增 Provider 的接入模板名称展示
- 模板编辑区若发现缺少所需 Provider 或 Provider 已禁用，必须展示不可用原因，阻止保存或启动 run，并提供打开设置弹窗并定位到 `模型提供商` 页面的入口

以下内容在 V1 中不进入模板配置 UI：
- 核心业务阶段的存在性
- 核心业务阶段顺序
- 方案设计审批检查点（`solution_design_approval`）
- 代码评审审批检查点（`code_review_approval`）
- 项目级 `DeliveryChannel` 配置
- 阶段输入输出契约
- 阶段结构化产物要求
- 工具权限边界
- 后端系统内置提示词资产、提示词版本切换、`runtime_instructions`、结构化输出修复提示词或 `compression_prompt`
- 平台运行上限
- 环境变量或运行数据目录

前端可以通过简短说明文本告知用户当前使用固定研发流程，但不得把上述内容实现为只读配置表单、只读配置列表或占据主要视觉层级的固定说明面板。

模板修改后的保存语义必须满足以下规则：
- 当来源模板是 `system_template` 时，只允许 `一键使用` 或在修改后 `另存为新模板`
- 当来源模板是 `user_template` 时，必须同时支持 `覆盖当前模板` 与 `另存为新模板`
- 当来源模板是 `user_template` 时，必须提供 `删除模板` 入口
- `删除模板` 入口只允许作用于 `user_template`，不得对 `system_template` 展示
- 删除成功后，当前空白会话必须自动切换到一个仍然可用的模板；当当前被删模板正被选中时，优先回退到默认系统模板 `新功能开发流程`
- 覆盖或另存成功后，当前空白会话必须立即切换到新保存的模板结果
- 模板保存影响的是后续运行，不得回写修改已经开始执行的运行

首条需求消息提交后：
- 当前会话使用已选模板启动执行
- 本次运行绑定的模板配置视为已冻结
- 模板配置区只服务于启动前，不再作为运行中主界面的一部分保留
- 主界面必须完全切回 `Narrative Feed + Composer`
- 审批块作为 `Narrative Feed` 内的顶层交互条目出现，不构成独立于 `Narrative Feed` 和 `Composer` 之外的第三主区
- 前端不得再把本次运行中的模板配置展示为可直接编辑状态
- 前端只保留当前运行所用模板的只读摘要，不保留模板编辑入口

## 9. Composer 设计

Composer 固定在中栏底部，是用户在整个会话周期中的 `需求输入 / 澄清输入统一入口`。

Composer 必须同时承担以下职责：
- 初始需求输入
- 需求澄清阶段的补充信息输入

Composer 规则如下：
- 在同一会话内不更换为其他主输入控件
- 当系统需要用户补充信息时，Composer 应显式进入对应输入语义
- 输入发送后，新内容必须立即追加到叙事流中
- Composer 右端必须承载运行生命周期按钮区域，用于表达 `发送`、`暂停`、`恢复` 三种互斥状态

Composer 右端生命周期按钮必须满足以下规则：
- 当会话处于 `draft` 时，按钮显示为 `发送`
- 当当前活动 run 处于 `waiting_clarification`，且系统正在等待人工回复时，输入框保持激活，按钮显示为 `发送`
- 当当前活动 run 处于 `waiting_clarification` 且主按钮承载 `发送` 语义时，前端仍必须通过当前活动 run 的分界头部、顶部工具区或等价次级入口暴露 `暂停` 动作，确保运行中任意阶段都可暂停
- 当当前活动 run 处于 `running` 且 `current_stage_type = requirement_analysis` 时，前端必须将其理解为聊天式澄清链路中的 `等待 Agent 继续回复 / 继续分析` 回合；此时 Composer 输入框不承担发送动作，按钮显示为 `暂停`
- 当当前活动 run 处于自动执行中时，按钮显示为 `暂停`
- 当当前活动 run 处于 `waiting_approval` 时，按钮保持运行态语义，显示为 `暂停`
- 当当前活动 run 处于 `waiting_tool_confirmation` 时，按钮保持运行态语义，显示为 `暂停`；工具确认的主操作必须出现在 Narrative Feed 中的工具确认块内
- 当当前活动 run 处于 `paused` 时，按钮显示为 `恢复`
- 当当前活动 run 已进入 `solution_design` 及之后的正式研发链路时，只要 run 未暂停且未进入终态，Composer 右端按钮必须持续保持 `暂停` 语义
- 当当前活动 run 处于 `failed`、`terminated` 或 `completed` 时，按钮必须继续保留在 Composer 右端，但处于禁用状态
- 当当前活动 run 处于 `completed` 时，Composer 整体按只读结果态展示；右端按钮仅保留禁用的生命周期占位，不提供重新尝试或恢复语义
- 当按钮处于 `暂停` 或 `恢复` 语义时，优先表达运行控制，不与普通消息发送动作混用
- 页面中即使展示历史 run，该按钮区域仍然绑定当前活动 run；历史 run 不夺取该按钮的控制目标

Composer 不承担以下职责：
- 人工审批拒绝时的补充理由输入
- 高风险工具确认的允许或拒绝主操作
- 正式执行阶段中的通用追加说明输入

当进入正式执行链路且当前不存在需求澄清请求时：
- Composer 不作为主交互入口使用
- 中栏以阅读执行链路为主

除 Composer 右端按钮外，中栏右上角必须提供独立且更显眼的 `终止` 动作，用于终止当前活动 run。

`终止` 动作必须满足以下规则：
- 只对当前活动 run 生效
- 页面中即使存在历史 run，该动作也只作用于当前活动 run
- 触发前必须要求用户确认
- 允许在当前活动 run 仍处于链路执行过程中的任意阶段调用，不因运行停留在 `waiting_approval`、`waiting_tool_confirmation` 等中间态而隐藏
- 执行后不关闭或移除该 run 已存在的审批块、澄清记录或其他历史条目
- 执行后该 run 尾部必须出现一个明确的终止提示条目

## 10. Narrative Feed 设计

Narrative Feed 是主瀑布流，是本系统最核心的前端表现。

Narrative Feed 在产品体验上的定位必须进一步明确如下：
- 它是中栏唯一的主内容流
- 它不是单纯的聊天记录流
- 它也不是脱离会话体验的独立链路图
- 它必须同时承载 `Requirement Analysis` 结点内部的多轮需求澄清对话与后续执行链路瀑布流
- 它的外部体验应接近对话产品的主窗口
- 它的内部内容语义应体现流程系统的执行链路

Narrative Feed 的主呈现方式必须是：
- `会话式瀑布流`

该主呈现方式的含义如下：
- 用户从视觉上沿一条连续主流向下阅读
- 新内容持续追加到流尾部
- 不同阶段以显式结点顺序出现
- 每个结点内部再承载不同条目形态
- 内容可以折叠、展开、嵌套，但不得破坏主流连续性

### 10.1 基本要求

Narrative Feed 必须满足以下要求：
- 内容按时间顺序自上而下生长
- 新内容实时追加到流尾部
- 已有内容不应因为后续状态变化而被整体重排
- 用户能够在一条连续链路中看到需求、澄清、执行、审批与交付
- 用户能够清楚区分“阶段结点”与“结点内部条目”

### 10.2 顶层条目与结点内部条目模型

Narrative Feed 必须采用 `顶层条目 + 结点内部条目` 的两层模型。

第一层是 `顶层条目`：
- 用户输入条目
- 阶段结点
- 审批请求块
- 高风险工具确认块
- 控制型条目
- 审批结果条目
- 交付结果条目
- 系统状态条目

其中 `交付结果条目` 必须满足以下规则：
- 中栏结果块承载的是 `delivery_result` 顶层条目的高信号摘要，不承担完整交付详情承载职责
- `delivery_result` 顶层条目必须提供当前正式 feed 契约已公开的交付摘要、模式特定结果要点和深看入口
- 完整输入来源、原始交付过程记录、完整输出、完整产物与量化信息必须通过右栏 Inspector 的 `DeliveryResultDetailProjection` 查看
- 前端可以在 `delivery_result` 顶层条目与 `DeliveryResultDetailProjection` 之上构建共享显示结构，但该结构属于前端内部展示模型，不构成新的后端公开投影契约

其中 `阶段结点` 至少包括：
- Requirement Analysis
- Solution Design
- Code Generation
- Test Generation & Execution
- Code Review
- Delivery Integration

其中 `控制型条目` 至少包括：
- 澄清等待提示
- 回退说明
- 重试说明

第二层是 `结点内部条目`：
- 对话条目
- 推理与决策条目
- 执行动作条目
- 结果条目

顶层条目与结点内部条目的关系必须满足以下规则：
- 阶段结点是主流中的阶段级大框
- 结点内部条目是某个阶段结点内部逐步出现的内容单元
- 同一会话中的不同 run 分段只作为页面内的纯视觉分界与导航锚点存在，不作为正式顶层条目类型参与 Narrative Feed 数据建模
- 初始需求输入以顶层用户输入条目出现
- `Requirement Analysis` 阶段内的澄清问答挂载在该结点内部
- 审批请求、审批结果和交付结果以顶层条目形式插入主流
- 高风险工具确认以顶层交互块形式插入主流，不作为 Approval Request，也不作为阶段结点内部条目
- 澄清等待、回退与重试说明以顶层控制型条目形式插入主流，不作为正式阶段结点
- 回退控制条目必须表达“为何需要订正、回到哪个正式阶段、历史链路如何保留、接下来如何继续”，不得表达为撤销或抹除既有执行
- `system_status` 只允许作为顶层条目出现，不挂载为任何阶段结点的内部条目
- `failed` 与 `terminated` run 的终态提示属于独立 `system_status` 顶层条目，不属于 `control_item`
- `completed` run 以 `delivery_result` 作为尾部收束条目，不追加完成态 `system_status`
- `failed` 与 `terminated` run 必须在各自 run 尾部追加顶层 `system_status` 终态条目
- 高风险工具确认被拒绝后，若后端未找到低风险替代路径并使 run 进入 `failed`，Narrative Feed 必须在当前 run 尾部追加 `system_status` 终态条目
- 只有阶段结点拥有内部条目
- 用户阅读顺序以顶层条目顺序为主，以结点内条目顺序为辅
- 不允许把所有内容直接摊平成无阶段归属的单层瀑布流

结点内部条目必须至少支持以下内容类型：
- 系统澄清提问
- 用户澄清回复
- 部分推理思考片段
- 关键决策说明
- 工具调用记录
- Provider 调用状态
- diff 预览
- 阶段结果内容
- 回退原因与下一步执行说明

换言之，Narrative Feed 是一条统一主流：
- 开始部分可以是顶层需求输入与 `Requirement Analysis` 结点内的多轮澄清对话
- 后续部分可以是结构化阶段结点序列与穿插其中的审批块、工具确认块、审批结果和交付结果
- 两者之间不切页面、不切模式、不切主视图

### 10.3 结点内条目层级与密度规则

结点内部条目必须按信息密度分层显示，不允许所有条目用同一信息强度铺满主流。

条目层级至少分为以下四类：
- `对话条目`
- `推理与决策条目`
- `执行动作条目`
- `结果条目`

`对话条目` 包括：
- 系统澄清提问
- 用户澄清回复

`推理与决策条目` 包括：
- 部分推理思考片段
- 关键决策说明
- 回退原因
- 下一步执行说明

`执行动作条目` 包括：
- 工具调用记录
- 代码读取记录
- 搜索记录
- 命令执行记录
- 修改代码记录
- diff 预览

`结果条目` 包括：
- 需求理解结果
- 方案设计结果
- 方案校验结果
- 代码实施结果
- 测试生成与执行结果
- 代码评审结果
- 阶段内修正后的最新结果

不同层级条目必须遵循以下密度规则：
- `对话条目` 优先保证连续阅读体验
- `推理与决策条目` 优先展示高信号结论
- `执行动作条目` 优先保持紧凑，不得默认展开全部日志
- `结果条目` 优先展示当前阶段最重要的阶段性结论
- 平台级运行日志不属于 Narrative Feed 的默认展示内容；Narrative Feed 只呈现经过后端投影后的高信号过程条目、控制条目与结果条目

### 10.4 需求澄清规则

需求澄清必须满足以下要求：
- 不作为独立页面呈现
- 不作为独立审批卡呈现
- 必须挂载在 `Requirement Analysis` 结点中发生
- 必须直接作为 `Requirement Analysis` 结点内部的连续对话内容出现

澄清交互流程如下：
- 用户先输入原始需求
- 主流中显式出现 `Requirement Analysis` 结点
- 系统在该结点中分析当前信息是否足够
- 当信息足够时，该结点完成并进入后续结点
- 当信息不足时，该结点在主链路中显式中断，并在结点内部提出澄清问题
- 等待人工回复期间，Composer 保持聊天输入激活，右端主按钮为 `发送`
- 用户继续使用 Composer 回复
- 用户提交后，当前结点恢复执行；在 Agent 继续分析与继续回复期间，Composer 输入框不承担发送动作，右端主按钮切回 `暂停`
- `Requirement Analysis` 结点继续分析补充后的信息
- 系统可在同一结点内进行多轮澄清
- 澄清完成后，该结点结束并进入 `Solution Design` 或其他后续结点

这意味着需求澄清在前端上应被理解为：
- 同一主流中 `Requirement Analysis` 结点内部的多轮对话片段
- 正式执行链路的一部分，而不是链路外附加流程
- 一个显式阶段结点在信息不足时的中断与恢复过程

### 10.5 结点内条目显示规则

结点内部不同类型条目必须采用不同显示规则。

#### 10.5.1 部分推理思考片段

部分推理思考片段必须满足以下规则：
- 只展示可对外显示的原生推理内容
- 默认按原生内容截断显示
- 默认显示 1 到 3 行原生内容
- 支持用户主动展开查看更多
- 不得在默认状态下铺开长篇思考文本

#### 10.5.2 关键决策说明

关键决策说明必须满足以下规则：
- 默认直接展示原生决策内容
- 必须说明为何做出该决策
- 必须说明该决策对后续链路的影响
- 原生决策内容过长时必须截断显示
- 不得只显示一个结论标签而无原因

#### 10.5.3 工具调用记录

工具调用记录必须满足以下规则：
- 默认展示原生工具调用信息中的工具名称、目标对象与命令或参数原文片段
- 原文片段过长时必须截断显示
- 工具调用的详细执行内容必须折叠在该条目下
- 长输出、命令输出、搜索命中列表不得默认全部铺开

工具调用条目的典型显示形式包括：
- `read_file app/orchestrator.ts`
- `grep "PipelineRun" src/`
- `glob src/workspace/**`
- `edit 3 files`
- `bash pytest tests/pipeline`

#### 10.5.4 Provider 调用状态

Provider 调用状态必须满足以下规则：
- 默认展示 Provider 名称、模型名称或模型绑定摘要、调用状态和耗时
- 当发生超时、网络错误或限流错误并进入重试时，必须展示当前重试次数、指数退避等待状态和下一次尝试时间或等待摘要
- 当连续失败达到熔断条件时，必须展示熔断状态、失败原因摘要和对当前阶段的影响
- Provider 调用状态不得暗示前端会自动切换已经启动 run 的 Provider 或模型绑定快照
- 完整调用过程、错误详情、重试轨迹和熔断状态必须在 Inspector 中可见

#### 10.5.5 diff 预览

diff 预览必须满足以下规则：
- 默认显示原生影响文件列表和代表性变更片段
- 默认不展示全量 diff
- 支持用户展开查看更多 diff 片段
- 超长 diff 必须继续折叠，不得一次性完全铺开

#### 10.5.6 审批块

审批块属于 Narrative Feed 中的顶层交互块，不属于某个执行结点的内部条目。

审批块必须满足以下规则：
- 明显区别于普通执行条目
- 默认展示当前审批对象内容、审批参考内容和操作按钮
- 必须支持 `Approve`
- 必须支持 `Reject`
- `Reject` 后必须直接提供补充输入能力

#### 10.5.7 高风险工具确认块

高风险工具确认块属于 Narrative Feed 中的顶层交互块，不属于某个执行结点的内部条目，也不属于 Approval Request。

高风险工具确认块必须满足以下规则：
- 明显区别于普通执行条目和人工审批块
- 默认展示工具名称、命令或参数摘要、目标路径或目标资源、触发原因、风险级别、预期副作用和后端处理说明
- 必须支持 `允许本次执行`
- 必须支持 `拒绝本次执行`
- 当当前活动 run 处于 `paused` 时，工具确认块必须禁用提交；`恢复` 后应回到同一个待确认工具动作继续等待用户决定
- 当工具确认块属于历史 run、已终止 run 或已失败 run 时，必须退化为只读历史态
- 用户拒绝后，前端不得表达为审批拒绝，也不得暗示系统会自动回到 `Solution Design` 或 `Code Generation`
- 若后端返回存在低风险替代路径，工具确认块必须展示后续将继续当前阶段；若后端返回无替代路径，必须展示当前 run 将失败或等待用户显式暂停、终止

### 10.6 结点展示规则

每个执行结点必须具备最少公共信息：
- 结点标题
- 结点类型
- 当前状态
- 执行顺序
- 第几次尝试
- 开始时间与结束时间
- 是否可展开

以下公共信息只构成结点外壳，不构成完整正文。

执行结点正文必须根据结点类型分别设计，不允许把所有结点压缩为统一的短摘要模板。

结点正文的主呈现规则如下：
- 结点首先以一个显式的大框出现
- 结点内部内容随着执行推进逐步追加
- 结点内部按照 `10.5` 定义的条目类型展示原生内容
- 当条目内容过长时，默认采用截断显示，而不是改写为新的摘要文本
- 中栏正文优先展示原生推理内容、原生决策内容、工具调用原文片段、diff 片段和阶段结果原文片段
- 结点内部不展示该结点的全量信息
- 更全面的信息统一放入右侧 Inspector

结点内部必须按阶段语义逐步显示足够的阶段性信息，使用户在中栏中无需依赖右侧栏也能理解当前阶段在做什么、为何这样做、当前卡在哪里以及下一步将如何继续。

凡属于 `10.5` 中定义的结点内条目，且能够直接帮助用户理解当前阶段状态、阻塞原因、回退原因和后续推进路径的内容，都应直接在中栏显示；当原生内容过长时，使用截断而不是重新摘要。

### 10.7 右栏 Inspector 中的结点完整信息要求

执行结点在中栏中不承担全量展开职责。
结点更完整的信息必须通过右侧 Inspector 查看。

Inspector 承接的是阶段结点、工具确认块、可深看的控制型条目或交付结果的完整信息，不承接 `Approval Request` 审批块本身的独立详情页。

Inspector 必须被定义为 `公开透明的信息盒子`。
对任一可打开对象，Inspector 都必须尽可能按 `接收输入 -> 过程记录 -> 结果输出 -> 关联引用与附件 -> 量化信息` 的顺序展开。
这里的“完整信息”指后端已经流通或持久化的原始信息；前端可以对长内容做折叠、分组、语法高亮与原文展开，但不得把原始信息压缩成只剩摘要的二次解释版本。

### 10.7.1 `requirement_analysis` 在 Inspector 中的完整信息

Inspector 中必须显示：
- 本阶段接收的原始需求输入全文与每次补充输入全文
- 完整结构化需求
- 完整验收标准
- 完整约束与假设列表
- 完整待确认事项列表
- 完整澄清记录，包括系统问题与用户回复
- 系统如何从原始需求推导到结构化需求的决策轨迹
- 本阶段的原始过程记录，如推理片段、决策片段、检索或工具调用记录
- 本结点使用的关键上下文
- 本阶段输出快照及其稳定引用
- 本结点的量化信息

### 10.7.2 `solution_design` 在 Inspector 中的完整信息

Inspector 中必须显示：
- 本阶段接收的结构化需求、澄清结论与前序产物引用
- 完整方案说明
- 完整可执行任务计划，包括任务拆分、目标文件或模块、执行顺序、验证动作、依赖假设和风险处理
- 完整影响范围
- 完整设计决策列表
- 关键文件或模块清单
- 风险分析
- 方案校验结论
- 方案校验检查项
- 方案校验发现的问题及依据
- 方案校验通过项与未通过项
- 若校验失败，明确修正方向
- 方案形成过程中的关键决策轨迹
- 本阶段的原始过程记录，如推理、设计权衡、校验过程与工具调用记录
- 本阶段输出快照及其稳定引用
- 本结点的量化信息

方案校验不作为独立阶段结点或独立右栏打开对象。
其完整信息必须合并在 `solution_design` 的 Inspector 中查看。

### 10.7.3 `code_generation` 在 Inspector 中的完整信息

Inspector 中必须显示：
- 本阶段接收的方案输入、上下文引用与目标变更范围
- 完整变更说明
- 完整影响文件列表
- 完整 diff、变更片段或稳定变更引用；若中栏只展示片段，Inspector 中不得只保留代表性片段
- 关键实现决策
- 本次实施与方案设计的对应关系
- 过程中执行的原始过程记录，如工具调用、文件修改、命令执行与失败重试记录
- 本阶段输出快照及其稳定引用
- 本结点的量化信息

### 10.7.4 `test_generation_execution` 在 Inspector 中的完整信息

Inspector 中必须显示：
- 本阶段接收的代码变更输入、测试目标与相关上下文引用
- 新增或修改的测试对象清单
- 测试运行环境识别结果、依赖缺失判断和命令风险判断
- 测试执行结果详情
- 测试执行原始过程记录，如测试命令、stdout / stderr、退出码、Provider 重试状态或等价过程记录
- 高风险工具确认记录，包括确认对象、命令、目标路径、用户决定和后续处理结果
- 失败项详情
- 测试缺口分析
- 关键测试决策轨迹
- 本阶段输出快照及其稳定引用
- 本结点的量化信息

### 10.7.5 `code_review` 在 Inspector 中的完整信息

Inspector 中必须显示：
- 本阶段接收的方案、代码变更、测试结果与历史意见输入
- 完整评审结论
- 完整问题列表
- 每个问题的依据
- 风险分级与说明
- 评审过程中的原始过程记录，如推理、比对、工具调用与问题定位记录
- 是否触发回退以及触发原因
- 本阶段输出快照及其稳定引用
- 本结点的量化信息

### 10.7.6 高风险工具确认在 Inspector 中的完整信息

高风险工具确认不作为正式业务阶段结点打开，而是作为顶层交互块或可深看的工具确认条目打开。

Inspector 中必须显示：
- 工具确认标识、所属 run、所属阶段和状态
- 触发确认的工具名称、命令或参数全文、目标路径或目标资源
- 风险级别、风险分类、触发原因和预期副作用
- 关联的项目说明、脚本配置、依赖声明、工具观察或测试输出引用
- 用户确认决定、决定时间和拒绝原因或备注（如适用）
- 用户同意后的实际工具执行结果、审计记录和副作用引用
- 用户拒绝后的替代路径判断、后续阶段状态或 run 终态结果
- 本次确认的量化信息和关联日志摘要

### 10.7.7 回退或重试控制条目在 Inspector 中的完整信息

回退或重试不作为正式业务阶段结点打开，而是作为控制型条目打开。

Inspector 中必须显示：
- 触发回退或重试的源头
- 本次控制动作接收的原始上下文与原始触发载荷
- 本次控制动作关联的业务阶段
- 本次回退携带的上下文
- 本次控制动作的原始过程记录
- 先前执行链路如何被保留
- 将重新进入的目标阶段
- 历史尝试记录
- 本次控制动作产出的结果快照与稳定引用
- 本次回退或重试的量化信息

### 10.7.8 `approval_result` 在 Inspector 中的完整信息

`approval_result` 不作为独立右栏打开对象。
其完整信息应通过所属阶段结点的 Inspector 查看。

Inspector 中必须显示：
- 审批决定
- 当审批为 `Reject` 时的用户拒绝理由全文
- `Approve` 不要求携带备注；未提供备注时不得展示空备注占位
- 审批后链路如何继续推进

### 10.7.9 `delivery_result` 在 Inspector 中的完整信息

`delivery_result` 是顶层结果条目，不替代 `delivery_integration` 阶段过程展示。
中栏只展示摘要；右栏 Inspector 承载完整交付详情。

Inspector 中必须显示：
- 生成最终交付结果时接收的上游输入引用，如交付快照、评审结果与测试结果引用
- 最终交付说明全文
- 最终变更结果
- 最终测试结论
- 最终评审结论
- 与最终交付结果直接相关的原始交付过程记录或其稳定引用
- 分支、提交、MR/PR 或交付对象详情
- 交付产物与相关稳定引用
- 本顶层条目的量化信息

### 10.7.10 `delivery_integration` 在 Inspector 中的完整信息

`delivery_integration` 是正式阶段结点，用于表达交付执行过程；`delivery_result` 是该阶段成功完成后追加的最终结果条目。

Inspector 中必须显示：
- 本阶段接收的审批结果、交付快照与上游产物引用
- 本次读取的 `delivery_mode`
- 已固化的交付快照摘要
- 本次交付目标，如仓库、分支、代码评审请求类型或演示目标
- 交付步骤的原始过程记录，如分支准备、提交创建、推送与 MR/PR 创建
- 各步骤执行结果与失败原因
- 已生成的中间交付物或引用
- 本阶段输出快照及其稳定引用
- 本结点的量化信息

### 10.8 量化指标定义与展示规则

运行量化信息必须在执行结点或相关顶层条目中可见。

### 10.8.1 通用量化指标

所有适用的执行结点必须至少支持以下通用量化指标：
- `duration_ms`
- `input_tokens`
- `output_tokens`
- `total_tokens`
- `attempt_index`

### 10.8.2 类型专项量化指标

不同对象类型必须支持各自专项量化指标，不适用的指标不得硬性展示为 `0` 或空占位。

`requirement_analysis`、`solution_design`、`code_review` 必须至少支持：
- `context_file_count`
- `reasoning_step_count`
- `tool_call_count`

`code_generation` 必须至少支持：
- `context_file_count`
- `tool_call_count`
- `changed_file_count`
- `added_line_count`
- `removed_line_count`

`test_generation_execution` 必须至少支持：
- `tool_call_count`
- `generated_test_count`
- `executed_test_count`
- `passed_test_count`
- `failed_test_count`
- `skipped_test_count`
- `test_gap_count`

回退或重试控制条目必须至少支持：
- `retry_index`
- `source_attempt_index`

顶层审批条目在当前版本不强制要求额外量化指标。

`delivery_result` 必须至少支持：
- `delivery_artifact_count`
- `changed_file_count`

展示规则如下：
- 结点正文必须展示对当前理解最有帮助的少量高信号指标
- 全量量化信息必须在右侧 Inspector 中可见
- 右侧 Inspector 中必须存在更完整的量化信息展示区
- 历史会话回访时，量化信息必须保留
- 不适用的指标必须隐藏，不允许使用统一空模板占位

## 11. 人工审批设计

人工审批属于 Narrative Feed 中的显式交互型内容，不属于独立页面。

### 11.1 审批出现方式

当系统运行到正式人工审批检查点时，Narrative Feed 中必须插入一个 `Approval Request`。

功能一 V1 的人工审批检查点范围固定如下：
- `Solution Design` 相关审批
- `Code Review` 相关审批

以下场景不属于人工审批检查点：
- `Requirement Analysis` 阶段内的需求澄清
- `Delivery Integration` 阶段的交付结果整理与展示

Approval Request 必须显式展示：
- 当前为何需要人工审批
- 当前审批对象内容
- 当前风险或影响内容
- `Approve`
- `Reject`

当审批块属于 `Code Review` 相关审批时，还必须满足以下规则：
- 若当前项目 `delivery_mode = git_auto_delivery` 且交付配置或凭据尚未达到 `ready`，审批块必须显式展示交付配置未就绪的阻塞原因
- 上述阻塞态下，审批块必须提供打开设置弹窗并定位到 `通用配置` 页面的入口
- 上述阻塞态下，`Approve` 必须不可提交，直到交付配置状态恢复为 `ready`
- 若当前项目 `delivery_mode = demo_delivery`，审批块不得因交付配置或凭据缺失而额外阻塞

当审批块属于历史 run 时：
- 审批块必须退化为只读历史态
- 不展示可点击的 `Approve` / `Reject` 主操作
- 只展示历史审批状态、审批对象摘要与审批结果
- 若当前活动 run 已进入 `terminated`，即使该审批块原本属于当前活动 run 且仍待处理，也必须退化为不可提交状态
- 若当前活动 run 只是 `paused` 且审批仍待处理，审批块必须退化为禁用待处理态，不允许提交 `Approve` / `Reject`
- 上述禁用待处理态必须明确表达“当前运行已暂停，恢复后继续等待审批”

### 11.2 审批交互规则

审批必须满足以下规则：
- `Approve` 可直接提交
- `Reject` 后必须在审批块内联提供补充输入区
- 用户提交 rejection 理由后，该内容必须追加到 Narrative Feed
- 审批结果必须作为链路的一部分继续显示
- 当当前活动 run 处于 `paused` 时，审批块必须禁用提交；`恢复` 后应回到同一个 `waiting_approval` 检查点继续等待审批
- 交付配置或凭据相关提示只允许出现在人工审批阶段，不允许在 `Delivery Integration` 阶段以配置弹窗或额外阻塞形式再次出现

审批结果必须至少出现以下两类叙事内容：
- `审批通过，系统继续进入下一个执行结点`
- `审批拒绝，系统根据用户补充信息回退到 Code Generation 并开始订正性重执行`

### 11.3 审批与澄清的边界

需求澄清与人工审批必须严格区分：
- 需求澄清使用普通会话输入与系统提问
- 人工审批使用 Approval Request 交互块
- 需求澄清不出现 `Approve` / `Reject`
- 审批请求不替代前置需求澄清

### 11.4 审批与工具确认的边界

人工审批与高风险工具确认必须严格区分：
- 人工审批使用 Approval Request 交互块，审批对象是阶段产物质量
- 高风险工具确认使用 Tool Confirmation 交互块，确认对象是具体工具动作和预期副作用
- 高风险工具确认不出现在人工审批检查点范围内
- 高风险工具确认不使用 `Approve` / `Reject` 文案，必须使用 `允许本次执行` / `拒绝本次执行` 或等价权限动作
- 工具确认拒绝不得展示为审批拒绝，也不得触发代码评审拒绝后的回归叙事
- 方案设计审批与代码评审审批仍是 V1 仅有的两个人工审批检查点

## 12. 右栏 Inspector 设计

Inspector 是按需展开的结构化详情区域，默认关闭。

### 12.1 打开方式

Inspector 必须通过以下动作打开：
- 点击某个执行结点
- 点击某个工具确认块
- 点击某个支持详情查看的控制型条目
- 点击某个交付结果块

### 12.2 职责边界

Inspector 的职责如下：
- 展示某个阶段结点接收的完整输入、过程记录、结果输出、关联引用、附件与量化信息
- 展示工具确认块对应的确认对象、风险判定、用户决定、后续处理、审计记录与副作用引用
- 展示可深看的控制型条目接收的上下文、过程记录、结果输出、关联引用与量化信息
- 展示交付结果对应的完整输入来源、过程记录、结果输出、交付产物、稳定引用与量化信息

Inspector 不承担审批主操作职责。

Inspector 不承担以下职责：
- 替代主链路阅读
- 承担主要输入流程
- 取代 Narrative Feed 的叙事功能

### 12.3 内容规则

点击普通执行结点时，Inspector 必须至少展示：
- 结点标题
- 结点类型与状态
- 标识字段与时间信息，如 `stage_run_id`、开始时间、结束时间
- 中栏中已展示的核心信息
- 本阶段接收的完整输入与上下文引用
- 本阶段的完整过程记录
- 本阶段的完整结构化输出
- 全量推理或决策过程
- 全量量化信息
- 关联产物、附件与稳定引用
- 该阶段关联的审批结果或控制记录引用（如适用）

点击工具确认块时，Inspector 必须至少展示：
- 工具确认标识、所属 run、所属阶段和状态
- 触发确认的工具名称、命令或参数全文、目标路径或目标资源
- 风险级别、风险分类、触发原因和预期副作用
- 用户确认决定、决定时间和拒绝原因或备注（如适用）
- 后续工具执行结果、替代路径判断、审计记录和副作用引用
- 相关量化信息

点击可深看的控制型条目时，Inspector 必须至少展示：
- 控制条目类型
- 关联 run 与关联业务阶段
- 标识字段与时间信息
- 接收的原始上下文与触发原因
- 原始过程记录与触发载荷
- 后续推进目标、结果输出或终态说明
- 关联附件与稳定引用
- 相关量化信息

点击交付结果时，Inspector 必须至少展示：
- 标识字段与时间信息
- 最终交付结果接收的输入来源与引用
- 交付说明
- 原始交付过程记录或其稳定引用
- 变更结果
- 测试结论
- 评审结论
- 分支、提交、MR/PR 与交付产物信息
- 全量量化信息

Inspector 内容规则如下：
- Inspector 中的信息量必须大于中栏结点正文
- Inspector 中的信息不得弱化本规格已承诺的结点完整信息
- Inspector 中必须保留输入、过程、输出、引用与量化信息的完整区块
- Inspector 默认展示的是后端已经流通或持久化的原始信息，而不是前端重新生成的摘要文本
- 只要某项原始信息与当前打开对象直接相关，且已在后端形成稳定记录、结构化载荷或稳定引用，Inspector 中就必须可见
- 前端只允许对 Inspector 做呈现层处理，如分组、折叠、排序、语法高亮、原文展开与复制；不得改写语义、抹平过程差异或删除关键过程信息
- Inspector 作为深看区域，不得替代 Narrative Feed 的主叙事职责
- 审批动作的主操作入口保留在中栏审批块，不作为右栏主操作入口
- Inspector 可以展示与当前对象直接相关的日志摘要、裁剪后的日志片段或诊断定位信息，但不承担全量日志浏览、日志搜索、运行审计台账与系统可观测性职责
- 平台级日志审计能力服务于后端运行治理；前端不得依赖日志记录拼装阶段状态、审批状态、交付结果或 Narrative Feed 条目语义

## 13. 会话生命周期中的前端行为

### 13.1 新建会话

用户在当前项目下新建会话后：
- 中栏进入新的空白会话
- 空白会话默认展示 `新功能开发流程` 模板及其配置摘要
- 用户在首条需求消息发送前可以切换模板并编辑允许字段
- Composer 立即可输入
- 首条用户消息提交后，后端自动创建并启动该会话的执行链路

模板编辑交互必须满足以下规则：
- 当模板配置存在未保存修改时，前端必须以显式状态提示当前模板处于脏状态
- 若用户尝试以脏 `system_template` 直接启动运行，前端必须先要求用户 `另存为新模板` 或放弃未保存修改
- 若用户尝试以脏 `user_template` 直接启动运行，前端必须先要求用户完成 `覆盖当前模板`、`另存为新模板` 或放弃未保存修改
- 当来源模板是系统模板时，前端不得提供覆盖系统模板的入口
- 当来源模板是用户模板时，前端必须允许用户选择覆盖原模板或另存为新模板
- 当来源模板是用户模板时，前端必须允许用户删除该模板
- 当模板处于脏状态时，前端不得直接删除该模板；必须先要求用户放弃未保存修改或先完成保存
- 当前会话若正在使用将被删除的用户模板，删除完成后前端必须立即刷新模板摘要与可选模板列表

### 13.2 Run 切换

当同一会话存在多个 `PipelineRun` 时：
- 前端必须允许用户通过 `Run Switcher` 在同一页面内跳转到不同 run 分段
- 默认进入会话时必须优先定位到当前活动 run，或在当前活动 run 可见的区域落点
- 历史 run 段落与当前活动 run 段落都必须保留在同一条会话主流中
- 若当前会话存在等待澄清或等待审批的活动 run，前端必须优先把该 run 暴露为默认焦点对象
- 若当前会话存在等待高风险工具确认的活动 run，前端必须优先把该 run 暴露为默认焦点对象，并滚动到对应工具确认块
- `Run Switcher` 只负责页面内导航；恢复当前 run 不是 `Run Switcher` 的职责

每个 run 分段必须满足以下规则：
- 必须以显式 run 分界头部开始
- 该头部至少展示：run 次序、状态、开始时间、结束时间或进行中状态、触发来源
- 不同 run 分段之间必须通过强分界线、间距、标题区或其他明确视觉结构断开
- 用户应能一眼分辨“上一轮运行已经结束，下一轮运行从此处开始”
- 当当前活动 run 的 Composer 主按钮因 `发送` 语义不可承载 `暂停` 时，run 分界头部或等价当前 run 工具区必须提供可触发的 `暂停` 入口
- 历史 run 的分界头部只承担信息展示与导航定位，不承载可执行的暂停、恢复或终止操作

当当前活动 run 处于 `failed` 或 `terminated` 时：
- 该 run 尾部终态条目必须提供 `重新尝试` 动作
- `重新尝试` 是针对当前会话创建新的 `PipelineRun` 的上下文化动作，不是恢复旧 run
- 触发前必须向用户明确说明：新 run 将从 `Requirement Analysis` 重新开始，并且不会继承旧 run 未交付的工作区改动
- 用户确认后，前端调用创建新 run 的后端命令，并在成功后把页面焦点移动到新 run 分段

### 13.3 会话进行中

会话进行中时：
- Narrative Feed 持续追加内容
- 用户主要以阅读为主
- 模板配置区不得继续占据中栏主工作区域
- 当前模板只允许以只读摘要形式存在，不提供编辑入口
- 当系统需要澄清时，用户通过 Composer 继续补充
- 当系统需要审批时，用户通过 Approval Request 完成审批
- 当系统需要高风险工具确认时，用户通过 Tool Confirmation 交互块完成允许或拒绝
- 当前模板只读摘要与当前活动 run 绑定，不得伪装成可继续编辑的共享配置
- `暂停`、`恢复` 与 `终止` 属于前端工作台的正式用户能力

### 13.4 会话完成

会话完成后：
- Narrative Feed 尾部必须出现交付结果内容
- 会话在左栏中切换为完成态
- 用户可以继续回看整条链路

### 13.5 会话回访

历史会话重新打开后：
- Narrative Feed 必须按原有时间顺序回放整个会话中的多 run 链路
- 已经发生的澄清、审批、高风险工具确认、回退和交付结果必须可见
- 右栏仍可查看历史结点详情
- 若该会话存在多个 run，前端必须允许跳转查看其他历史 run 分段
- 历史会话回访只用于回看、追溯和诊断，不得表现为新会话会自动继承历史会话、历史 run 或历史产物作为 Agent 长期记忆

## 14. 视觉层级与表现形式

### 14.1 主体风格

整体风格必须满足以下要求：
- 以阅读流为中心
- 减少后台式信息噪声
- 减少大面积仪表盘卡片布局
- 让用户始终聚焦当前会话的执行链路

### 14.2 条目表现

Narrative Feed 中的条目应遵循以下规则：
- 用户输入与系统输出必须有清晰区分
- 普通执行结点、审批请求、失败回退、最终交付必须有不同视觉语义
- 结点之间通过层级、间距、分隔与折叠关系区分，不依赖统一重卡片样式
- 可展开内容必须有明确展开入口与收起入口

### 14.3 审批块表现

审批块必须具备更强的视觉强调，以确保用户能快速识别当前需要做出决策。

审批块必须显式体现：
- 当前流程被人工决策阻塞
- 当前需要处理的对象类型
- 当前可执行动作

## 15. 响应式与布局收缩

桌面端保留三栏结构。

窄屏场景下：
- 左栏收缩为抽屉
- 右栏收缩为抽屉
- 中栏保持完整 Narrative Feed 与底部 Composer
- Narrative Feed 的阅读连续性不得被破坏

## 16. 可用性要求

前端必须满足以下可用性要求：
- 用户可以在不打开右栏的情况下理解主要执行链路
- 用户可以在任意需要深看的位置打开右栏查看详情
- 用户可以在会话中连续完成需求输入、澄清补充和审批拒绝补充
- 用户可以清楚区分系统是在 `暂停中`、`等待澄清`、`自动执行中`、`等待审批`、`已完成` 还是 `已终止`
- 用户可以清楚看见一次回退为何发生、回到了哪里、后续如何继续

## 17. 非目标

本设计不要求当前版本支持以下内容：
- 独立全局指标驾驶舱
- 独立审批中心替代主链路审批
- 多窗口协同编辑
- 功能二网页内浮层交互
- 复杂组织管理入口
- 面向运维人员的全量日志管理与审计检索控制台
- 本地日志文件查看器、日志全文搜索、错误聚合与 trace 可视化

## 18. 前端验收标准

前端工作台设计落地后，必须至少满足以下验收标准：

1. 支持项目加载入口，并能在左栏切换项目
2. 默认存在平台仓库自身项目
3. 每个项目下支持新建需求会话
4. 新建会话后的空白会话支持模板选择与允许字段编辑
5. 模板编辑只开放 `AgentRole` 选择、`system_prompt` 编辑、`AgentRole` 到 `Provider` 绑定、自动回归开关与最大重试次数，不开放 `AgentRole` 名称编辑，也不开放后端系统内置提示词资产或提示词版本切换
6. 前端将模板允许字段视为运行配置，并在 run 启动后冻结为只读快照
7. 系统启动时至少提供 `Bug 修复流程`、`新功能开发流程`、`重构流程` 三个可选系统模板
8. 新建会话默认预选 `新功能开发流程`
9. 前端只在模型选择区域显示已经配置且可用的 Provider，不把未配置的内置 Provider 当作默认可选项
10. 前端允许用户新增多个自定义 Provider，并在模板配置中按用户自定义名称选择
11. `OpenAI Completions compatible` 只作为自定义 Provider 的接入协议存在；`OpenAI Completions` 可以作为新增 Provider 的接入模板名称展示
12. 三个系统模板共享固定核心阶段骨架，其差异只体现在 `AgentRole`、`system_prompt`、`AgentRole` 到 `Provider` 绑定和自动回归默认策略
13. 前端不允许用户删除、禁用或重排功能一 V1 的核心业务阶段，也不允许关闭两个必需审批检查点
14. 系统模板只允许一键使用和修改后另存；用户模板修改后支持覆盖、另存或删除
15. 前端只允许删除 `user_template`，不允许删除 `system_template`
16. 删除当前选中的用户模板后，空白会话自动回退到默认系统模板 `新功能开发流程` 或其他仍可用模板
17. 中栏以 `Composer + Narrative Feed` 作为固定主形态
18. 空白会话中的模板区属于 `Narrative Feed` 的空态内容，不构成独立第三主区
19. 同一会话存在多个 run 时，前端提供独立 `Run Switcher`，用于在同一页面内定位不同 run 分段
20. 同一会话中的多个 run 在同一条会话主流中按时间顺序连续展示，且 run 与 run 之间存在明确分界
21. 需求澄清直接在 Narrative Feed 中完成，不使用审批卡
22. 不同执行结点使用异构内容展示，不采用统一摘要模板
23. 右侧 Inspector 可展示阶段结点、控制型条目和交付结果的完整输入、过程、输出、引用与量化信息，且以后台原始信息为准
24. 量化信息至少覆盖耗时与 Token
25. 人工审批以内联审批块方式出现在 Narrative Feed 中
26. `Reject` 必须提供补充输入能力
27. Composer 始终固定在会话页面底部，并服务当前活动 run
28. Composer 右端根据当前活动 run 状态在 `发送`、`暂停`、`恢复` 三种按钮形态间切换
29. `waiting_clarification` 仅表示当前活动 run 正在等待人工回复；此时 Composer 输入框保持激活，右端按钮为 `发送`
30. `Requirement Analysis` 结点内若当前活动 run 已恢复为 `running`，则表示系统正在继续分析或继续回复；此时右端按钮为 `暂停`
31. `waiting_clarification` 时，若 Composer 主按钮承担 `发送`，当前活动 run 仍存在可触发的次级 `暂停` 入口
32. `waiting_approval` 时，Composer 输入框不承担聊天输入，右端按钮保持 `暂停` 运行态语义
33. `waiting_tool_confirmation` 时，Composer 输入框不承担聊天输入，右端按钮保持 `暂停` 运行态语义，主操作只能在工具确认块内提交
34. 一旦进入 `solution_design` 及之后的正式研发链路，Composer 右端按钮在运行期持续保持 `暂停` 语义
35. 中栏右上角提供显眼的 `终止` 动作，并只作用于当前活动 run
36. 当前活动 run 处于 `failed`、`terminated` 或 `completed` 时，Composer 右端按钮保留但禁用；其中 `completed` 为只读展示态
37. 历史 run 中的审批块与工具确认块为只读历史态，不提供交互提交能力
38. 当前活动 run 处于 `failed` 或 `terminated` 时，Narrative Feed 尾部终态条目提供上下文化的 `重新尝试` 动作
39. 终止当前活动 run 后，Narrative Feed 尾部出现明确的终止提示条目，且该条目不替代既有审批块、工具确认块或澄清记录
40. 右栏默认关闭，点击结点、支持详情查看的控制型条目、工具确认块或交付结果后展开
41. 用户能够回看完整历史会话链路，并可按 run 分段导航
42. 用户能够在单一工作台中完成输入、澄清、审批、高风险工具确认和结果查看
43. 中栏结点正文与结点内条目的信息内容符合本规格定义
44. 右侧 Inspector 中的完整信息符合本规格定义，且不以二次摘要替代后端原始记录
45. 不适用的量化指标在前端隐藏，不使用统一空模板占位
46. 会话状态至少可区分 `执行中`、`暂停中`、`等待澄清`、`等待审批`、`等待工具确认`、`已完成`、`失败`、`已终止`
47. 当前项目摘要能够显示当前默认交付模式摘要
48. 前端在控制台右上角提供统一 `设置` 入口，并以左侧导航加右侧内容区的弹窗承载配置
49. 设置弹窗至少包含 `通用配置`、`模型提供商` 与 `导入导出` 三个子页面
50. 项目级 `DeliveryChannel` 配置只出现在设置弹窗的 `通用配置` 页面，不出现在模板配置区
51. 保存项目级默认 `DeliveryChannel` 配置至少影响后续新启动的 run；对尚未进入 `Delivery Integration` 的当前活动 run，可用于后续交付就绪校验
52. 设置弹窗中的 `模型提供商` 页面允许通过 `火山引擎`、`DeepSeek`、`OpenAI Completions` 三个接入模板新增多个 Provider，允许编辑 Provider 显示名称、启用状态、连接字段、API key、默认模型、模型列表和折叠 `高级设置` 中的 Provider 模型能力字段，并允许删除 Provider；内置 Provider 不允许修改 `provider_id`、`provider_source` 和协议归属
53. Provider 模型能力配置必须允许在 `高级设置` 中填写 `context_window_tokens`、`max_output_tokens`、`supports_tool_calling`、`supports_structured_output`、`supports_native_reasoning`，并明确这些字段不属于平台运行设置且都有后端默认值
54. 设置弹窗中的 `导入导出` 页面支持当前项目作用域下用户可见配置包的导出和导入，且不导出真实密钥、平台隐性运行设置、系统内置提示词正文、运行快照、历史 run 或审计正文
55. 设置弹窗不展示或编辑环境变量、平台运行数据目录、SQLite 文件路径、平台运行上限、日志策略、`compression_threshold_ratio`、后端系统内置提示词资产或 `deterministic test runtime`
56. Provider 配置变更只影响后续 run 或尚未启动 run 的模板选择；模板 Provider 下拉框只允许选择当前已配置且启用的 Provider，已启动 run 的 Provider 与模型绑定快照不被改变
57. Provider 调用失败重试、指数退避和熔断状态必须能在对应阶段结点或 Inspector 中被用户看见
58. 当 `Code Review` 相关审批块对应 `git_auto_delivery` 且当前项目交付配置状态不是 `ready` 时，前端在审批块内阻止 `Approve` 提交，并提供打开 `通用配置` 页面的入口
59. 高风险工具确认块不使用 Approval Request 交互，不计入两个人工审批检查点，且拒绝后不得展示为审批拒绝回归
60. 历史会话回访只用于回看、追溯和诊断，不得表现为新会话自动继承跨会话长期记忆

## 19. 结论

功能一前端工作台应被正式定义为：

`一个以项目上下文为基础、以需求会话为单位、以输入框和执行叙事瀑布流为中心、以按需详情侧栏和内联人工审批为辅助的 AI 交付工作台`

该工作台必须保证两件事同时成立：
- 它具备对话式产品的自然输入体验
- 它具备流程型系统的可解释执行链路

只有同时满足这两点，前端才既能让用户自然发起需求，也能让用户真正理解系统如何把需求推进到交付。
