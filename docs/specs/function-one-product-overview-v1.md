# 功能一产品总纲 V1

## 1. 文档目标

本文档用于定义 `AI 驱动的需求交付流程引擎` 的 `功能一` 产品总纲，作为功能一范围边界、业务阶段边界、运行时控制点边界和验收口径的正式依据。

本文档聚焦：
- 功能一的产品定位、版本目标与非目标
- 功能一的会话、模板、run 与业务阶段边界
- 需求澄清、人工审批、工具确认、暂停恢复、回退重试与交付完成之间的产品边界
- 功能一的产品级验收标准

本文档不展开：
- 后端引擎内部模块划分、执行图模型、领域对象与事件设计
- Agent 执行器、模型调用、工具调用与运行时持久化细节
- REST API、查询投影、实时推送协议与载荷结构
- 前端控制台页面布局、Narrative Feed、Inspector 与交互实现细节

## 2. 文档关系与口径裁决原则

当前功能一拆分规格的职责边界如下：

1. `docs/specs/function-one-product-overview-v1.md`
用于定义功能一的正式产品边界、业务阶段边界、运行边界与验收边界。

2. `docs/specs/frontend-workspace-global-design-v1.md`
用于定义前端控制台的正式交互口径，尤其是会话结构、Narrative Feed、Run Switcher、Composer、Inspector、需求澄清与人工审批的前端表现。

3. `docs/specs/function-one-backend-engine-design-v1.md`
用于定义功能一后端引擎、执行图模型、领域对象、事件模型、API、查询投影与前后端协作契约。

当文档之间出现交叉时，按以下原则裁决：
- 产品边界与业务阶段边界，以 `docs/specs/function-one-product-overview-v1.md` 为准
- 前端交互与展示语义，以 `docs/specs/frontend-workspace-global-design-v1.md` 为准
- 后端领域模型、执行图、API、投影与事件语义，以 `docs/specs/function-one-backend-engine-design-v1.md` 为准

`docs/archive/function-one-design-v2.md` 不再作为当前正式规格，只保留为历史迁移参考。

## 3. 产品定位

功能一被正式定义为：

`一个以项目为上下文、以需求会话为单位、由 AI 推动需求到代码交付闭环、并在关键质量关口保留人工介入的研发流程平台`

其核心价值不只是让模型生成代码，而是把以下能力组织成一条可解释、可回看、可纠偏、可交付的研发主链：
- 将自然语言需求转换为结构化需求与验收标准
- 基于代码库上下文形成技术方案并完成阶段内方案校验
- 基于已确认方案产出代码、测试与评审结果
- 在关键质量关口执行人工审批
- 在高风险工具动作前执行用户确认
- 在运行中支持暂停、恢复、终止、回退与重试
- 输出可追溯的最终交付结果

## 4. 版本目标

功能一第一阶段的版本目标如下：
- 支持在项目上下文中创建需求会话并发起一次完整执行
- 支持从需求输入到交付结果的完整链路运行
- 支持在会话启动前选择模板并编辑允许的运行配置
- 支持需求分析、方案设计、代码生成、测试生成执行、代码评审与交付集成六个正式业务阶段
- 支持在 `Solution Design` 阶段内部执行方案校验
- 支持 `Solution Design` 阶段输出可执行任务计划，覆盖任务拆分、目标文件或模块、执行顺序、验证动作、依赖假设与风险处理
- 支持需求澄清、人工审批、高风险工具确认、暂停、恢复、终止、回退与重新尝试
- 支持同一需求链路在同一会话下形成多个 `PipelineRun` 执行尝试并独立回看
- 支持真实代码库上下文读取、代码修改、测试执行与交付记录生成
- 支持受控 `bash` 命令按风险级别执行，并在安装依赖、联网下载、删除或移动文件、大范围生成或覆盖、数据库迁移等高风险动作前请求用户确认
- 支持 Provider 调用失败时执行指数退避重试、连续失败熔断、过程记录和前端可见状态展示
- 支持 `demo_delivery` 与 `git_auto_delivery` 两类交付模式
- 支持在项目级配置默认交付模式与交付通道，并在最终人工审批通过后、进入 `Delivery Integration` 前固化为交付快照
- 支持前端控制台通过统一设置弹窗完成项目级交付配置、模型提供商配置与用户可见配置的导入导出
- 支持前端控制台查看链路、补充澄清信息、执行人工审批与回看历史
- 支持已加载项目与历史会话在系统重启后继续可见，并支持在左栏重命名会话、删除会话和移除已加载项目
- 支持所有核心命令与查询通过 REST API 暴露，并通过 SSE 提供会话级实时增量更新，同时提供完整 OpenAPI 文档
- 为功能二保留可复用的后端扩展对象与接口边界

## 5. 非目标

以下内容不属于功能一第一阶段的必须范围：
- 浏览器注入、页面圈选与网页内悬浮操作
- 多租户、权限中心、组织管理与计费体系
- 独立审批中心、独立报表后台与复杂运营后台
- 任意自定义业务阶段编排器
- 语义索引驱动的大规模代码检索体系
- 云端多仓库托管与大规模分布式执行集群
- 面向用户的无限制通用平台配置中心、密钥管理系统或运行时切换面板
- 把本地可编辑启动配置文件作为用户长期配置真源
- 将测试运行能力作为正式执行模式暴露给用户
- 在移除项目时删除本地项目文件夹、目标仓库文件、远端仓库、远端分支、提交或代码评审请求

## 6. 用户与核心场景

功能一面向以下用户：
- `操作者`：发起需求、补充澄清信息、执行审批、确认高风险工具动作、控制运行状态、查看交付结果
- `观察者/管理者`：关注链路是否完整、风险是否暴露、审批是否清晰、交付是否可解释
- `平台开发者`：使用平台对自身仓库或外部本地仓库发起需求交付任务

功能一覆盖以下核心场景：
- `预设演示场景`：使用平台自身仓库完成稳定的端到端演示
- `临时需求场景`：面对小范围需求改动，完成分析、设计、实施、测试、评审与交付
- `平台自举场景`：目标仓库为平台自身，展示“用平台给平台加功能”

## 7. 核心产品原则

功能一必须遵循以下产品原则：

1. `流程优先于单轮对话`
功能一不是单次聊天生成代码工具，而是一条从需求到交付的显式研发流程。

2. `业务阶段显式化`
需求分析、方案设计、代码生成、测试生成执行、代码评审与交付集成必须作为正式业务阶段存在。

3. `运行时控制点独立于业务阶段`
澄清、审批、工具确认、暂停、恢复、终止、回退与重试属于运行时控制语义，不是额外业务阶段。

4. `阶段产物显式化`
每个业务阶段都必须产出可被后续阶段读取、展示和审查的结果，而不是只保留隐式文本。

5. `方案设计必须可执行`
`Solution Design` 不只输出高层方案，还必须输出可被 `Code Generation`、`Test Generation & Execution` 与 `Code Review` 读取的执行任务计划。该计划必须表达任务拆分、目标文件或模块、执行顺序、验证动作、依赖假设与风险处理，并作为方案审批对象的一部分。

6. `澄清属于需求分析内部`
需求澄清属于 `Requirement Analysis` 阶段内部的连续对话，不单独形成审批节点。

7. `人工审批只保留在质量关口`
功能一第一阶段的人工审批只保留在 `Solution Design` 与 `Code Review` 两个质量关口。

8. `审批嵌入主链路`
人工审批是执行链路中的显式控制点，不通过独立审批系统脱离主链路运行。

9. `工具确认不同于人工审批`
高风险工具确认是运行时权限控制点，确认对象是具体工具动作，不是阶段质量产物。用户拒绝高风险工具动作后，系统只能在存在低风险替代路径时继续当前阶段；不存在替代路径时，当前阶段和当前 run 必须进入失败或等待用户显式暂停、终止，不得把工具拒绝解释为审批拒绝后的自动回归。

10. `交付必须可追溯`
最终结果必须能够回溯到需求、方案、代码、测试和评审结论。

11. `平台运行必须可审计`
系统必须记录可用于运行排障、审计复盘、安全追踪与后续运维分析的平台级日志与审计链路。日志审计服务于平台运行治理，不替代 Narrative Feed、Inspector 或领域事件。

12. `运行前配置服务执行链路`
会话启动前允许编辑的模板、`AgentRole`、`Provider` 与自动回归配置，必须用于定义后续阶段 Agent 的实际执行行为；运行启动后，这些配置必须固化为本次执行快照。

13. `交付配置归属项目上下文`
项目级默认 `DeliveryChannel` 用于定义该项目后续新启动 run 的默认交付方式；该配置不属于模板编辑范围，且当前活动 run 的交付快照只在最终人工审批通过后、进入 `Delivery Integration` 前固化。

14. `项目与会话历史可持久、可管理`
已加载且未移除的项目、项目下未删除的会话以及这些会话的历史链路属于控制面历史。除用户显式删除会话或移除项目外，系统重启后必须继续保留这些历史并允许从前端左栏访问。会话重命名只改变会话展示名，不改变执行历史、审批记录、产物或 run 归属；会话删除和项目移除属于显式历史管理动作，不等同于 `terminate`、`rollback` 或 `retry`。会话删除和项目移除的产品层结果是从常规项目/会话历史、回看入口和产品查询投影中删除；跨库存储的物理删除、软删除或归档策略由后端规格统一固定，并必须保留审计边界要求保留的安全审计事实。

15. `会话历史不等于长期记忆`
功能一 V1 不提供跨会话长期记忆。历史会话、历史 run、历史产物和审计记录用于回看、追溯和诊断，不自动进入新会话的 Agent 上下文；每个新 `Session` 必须以当前项目上下文、用户输入、所选模板和本次 run 快照为执行依据。同一 `Session` 下多个 `PipelineRun` 只表示同一需求链路的不同执行尝试，不构成跨会话记忆同步。

16. `配置真源与运行快照分层`
环境变量只服务系统启动、本地运行、前后端连接、工作区隔离、日志审计落点和凭据引用解析。用户可见配置的真源是后端控制面数据库与对应 API，前端设置弹窗、模板编辑能力和配置包导入都必须通过同一套 API 校验、审计和版本化规则写入正式配置对象。平台隐性运行设置存储在后端控制面数据库中，但不作为普通用户可见配置。会影响执行语义的模板、Provider、模型绑定、运行上限和执行图配置必须在 run 启动时固化为快照；交付通道配置必须在最终人工审批通过后、进入 `Delivery Integration` 前固化为快照。

17. `正式运行与测试运行能力分离`
功能一的正式 Agent 编排路径由 LangGraph runtime 承担。`deterministic test runtime` 只服务测试、前端联调、端到端验证和回归验证，不属于用户可选择的运行模式，不进入前端设置，不作为平台环境变量暴露。

18. `API 契约与文档同等重要`
所有核心命令与查询必须通过 REST API 暴露，会话级实时增量更新必须通过 SSE 提供，并提供与运行接口一致的 OpenAPI 文档。

19. `受控工具最小面`
Agent 运行时必须通过受控工具完成仓库读取、文件编辑、路径匹配、内容搜索、命令执行与变更追踪。功能一 V1 的工作区工具设计参考 Claude Code 的工具工作机制，但正式工具契约名使用小写机器可读 ID，并固定为 `bash`、`read_file`、`edit_file`、`write_file`、`glob`、`grep`。diff 生成、工作区状态采集和 `ChangeSet` 构建属于后端服务侧能力，不作为独立模型可调用工具；`bash` 是工具契约名，不要求底层必须是 Unix bash；`grep` 必须具备受控的本地 `ripgrep` / `rg` 可用性校验。工具不得绕过阶段权限、工作区边界、审计记录和变更集构建。

20. `Provider 失败必须可解释`
Provider 调用失败必须按统一策略处理。可恢复的超时、网络错误和限流错误必须执行指数退避重试；连续失败达到熔断条件后，当前阶段必须形成可展示的失败原因、重试轨迹和熔断状态。Provider 配置由统一配置管理维护，已启动 run 继续使用自身固化的 Provider 与模型绑定快照，不因运行外 Provider 配置变化改变当前执行语义。

21. `为功能二预留复用边界`
功能一在后端对象与接口层必须为 `ChangeSet`、`ContextReference`、`PreviewTarget`、`DeliveryRecord` 等后续能力保留扩展空间。

## 8. 会话、模板与运行边界

功能一的产品运行边界如下：

1. 所有执行都必须挂载在一个 `Project` 之下，且系统启动后默认存在一个绑定平台仓库自身路径的项目。
2. 已加载且未移除的 `Project` 必须在系统重启后继续保留在项目历史中，用户不需要重新导入。
3. 每个 `Project` 都必须具备一个项目级默认 `DeliveryChannel`；前端必须能够在统一设置弹窗中查看其状态并完成配置。
4. `DeliveryChannel` 属于项目级交付配置，不属于模板编辑范围；当前活动 run 的交付快照只在最终人工审批通过后、进入 `Delivery Integration` 前固化。
5. 用户允许移除已加载的非默认 `Project`；移除后该项目必须从项目历史中消失，并一并从常规产品查询与回看入口中删除该项目下用于产品回看的 `Session` 记录和运行历史。
6. 移除 `Project` 不得删除本地项目文件夹、目标仓库文件、远端仓库、远端分支、提交或代码评审请求；平台安全审计记录按审计边界保留，不作为可继续打开的项目或会话历史。
7. 存在活动 run 的 `Project` 不允许被移除；必须先让相关活动 run 进入终态。
8. 每次需求推进都必须归属到一个 `Session`。
9. 每个 `Session` 承载同一个需求链路，不承载多个彼此独立的需求。
10. 新建会话时必须绑定一个 `PipelineTemplate`。
11. 同一 `Session` 的首条需求消息必须自动启动首个 `PipelineRun`。
12. 同一 `Session` 可因重新尝试或运维重启产生多个 `PipelineRun`，但这些运行只表示同一需求链路的不同执行尝试；暂停后的恢复属于原 run 的继续执行，不创建新的 `PipelineRun`。
13. 新的 `PipelineRun` 只允许在前一个活动 run 处于 `failed` 或 `terminated` 后创建；`completed` 表示该会话链路已经完成，不再在同一会话中开启新的 run。
14. 新的 `PipelineRun` 必须从完整链路起点重新开始，而不是从上一个 run 的中间业务阶段断点续跑。
15. 不同 `PipelineRun` 之间不得共享未交付的工作区改动；新 run 必须基于干净基线启动。
16. 新建 `Session` 不自动读取其他会话的历史 run、历史产物、历史审批或历史工具过程作为 Agent 记忆；历史会话只通过产品回看入口和审计诊断能力可见。
17. 同一 `Session` 的多个 `PipelineRun` 必须在同一会话页面中按时间顺序串联展示，而不是拆成多个彼此隔离的会话页面。
18. 不同 `PipelineRun` 之间必须有明确分界，不得在视觉或叙事上伪装成同一次连续运行。
19. 当前活动 run 负责驱动页面底部统一 `Composer` 与生命周期控制；历史 run 只用于回看，不承载新的输入或审批操作。
20. 用户允许在左栏重命名 `Session`；重命名只改变会话展示名，不改变原始需求、运行历史、审批记录、产物、交付记录、事件归属或 `PipelineRun` 归属。
21. 用户允许在左栏删除没有活动 run 的 `Session`；删除后该会话不得继续出现在会话历史中，也不得继续作为可回看的历史会话打开。
22. 存在活动 run 的 `Session` 不允许被删除；删除会话不得替代 `terminate`，也不得隐式暂停、恢复、回退或重试当前 run。
23. 用户在同一会话中完成：
- 原始需求输入
- 需求澄清补充
- 方案审批
- 代码评审审批
- 高风险工具确认
- 暂停、恢复、终止
- 交付结果查看

补充约束如下：
- 在首条需求消息提交前，`Session` 必须保持 `draft`，且 `current_run_id = null`；系统不存在“等待首条需求输入的活动 run”状态。
- 本节所称活动 run 指尚未进入 `completed`、`failed` 或 `terminated` 的 `PipelineRun`；`draft` 会话没有活动 run。
- `completed` run 以顶层 `delivery_result` 条目作为尾部收束；`failed` 与 `terminated` run 必须在各自 run 尾部追加顶层 `system_status` 终态条目。
- `重新尝试` 只挂载在当前活动 run 的 `failed` 或 `terminated` 尾部 `system_status` 终态条目上，不挂载在历史 run 或 `completed` run 上。
- 会话删除与项目移除只能删除产品历史可见性、常规查询入口和对应查询投影，不得删除平台审计边界要求保留的安全审计事实。

### 8.1 配置真源、导入导出与运行快照边界

功能一必须区分 `启动环境设置`、`用户可见配置`、`平台隐性运行设置`、`配置导入导出包`、`系统内置提示词资产` 与 `运行快照`。这些对象的入口、存储真源和生效时机必须固定，避免配置文件、前端设置、数据库记录和运行快照之间形成并行真源。

`启动环境设置` 只允许覆盖系统启动前必须确定的基础落点、连接信息和密钥引用解析规则。它可以由环境变量、部署命令参数或等价启动机制加载，但不属于用户业务配置。允许表达的内容至少包括：
- 前端访问后端服务所需的 API base URL
- 本地前后端分端运行所需的 CORS 来源
- 平台运行数据根目录或由系统默认派生的数据落点
- 默认 Project 根路径
- 隔离工作区根路径
- 平台级日志与审计记录的本地落点
- `credential_ref` 与 `api_key_ref` 指向平台密钥来源的引用解析规则

`用户可见配置` 的运行时真源是后端控制面数据库与对应 API。前端统一设置弹窗、模板编辑能力和配置包导入都只是写入这些正式配置对象的入口，必须复用同一套后端校验、审计、版本化和投影规则。功能一 V1 的用户可见配置只包括前端已提供编辑入口且后端正式消费的配置字段，至少包括：
- 项目级 `DeliveryChannel`，用于定义项目默认交付模式、交付通道、仓库目标、代码评审请求类型、凭据引用和就绪状态
- `LLMProvider` 与模型能力配置，包括内置 Provider 和用户新增 `custom Provider` 的连接字段、模型标识、凭据引用，以及 `context_window_tokens`、`max_output_tokens`、`supports_tool_calling`、`supports_structured_output`、`supports_native_reasoning`
- 模板运行配置，包括固定阶段骨架上的 `AgentRole` 绑定、用户可编辑 `system_prompt`、Provider 绑定、自动回归开关和最大自动回归重试次数

Provider 配置必须满足以下产品边界：
- 内置 Provider 和 `custom Provider` 都必须在正式 Provider 配置对象中具备运行所需连接字段、默认模型或模型列表、凭据引用和能力声明
- 内置 Provider 的默认配置必须由后端初始化种子或数据库迁移写入控制面数据库；初始化后，运行时不得从硬编码常量、环境变量或本地配置文件直接读取其业务字段来绕过正式配置对象
- `custom Provider` 由前端设置弹窗、Provider API 或配置包导入创建和更新
- Provider 密钥只能以 `api_key_ref` 或等价引用保存；真实密钥值只能由后端按启动环境设置中的解析规则读取，不能进入前端、配置导出包、日志、投影或审计正文
- `context_window_tokens`、`max_output_tokens`、`supports_tool_calling`、`supports_structured_output`、`supports_native_reasoning` 是 Provider 模型能力配置，必须按模型粒度保存并随 Provider 与模型绑定一起固化到运行快照。`context_window_tokens` 表示指定模型的最大上下文窗口 token 数，缺省值为 `128000`，用于上下文压缩阈值判断。`max_output_tokens` 用于模型输出预算和请求参数上限。`supports_tool_calling` 用于模板保存、run 启动和工具绑定前的能力校验。`supports_structured_output` 用于结构化输出与修复路径选择。`supports_native_reasoning` 用于记录是否期望接收 Provider 原生推理内容；系统不得在该能力为 false 时伪造原生推理。

`平台隐性运行设置` 存储在后端控制面数据库中，由后端管理能力维护，不进入普通前端设置界面，也不作为用户配置包的常规内容。它用于控制 Agent 运行上限、Provider 调用策略、上下文裁剪限制、日志裁剪限制、日志保留策略、日志查询分页上限和上下文压缩阈值比例。`compression_threshold_ratio` 属于平台隐性运行设置，缺省值为 `0.8`，必须大于 `0` 且小于 `1`；它与当前模型绑定快照中的 `context_window_tokens` 共同决定上下文压缩触发阈值。

`配置导入导出包` 只服务用户可见配置的备份、迁移和环境复制，不是系统启动配置文件，也不是模型调用、上下文构建、投影查询或历史回放的运行时读取来源。配置包必须满足以下约束：
- 导入必须经过与前端保存相同的后端 API 校验、权限边界、审计记录、版本化和错误返回语义
- 导出必须只包含可迁移的用户可见配置和引用信息，不包含真实密钥值、系统内置提示词正文、运行快照、历史 run、日志、审计正文或平台内部数据库路径
- Provider 模型能力在 V1 配置包中允许包含 `context_window_tokens`、`max_output_tokens`、`supports_tool_calling`、`supports_structured_output`、`supports_native_reasoning`；这些字段在前端 Provider 高级设置中折叠展示，均有默认值，不作为用户必须配置项
- 导入 Provider、DeliveryChannel 或模板运行配置只影响后续新建 Session、后续新启动 run 或尚未启动 run 的模板选择；不得回写已启动 run 或历史 run 的快照
- V1 配置包不包含 `compression_threshold_ratio` 等平台隐性运行设置；任何包含平台隐性运行设置的诊断或管理包都不属于功能一 V1 用户可见配置导入导出范围

运行快照是执行语义的最终依据。会影响执行语义的模板、Provider、模型绑定、运行上限和执行图配置必须在 run 启动时固化为模板快照、Provider 与模型绑定快照、`RuntimeLimitSnapshot` 和 `GraphDefinition`；已启动 run 不读取最新配置来改变当前语义。交付通道配置必须在最终人工审批通过后、进入 `Delivery Integration` 前固化为交付通道快照；已固化交付通道快照不读取最新项目配置来改变当前交付语义。

系统内置提示词资产属于后端版本化运行资产，不属于启动环境设置、用户可见配置、平台隐性运行设置、配置导入导出包、前端设置、模板编辑字段或用户业务配置。用户可编辑的 `system_prompt` 只属于模板运行配置，不能升级为系统可信提示词资产。

所有配置边界必须共同满足以下约束：
- 后端规格中定义的多 SQLite 职责库属于系统内部持久化边界，其文件路径由系统默认从平台运行数据根目录派生；不得要求用户或前端设置逐个配置这些数据库路径
- 测试环境需要替换数据库或目录落点时，必须通过测试 fixture、settings override 或等价测试机制完成，不进入正式产品配置面
- 启动环境设置不得承载 Provider 的模型参数、Provider 模型能力字段、`system_prompt`、系统内置提示词正文或版本切换、模板角色绑定、项目交付仓库、目标分支、代码评审请求类型、交付模式、Agent 运行上限、日志策略或 `compression_threshold_ratio`
- 平台隐性运行设置可以热重载，但不得改变已经启动 run 的执行语义；诊断类设置可以即时生效，但不得改变领域事件、查询投影、审批状态、交付结果或 Narrative Feed 条目语义
- Provider 调用策略必须覆盖请求超时、网络错误重试、限流重试、指数退避上限、连续失败熔断阈值和熔断恢复条件；这些策略必须在 run 启动时固化为运行上限或 Provider 调用策略快照的一部分
- 所有用户或管理接口可写入的运行上限必须受平台硬上限约束，超过硬上限时必须拒绝保存或拒绝启动 run

### 8.2 正式运行与 deterministic test runtime 边界

功能一对用户只呈现一条正式研发主链和一套业务阶段语义。正式运行必须使用真实 Agent 编排路径，后端执行内核的正式实现由 LangGraph runtime 承担。

`deterministic test runtime` 是工程验证能力，必须满足以下约束：
- 只由测试夹具、前端联调入口、端到端测试或回归测试调用
- 只用于稳定验证状态机、投影、SSE、人工审批、工作区边界、交付记录和前端流程
- 不作为用户可选择的运行模式
- 不出现在前端设置、项目设置、模板配置或平台环境变量中
- 不改变正式业务阶段、运行时控制点、领域事件、查询投影、Inspector 或 DeliveryRecord 的语义
- 与正式 LangGraph runtime 共享后端定义的稳定执行契约，但不得把内部测试实现细节泄漏到产品 API 或前端展示
- 可以随工程演进被重构或替换，但在 V1 中作为稳定回归验证能力保留

## 9. 功能一业务阶段边界

功能一第一阶段包含以下正式业务阶段：

1. `Requirement Analysis`
2. `Solution Design`
3. `Code Generation`
4. `Test Generation & Execution`
5. `Code Review`
6. `Delivery Integration`

以上六个阶段共同构成功能一第一阶段的正式研发主链。

### 9.1 Requirement Analysis

职责：
- 读取用户原始需求与必要上下文
- 产出结构化需求、验收标准、约束与待确认事项
- 当信息不足时，在同一阶段内部发起澄清提问

边界：
- 需求澄清属于本阶段内部行为
- 本阶段不创建 `Approve / Reject` 审批动作
- 本阶段完成后，进入 `Solution Design`

### 9.2 Solution Design

职责：
- 基于结构化需求、澄清结论与代码库上下文形成技术方案
- 给出影响范围、关键设计决策、文件范围、实现路径与可执行任务计划
- 在本阶段内部完成方案校验

边界：
- 方案设计结果必须先经过本阶段内部的方案校验子步骤
- 可执行任务计划属于 `Solution Design` 的正式产物，不形成独立业务阶段；后续代码生成、测试生成执行和代码评审必须读取该计划及其稳定引用
- 方案设计本身不是人工审批动作，但其结果必须进入方案审批检查点

### 9.3 Solution Design 内部方案校验子步骤

职责：
- 独立校验技术方案是否偏离需求、遗漏影响范围、缺少测试策略或引入过高风险

边界：
- 该校验子步骤属于 `Solution Design` 阶段内部，不形成独立正式业务阶段
- 校验通过后，进入方案人工审批
- 校验不通过时，返回 `Solution Design` 内部继续修正

### 9.4 Code Generation

职责：
- 按已通过校验且经人工批准的方案生成或修改代码
- 输出变更说明与代码变更集合

边界：
- 本阶段不触发人工审批
- 本阶段完成后进入测试生成与执行

### 9.5 Test Generation & Execution

职责：
- 生成测试或补充测试
- 执行测试并输出测试结果与测试缺口分析
- 读取项目说明、依赖声明、脚本配置和已批准方案中的测试策略，识别测试运行环境与依赖缺失

边界：
- 本阶段不触发人工审批
- 本阶段执行测试、构建、格式化或依赖相关命令时必须通过受控 `bash` 工具；当命令属于高风险工具动作时，必须先进入运行时工具确认
- 本阶段完成后进入 `Code Review`

### 9.6 Code Review

职责：
- 审查代码变更、测试结果、测试缺口与方案一致性
- 输出评审报告、问题列表、回退目标判断与风险判断

边界：
- 评审阶段允许先进行自动回归与返修循环
- 自动回归结束后，评审结果进入代码评审人工审批

### 9.7 Delivery Integration

职责：
- 整理已通过审批的变更、测试结论、评审结论与交付信息
- 读取本次 run 已固化的项目级交付快照
- 执行交付步骤并沉淀交付过程记录
- 根据交付模式生成对应的交付结果
- 生成最终交付记录

边界：
- `Delivery Integration` 不属于人工审批阶段
- `Delivery Integration` 表达的是交付执行过程；本阶段成功完成后，系统再以顶层 `delivery_result` 条目输出最终交付结果
- `delivery_result` 不替代 `Delivery Integration` 阶段
- 本阶段必须支持 `demo_delivery` 与 `git_auto_delivery` 两类交付模式
- `demo_delivery` 只生成展示型交付结果，不执行真实 Git 写操作
- `git_auto_delivery` 执行真实分支、提交与 MR/PR 交付动作
- 交付配置与凭据的阻塞性检查必须在进入本阶段前完成；本阶段只负责读取已固化的交付快照并输出结果展示
- 若本阶段失败，不生成 `delivery_result`，并由 run 尾部 `system_status` 终态条目表达失败结果
- 本阶段完成后，会话进入交付完成状态

## 10. 运行时控制点与人工介入边界

功能一必须严格区分 `业务阶段` 与 `运行时控制点`。

运行时控制点用于表达主链执行中的中断、等待、人工介入与重新推进，不替代正式业务阶段。

### 10.1 需求澄清边界

需求澄清必须满足以下规则：
- 只发生在 `Requirement Analysis` 阶段内部
- 表现为系统提问与用户补充信息的连续对话
- 使用会话统一输入入口，不使用 `Approve / Reject`
- 只有在明确等待用户补充时，当前 run 才进入 `waiting_clarification`
- 用户补充提交后，当前 run 必须恢复执行并继续同一个 `Requirement Analysis` 结点分析；若仍需继续澄清，可在同一结点内再次提出问题
- 不创建独立审批页面，不创建独立审批卡，不创建独立审批工作流

### 10.2 人工审批边界

功能一第一阶段只保留两个正式人工审批检查点：

1. `方案设计审批`
- 位于 `Solution Design` 阶段内部方案校验通过之后
- 审批对象为方案说明、影响范围、关键设计决策与相关风险
- `Reject` 后回到 `Solution Design`

2. `代码评审审批`
- 位于 `Code Review` 产出稳定评审结果之后
- 审批对象为代码变更内容、测试结果、测试缺口分析与评审结论
- 当当前项目 `delivery_mode = git_auto_delivery` 时，系统必须在该审批阶段检查交付配置与凭据是否就绪；未就绪时阻塞审批通过并提示用户完成配置
- 当当前项目 `delivery_mode = demo_delivery` 时，不因交付配置或凭据缺失额外阻塞该审批
- `Reject` 后默认回到 `Code Generation`

除以上两处外，功能一不再定义其他人工审批检查点。

### 10.3 高风险工具确认边界

高风险工具确认必须满足以下规则：
- 高风险工具确认属于运行时权限控制点，不是人工审批检查点，也不是正式业务阶段
- 确认对象是具体工具动作、命令、目标路径、预期副作用和风险说明，不是阶段产物质量
- 高风险工具动作至少包括安装或升级依赖、联网下载、删除或移动文件、大范围生成或覆盖文件、数据库迁移、修改锁文件、修改环境配置、执行不在项目说明或脚本配置中的未知命令
- `Test Generation & Execution` 阶段允许因测试执行、环境探测或依赖缺失触发高风险工具确认；其他具备 `bash` 或副作用工具的阶段也必须遵守同一边界
- 用户同意后，系统只能执行本次确认覆盖的工具动作，并必须记录确认结果、命令、目标、风险说明和实际副作用
- 用户拒绝后，系统可以寻找低风险替代路径继续当前阶段；不存在替代路径时，当前阶段和当前 run 必须进入 `failed`，或由用户显式选择 `pause` 或 `terminate`
- 工具确认拒绝不得被解释为人工审批 `Reject`，不得触发 `Solution Design` 或 `Code Generation` 的审批回退语义
- 工具确认结果必须进入 Narrative Feed，并能通过 Inspector 或等价查询投影查看完整详情，同时保留审计记录

### 10.4 暂停、恢复与终止边界

运行控制必须满足以下规则：
- `pause`、`resume` 与 `terminate` 属于主链运行控制动作，不是新的业务阶段
- `pause` 可发生在运行中的正式业务阶段、等待澄清状态、等待审批状态或等待高风险工具确认状态
- 若 `pause` 发生在 `waiting_approval`，该审批必须转为不可提交态；用户只有在 `resume` 之后，才能继续处理同一个待审批检查点
- 若 `pause` 发生在等待高风险工具确认状态，该确认必须转为不可提交态；用户只有在 `resume` 之后，才能继续处理同一个待确认工具动作
- `resume` 只允许继续同一个 `PipelineRun`
- `terminate` 只终止当前活动 run，不删除已产生的历史记录、审批记录与产物记录

### 10.5 回退与重新尝试边界

运行控制必须满足以下规则：
- `rollback` 属于当前活动 run 内的控制动作，用于在保留既有执行历史、产物、问题记录与审批记录的前提下，把链路退回到既定业务阶段并开始订正性重执行
- `retry` 在当前 run 内用于自动回归或阶段内再次尝试
- `重新尝试` 是在当前活动 run 已 `failed` 或 `terminated` 后创建新的 `PipelineRun`
- 回退与重试不构成新的正式业务阶段

- `approval_request`
- `tool_confirmation`
- `control_item`
- `approval_result`
- `delivery_result`
- `system_status`
- `delivery_record`

这些产物必须满足以下要求：
- 可被后续阶段读取
- 可被前端控制台展示
- 可被人工审批、工具确认或后续阶段按需引用
- 可在历史会话中回看

## 12. 功能一验收标准

功能一第一阶段至少满足以下产品级验收标准：

1. 支持在项目上下文中创建需求会话并启动一次完整执行。
2. 已加载且未移除的项目和未删除的历史会话在系统重启后仍可从前端左栏继续访问，不需要重新导入。
3. 用户能够从左栏重命名会话展示名，重命名后历史链路、审批记录、产物和 run 归属保持不变。
4. 用户能够从左栏删除没有活动 run 的会话；删除后该会话不再出现在历史会话列表中，也不能再作为历史会话打开。
5. 用户能够移除已加载的非默认项目；移除时该项目下用于产品回看的会话历史必须从常规产品查询与回看入口中消失，且不删除本地项目文件夹、目标仓库文件、远端仓库、远端分支、提交或代码评审请求。
6. 支持从需求输入到交付结果的完整链路运行。
7. 支持在会话启动前选择模板并编辑允许的运行配置；V1 允许字段固定为角色槽位绑定、`system_prompt`、`Provider` 绑定与自动回归配置，不包含 `AgentRole` 名称修改；运行启动后配置固化为快照。
8. 同一需求链路支持在同一会话下形成多个 `PipelineRun` 执行尝试，并通过独立 `run` 切换机制查看。
9. 需求澄清直接发生在 `Requirement Analysis` 阶段内部，不使用审批动作。
10. 人工审批只保留在 `Solution Design` 与 `Code Review` 两个检查点。
11. 人工审批必须支持 `Approve` 与 `Reject`，并要求拒绝理由进入后续上下文。
12. `Solution Design` 必须输出可执行任务计划，并让代码生成、测试生成执行和代码评审能够读取该计划及其稳定引用。
13. 用户可以在单一控制台中完成输入、澄清补充、审批、高风险工具确认、暂停恢复、终止与结果查看。
14. 阶段产物与控制产物可被查询、展示、回看与追溯。
15. 最终交付结果能够关联需求、方案、任务计划、代码、测试与评审结论。
16. 系统支持 `demo_delivery` 与 `git_auto_delivery` 两类交付模式，并能对平台自身仓库完成至少一次真实端到端演示。
17. 支持在项目级配置默认 `DeliveryChannel`，并在最终人工审批通过后、进入 `Delivery Integration` 前固化为交付快照；正式快照字段与时间字段命名以后端规格为准，必须覆盖 `delivery_mode`，并在 `delivery_mode = git_auto_delivery` 时覆盖托管平台类型、仓库标识、默认分支、代码评审请求类型、`credential_ref`、`credential_status` 与配置就绪状态。
18. 所有核心命令与查询通过 REST API 暴露，会话级实时增量更新通过 SSE 提供，并提供完整 OpenAPI 文档。
19. 系统具备平台级日志审计基础能力，能通过本地日志文件、轻量索引、审计记录与关联标识支撑运行排障、审计复盘、安全追踪与后续运维分析，且不把日志审计替代为 Narrative Feed、Inspector 或领域事件。
20. 系统具备基于 `ToolProtocol` 的受控工作区工具最小面，正式工具契约名为 `bash`、`read_file`、`edit_file`、`write_file`、`glob`、`grep`，并能通过审计、工作区隔离和 `ChangeSet` 构建记录工具副作用；受控 `bash` 命令必须按风险级别执行，高风险工具动作必须先获得用户确认。
21. 系统具备 Provider 统一管理、调用失败指数退避重试、连续失败熔断、过程记录和前端可见状态展示能力，且不通过运行外 Provider 变更改变已启动 run 的执行语义。
22. 系统具备启动环境设置、用户可见配置、平台隐性运行设置、配置导入导出包、系统内置提示词资产与运行快照的分层边界，能够支撑本地启动、前后端连接、数据落点、工作区隔离、日志审计落点、凭据引用解析、Provider 管理、交付配置管理、模板运行配置、配置备份迁移、运行上限管理、提示词资产版本留痕和日志策略管理，且不把业务配置、平台隐性运行设置或系统内置提示词正文放入启动环境矩阵。
23. 新建会话之间不存在自动同步的长期 Agent 记忆；历史会话只作为回看、追溯和诊断对象，不自动进入新会话上下文。
24. 正式运行使用 LangGraph runtime；`deterministic test runtime` 只用于测试、联调、端到端验证和回归验证，不作为用户可选择的运行模式或平台配置项。
25. 功能二所需的核心扩展对象在后端规格层具备明确预留边界。

## 13. 风险与约束

功能一第一阶段必须控制以下风险：
- 不得把需求澄清重新建模为人工审批
- 不得把高风险工具确认重新建模为人工审批检查点
- 不得把运行时控制点误写为正式业务阶段
- 不得把功能一退化为单轮聊天生成工具
- 不得把前端主链路拆成多个互相跳转的审批或澄清页面
- 不得让后续阶段绕过结构化需求、澄清结论与方案结论直接推进
- 不得让后续阶段绕过方案设计中的可执行任务计划直接推进
- 不得让模型绕过 `ToolProtocol`、阶段 `allowed_tools`、工作区边界或审计策略直接读取文件、编辑文件、搜索内容、执行命令或构建变更集
- 不得让模型通过提示词、项目文档、脚本说明或工具观察绕过命令风险分级和高风险工具确认
- 不得在交付阶段重新引入新的人工审批关口
- 不得把历史会话、历史 run 或历史产物作为新会话的隐式长期记忆来源
- 不得把日志审计能力弱化为前端展示日志，或把日志记录反向作为产品状态、阶段产物与交付结论的真源
- 不得在功能一 V1 中把完整日志检索台、日志全文搜索或运维可视化面板作为主线交付前提
- 不得把启动环境设置或本地配置文件扩展为业务配置中心，不得把 Provider、Provider 模型能力字段、DeliveryChannel、模板运行配置、Agent 运行上限、日志策略或 `compression_threshold_ratio` 下沉为启动环境字段
- 不得把配置导入导出包作为运行时真源；模型调用、上下文构建、投影查询和历史回放都必须读取正式配置对象或运行快照
- 不得让用户可见配置或平台隐性运行设置变更改变已经启动 run 的模板快照、Provider 与模型绑定快照、运行上限快照或交付通道快照
- 不得要求用户或前端设置逐个配置系统内部多 SQLite 职责库路径
- 不得把 `deterministic test runtime` 暴露为用户可选运行模式、前端设置项或平台环境变量
- 不得为长期扩展性引入与当前版本价值不匹配的复杂基础设施

## 14. 结论

功能一第一阶段应被正式理解为：

`一个在项目上下文中运行、以需求会话为载体、由 AI 推动从需求分析到交付集成、并在方案设计与代码评审两个质量关口保留人工介入的研发流程平台`

功能一的产品边界已经明确为：
- 六个正式业务阶段构成固定研发主链
- 需求澄清属于 `Requirement Analysis` 内部连续对话
- 人工审批只保留在 `Solution Design` 与 `Code Review`
- 高风险工具确认属于运行时权限控制点，不属于人工审批检查点
- 暂停、恢复、终止、回退与重试属于运行时控制点
- 交付集成负责输出结果，不承担额外审批职责
