# 功能一产品总纲 V1

## 1. 文档目标

本文档用于定义 `AI 驱动的需求交付流程引擎` 的 `功能一` 产品总纲，作为功能一范围边界、业务阶段边界、运行时控制点边界和验收口径的正式依据。

本文档聚焦：
- 功能一的产品定位、版本目标与非目标
- 功能一的会话、模板、run 与业务阶段边界
- 需求澄清、人工审批、暂停恢复、回退重试与交付完成之间的产品边界
- 功能一的产品级验收标准

本文档不展开：
- 后端引擎内部模块划分、执行图模型、领域对象与事件设计
- Agent 执行器、模型调用、工具调用与运行时持久化细节
- REST API、查询投影、实时推送协议与载荷结构
- 前端控制台页面布局、Narrative Feed、Inspector 与交互实现细节

## 2. 文档关系与口径裁决原则

当前功能一拆分规格的职责边界如下：

1. `docs/specs/function-one-product-overview-v1.md`
用于定义功能一的正式产品边界、业务阶段边界、运行边界与验收边界。

2. `docs/specs/frontend-workspace-global-design-v1.md`
用于定义前端控制台的正式交互口径，尤其是会话结构、Narrative Feed、Run Switcher、Composer、Inspector、需求澄清与人工审批的前端表现。

3. `docs/specs/function-one-backend-engine-design-v1.md`
用于定义功能一后端引擎、执行图模型、领域对象、事件模型、API、查询投影与前后端协作契约。

当文档之间出现交叉时，按以下原则裁决：
- 产品边界与业务阶段边界，以 `docs/specs/function-one-product-overview-v1.md` 为准
- 前端交互与展示语义，以 `docs/specs/frontend-workspace-global-design-v1.md` 为准
- 后端领域模型、执行图、API、投影与事件语义，以 `docs/specs/function-one-backend-engine-design-v1.md` 为准

`docs/archive/function-one-design-v2.md` 不再作为当前正式规格，只保留为历史迁移参考。

## 3. 产品定位

功能一被正式定义为：

`一个以项目为上下文、以需求会话为单位、由 AI 推动需求到代码交付闭环、并在关键质量关口保留人工介入的研发流程平台`

其核心价值不只是让模型生成代码，而是把以下能力组织成一条可解释、可回看、可纠偏、可交付的研发主链：
- 将自然语言需求转换为结构化需求与验收标准
- 基于代码库上下文形成技术方案并完成阶段内方案校验
- 基于已确认方案产出代码、测试与评审结果
- 在关键质量关口执行人工审批
- 在运行中支持暂停、恢复、终止、回退与重试
- 输出可追溯的最终交付结果

## 4. 版本目标

功能一第一阶段的版本目标如下：
- 支持在项目上下文中创建需求会话并发起一次完整执行
- 支持从需求输入到交付结果的完整链路运行
- 支持在会话启动前选择模板并编辑允许的运行配置
- 支持需求分析、方案设计、代码生成、测试生成执行、代码评审与交付集成六个正式业务阶段
- 支持在 `Solution Design` 阶段内部执行方案校验
- 支持需求澄清、人工审批、暂停、恢复、终止、回退与重新尝试
- 支持同一需求链路在同一会话下形成多个 `PipelineRun` 执行尝试并独立回看
- 支持真实代码库上下文读取、代码修改、测试执行与交付记录生成
- 支持 `demo_delivery` 与 `git_auto_delivery` 两类交付模式
- 支持在项目级配置默认交付模式与交付通道，并在最终人工审批通过后、进入 `Delivery Integration` 前固化为交付快照
- 支持前端控制台通过统一设置弹窗完成项目级交付配置与模型提供商配置
- 支持前端控制台查看链路、补充澄清信息、执行人工审批与回看历史
- 支持已加载项目与历史会话在系统重启后继续可见，并支持在左栏重命名会话、删除会话和移除已加载项目
- 支持所有核心命令与查询通过 REST API 暴露，并通过 SSE 提供会话级实时增量更新，同时提供完整 OpenAPI 文档
- 为功能二保留可复用的后端扩展对象与接口边界

## 5. 非目标

以下内容不属于功能一第一阶段的必须范围：
- 浏览器注入、页面圈选与网页内悬浮操作
- 多租户、权限中心、组织管理与计费体系
- 独立审批中心、独立报表后台与复杂运营后台
- 任意自定义业务阶段编排器
- 语义索引驱动的大规模代码检索体系
- 云端多仓库托管与大规模分布式执行集群
- 面向用户的无限制通用平台配置中心、密钥管理系统或运行时切换面板
- 将测试运行能力作为正式执行模式暴露给用户
- 在移除项目时删除本地项目文件夹、目标仓库文件、远端仓库、远端分支、提交或代码评审请求

## 6. 用户与核心场景

功能一面向以下用户：
- `操作者`：发起需求、补充澄清信息、执行审批、控制运行状态、查看交付结果
- `观察者/管理者`：关注链路是否完整、风险是否暴露、审批是否清晰、交付是否可解释
- `平台开发者`：使用平台对自身仓库或外部本地仓库发起需求交付任务

功能一覆盖以下核心场景：
- `预设演示场景`：使用平台自身仓库完成稳定的端到端演示
- `临时需求场景`：面对小范围需求改动，完成分析、设计、实施、测试、评审与交付
- `平台自举场景`：目标仓库为平台自身，展示“用平台给平台加功能”

## 7. 核心产品原则

功能一必须遵循以下产品原则：

1. `流程优先于单轮对话`
功能一不是单次聊天生成代码工具，而是一条从需求到交付的显式研发流程。

2. `业务阶段显式化`
需求分析、方案设计、代码生成、测试生成执行、代码评审与交付集成必须作为正式业务阶段存在。

3. `运行时控制点独立于业务阶段`
澄清、审批、暂停、恢复、终止、回退与重试属于运行时控制语义，不是额外业务阶段。

4. `阶段产物显式化`
每个业务阶段都必须产出可被后续阶段读取、展示和审查的结果，而不是只保留隐式文本。

5. `澄清属于需求分析内部`
需求澄清属于 `Requirement Analysis` 阶段内部的连续对话，不单独形成审批节点。

6. `人工审批只保留在质量关口`
功能一第一阶段的人工审批只保留在 `Solution Design` 与 `Code Review` 两个质量关口。

7. `审批嵌入主链路`
人工审批是执行链路中的显式控制点，不通过独立审批系统脱离主链路运行。

8. `交付必须可追溯`
最终结果必须能够回溯到需求、方案、代码、测试和评审结论。

9. `平台运行必须可审计`
系统必须记录可用于运行排障、审计复盘、安全追踪与后续运维分析的平台级日志与审计链路。日志审计服务于平台运行治理，不替代 Narrative Feed、Inspector 或领域事件。

10. `运行前配置服务执行链路`
会话启动前允许编辑的模板、`AgentRole`、`Provider` 与自动回归配置，必须用于定义后续阶段 Agent 的实际执行行为；运行启动后，这些配置必须固化为本次执行快照。

11. `交付配置归属项目上下文`
项目级默认 `DeliveryChannel` 用于定义该项目后续新启动 run 的默认交付方式；该配置不属于模板编辑范围，且当前活动 run 的交付快照只在最终人工审批通过后、进入 `Delivery Integration` 前固化。

12. `项目与会话历史可持久、可管理`
已加载且未移除的项目、项目下未删除的会话以及这些会话的历史链路属于控制面历史。除用户显式删除会话或移除项目外，系统重启后必须继续保留这些历史并允许从前端左栏访问。会话重命名只改变会话展示名，不改变执行历史、审批记录、产物或 run 归属；会话删除和项目移除属于显式历史管理动作，不等同于 `terminate`、`rollback` 或 `retry`。会话删除和项目移除的产品层结果是从常规项目/会话历史、回看入口和产品查询投影中删除；跨库存储的物理删除、软删除或归档策略由后端规格统一固定，并必须保留审计边界要求保留的安全审计事实。

13. `环境变量与配置设置分层`
环境变量只服务系统启动、本地运行、前后端连接、工作区隔离、日志审计落点和凭据引用解析。系统允许通过统一配置管理维护可热重载的运行设置和业务配置，但 Provider、DeliveryChannel、模板运行配置不得下沉为环境变量。会影响执行语义的模板、Provider、模型绑定、运行上限和执行图配置必须在 run 启动时固化为快照；交付通道配置必须在最终人工审批通过后、进入 `Delivery Integration` 前固化为快照。

14. `正式运行与测试运行能力分离`
功能一的正式 Agent 编排路径由 LangGraph runtime 承担。`deterministic test runtime` 只服务测试、前端联调、端到端验证和回归验证，不属于用户可选择的运行模式，不进入前端设置，不作为平台环境变量暴露。

15. `API 契约与文档同等重要`
所有核心命令与查询必须通过 REST API 暴露，会话级实时增量更新必须通过 SSE 提供，并提供与运行接口一致的 OpenAPI 文档。

16. `受控工具最小面`
Agent 运行时必须通过受控工具完成仓库读取、文件编辑、路径匹配、内容搜索、命令执行与变更追踪。功能一 V1 的工作区工具设计参考 Claude Code 的工具工作机制，但正式工具契约名使用小写机器可读 ID，并固定为 `bash`、`read_file`、`edit_file`、`write_file`、`glob`、`grep`。diff 生成、工作区状态采集和 `ChangeSet` 构建属于后端服务侧能力，不作为独立模型可调用工具；`bash` 是工具契约名，不要求底层必须是 Unix bash；`grep` 必须具备受控的本地 `ripgrep` / `rg` 可用性校验。工具不得绕过阶段权限、工作区边界、审计记录和变更集构建。

17. `为功能二预留复用边界`
功能一在后端对象与接口层必须为 `ChangeSet`、`ContextReference`、`PreviewTarget`、`DeliveryRecord` 等后续能力保留扩展空间。

## 8. 会话、模板与运行边界

功能一的产品运行边界如下：

1. 所有执行都必须挂载在一个 `Project` 之下，且系统启动后默认存在一个绑定平台仓库自身路径的项目。
2. 已加载且未移除的 `Project` 必须在系统重启后继续保留在项目历史中，用户不需要重新导入。
3. 每个 `Project` 都必须具备一个项目级默认 `DeliveryChannel`；前端必须能够在统一设置弹窗中查看其状态并完成配置。
4. `DeliveryChannel` 属于项目级交付配置，不属于模板编辑范围；当前活动 run 的交付快照只在最终人工审批通过后、进入 `Delivery Integration` 前固化。
5. 用户允许移除已加载的非默认 `Project`；移除后该项目必须从项目历史中消失，并一并从常规产品查询与回看入口中删除该项目下用于产品回看的 `Session` 记录和运行历史。
6. 移除 `Project` 不得删除本地项目文件夹、目标仓库文件、远端仓库、远端分支、提交或代码评审请求；平台安全审计记录按审计边界保留，不作为可继续打开的项目或会话历史。
7. 存在活动 run 的 `Project` 不允许被移除；必须先让相关活动 run 进入终态。
8. 每次需求推进都必须归属到一个 `Session`。
9. 每个 `Session` 承载同一个需求链路，不承载多个彼此独立的需求。
10. 新建会话时必须绑定一个 `PipelineTemplate`。
11. 同一 `Session` 的首条需求消息必须自动启动首个 `PipelineRun`。
12. 同一 `Session` 可因重新尝试或运维重启产生多个 `PipelineRun`，但这些运行只表示同一需求链路的不同执行尝试；暂停后的恢复属于原 run 的继续执行，不创建新的 `PipelineRun`。
13. 新的 `PipelineRun` 只允许在前一个活动 run 处于 `failed` 或 `terminated` 后创建；`completed` 表示该会话链路已经完成，不再在同一会话中开启新的 run。
14. 新的 `PipelineRun` 必须从完整链路起点重新开始，而不是从上一个 run 的中间业务阶段断点续跑。
15. 不同 `PipelineRun` 之间不得共享未交付的工作区改动；新 run 必须基于干净基线启动。
16. 同一 `Session` 的多个 `PipelineRun` 必须在同一会话页面中按时间顺序串联展示，而不是拆成多个彼此隔离的会话页面。
17. 不同 `PipelineRun` 之间必须有明确分界，不得在视觉或叙事上伪装成同一次连续运行。
18. 当前活动 run 负责驱动页面底部统一 `Composer` 与生命周期控制；历史 run 只用于回看，不承载新的输入或审批操作。
19. 用户允许在左栏重命名 `Session`；重命名只改变会话展示名，不改变原始需求、运行历史、审批记录、产物、交付记录、事件归属或 `PipelineRun` 归属。
20. 用户允许在左栏删除没有活动 run 的 `Session`；删除后该会话不得继续出现在会话历史中，也不得继续作为可回看的历史会话打开。
21. 存在活动 run 的 `Session` 不允许被删除；删除会话不得替代 `terminate`，也不得隐式暂停、恢复、回退或重试当前 run。
22. 用户在同一会话中完成：
- 原始需求输入
- 需求澄清补充
- 方案审批
- 代码评审审批
- 暂停、恢复、终止
- 交付结果查看

补充约束如下：
- 在首条需求消息提交前，`Session` 必须保持 `draft`，且 `current_run_id = null`；系统不存在“等待首条需求输入的活动 run”状态。
- 本节所称活动 run 指尚未进入 `completed`、`failed` 或 `terminated` 的 `PipelineRun`；`draft` 会话没有活动 run。
- `completed` run 以顶层 `delivery_result` 条目作为尾部收束；`failed` 与 `terminated` run 必须在各自 run 尾部追加顶层 `system_status` 终态条目。
- `重新尝试` 只挂载在当前活动 run 的 `failed` 或 `terminated` 尾部 `system_status` 终态条目上，不挂载在历史 run 或 `completed` run 上。
- 会话删除与项目移除只能删除产品历史可见性、常规查询入口和对应查询投影，不得删除平台审计边界要求保留的安全审计事实。

### 8.1 环境变量与配置设置边界

功能一必须区分 `环境变量`、`可热重载配置设置` 与 `运行快照`。

环境变量只允许覆盖系统启动前必须确定的基础落点、连接信息和密钥引用来源，至少包括：
- 前端访问后端服务所需的 API base URL
- 本地前后端分端运行所需的 CORS 来源
- 系统本地数据根目录或由系统默认派生的数据落点
- 默认 Project 根路径
- 隔离工作区根路径
- 平台级日志与审计记录的本地落点
- `credential_ref` 与 `api_key_ref` 指向平台密钥来源的引用解析规则

可热重载配置设置用于维护不需要重启服务即可更新的运行策略和业务配置。功能一 V1 允许统一配置管理能力，但该能力必须满足以下约束：
- Provider、DeliveryChannel、模板运行配置属于业务配置，只能通过对应 API、设置弹窗、模板编辑能力或后端管理接口更新
- 前端统一设置弹窗只管理项目级 `DeliveryChannel` 与 Provider 等 V1 用户可见业务配置；模板运行配置由模板编辑能力管理；二者均不管理环境变量
- Agent 运行上限、Provider 调用策略、上下文裁剪限制、日志裁剪限制、日志保留策略和日志查询分页上限属于平台运行设置，可以热重载，但不得改变已经启动 run 的执行语义
- 会影响执行语义的模板、Provider、模型绑定、运行上限和执行图配置必须在 run 启动时固化为模板快照、Provider 与模型绑定快照、GraphDefinition 和运行上限快照；已启动 run 不读取最新配置来改变当前语义
- 交付通道配置必须在最终人工审批通过后、进入 `Delivery Integration` 前固化为交付通道快照；已固化交付通道快照不读取最新配置来改变当前交付语义
- 诊断类配置可以即时生效，但不得改变领域事件、查询投影、审批状态、交付结果或 Narrative Feed 条目语义
- 所有用户或管理接口可写入的运行上限必须受平台硬上限约束，超过硬上限时必须拒绝保存或拒绝启动 run

环境变量与配置设置必须共同满足以下约束：
- 后端规格中定义的多 SQLite 职责库属于系统内部持久化边界，其文件路径由系统默认从本地数据根目录派生；不得要求用户或前端设置逐个配置这些数据库路径
- 测试环境需要替换数据库或目录落点时，必须通过测试 fixture、settings override 或等价测试机制完成，不进入正式产品配置面
- 环境变量不得承载 Provider 的模型参数、`system_prompt`、模板角色绑定、项目交付仓库、目标分支、代码评审请求类型、交付模式、Agent 运行上限或日志策略
- `credential_ref` 与 `api_key_ref` 只能在前端和业务对象中表现为引用，不得暴露真实密钥内容

### 8.2 正式运行与 deterministic test runtime 边界

功能一对用户只呈现一条正式研发主链和一套业务阶段语义。正式运行必须使用真实 Agent 编排路径，后端执行内核的正式实现由 LangGraph runtime 承担。

`deterministic test runtime` 是工程验证能力，必须满足以下约束：
- 只由测试夹具、前端联调入口、端到端测试或回归测试调用
- 只用于稳定验证状态机、投影、SSE、人工审批、工作区边界、交付记录和前端流程
- 不作为用户可选择的运行模式
- 不出现在前端设置、项目设置、模板配置或平台环境变量中
- 不改变正式业务阶段、运行时控制点、领域事件、查询投影、Inspector 或 DeliveryRecord 的语义
- 与正式 LangGraph runtime 共享后端定义的稳定执行契约，但不得把内部测试实现细节泄漏到产品 API 或前端展示
- 可以随工程演进被重构或替换，但在 V1 中作为稳定回归验证能力保留

## 9. 功能一业务阶段边界

功能一第一阶段包含以下正式业务阶段：

1. `Requirement Analysis`
2. `Solution Design`
3. `Code Generation`
4. `Test Generation & Execution`
5. `Code Review`
6. `Delivery Integration`

以上六个阶段共同构成功能一第一阶段的正式研发主链。

### 9.1 Requirement Analysis

职责：
- 读取用户原始需求与必要上下文
- 产出结构化需求、验收标准、约束与待确认事项
- 当信息不足时，在同一阶段内部发起澄清提问

边界：
- 需求澄清属于本阶段内部行为
- 本阶段不创建 `Approve / Reject` 审批动作
- 本阶段完成后，进入 `Solution Design`

### 9.2 Solution Design

职责：
- 基于结构化需求、澄清结论与代码库上下文形成技术方案
- 给出影响范围、关键设计决策、文件范围与实现路径
- 在本阶段内部完成方案校验

边界：
- 方案设计结果必须先经过本阶段内部的方案校验子步骤
- 方案设计本身不是人工审批动作，但其结果必须进入方案审批检查点

### 9.3 Solution Design 内部方案校验子步骤

职责：
- 独立校验技术方案是否偏离需求、遗漏影响范围、缺少测试策略或引入过高风险

边界：
- 该校验子步骤属于 `Solution Design` 阶段内部，不形成独立正式业务阶段
- 校验通过后，进入方案人工审批
- 校验不通过时，返回 `Solution Design` 内部继续修正

### 9.4 Code Generation

职责：
- 按已通过校验且经人工批准的方案生成或修改代码
- 输出变更说明与代码变更集合

边界：
- 本阶段不触发人工审批
- 本阶段完成后进入测试生成与执行

### 9.5 Test Generation & Execution

职责：
- 生成测试或补充测试
- 执行测试并输出测试结果与测试缺口分析

边界：
- 本阶段不触发人工审批
- 本阶段完成后进入 `Code Review`

### 9.6 Code Review

职责：
- 审查代码变更、测试结果、测试缺口与方案一致性
- 输出评审报告、问题列表、回退目标判断与风险判断

边界：
- 评审阶段允许先进行自动回归与返修循环
- 自动回归结束后，评审结果进入代码评审人工审批

### 9.7 Delivery Integration

职责：
- 整理已通过审批的变更、测试结论、评审结论与交付信息
- 读取本次 run 已固化的项目级交付快照
- 执行交付步骤并沉淀交付过程记录
- 根据交付模式生成对应的交付结果
- 生成最终交付记录

边界：
- `Delivery Integration` 不属于人工审批阶段
- `Delivery Integration` 表达的是交付执行过程；本阶段成功完成后，系统再以顶层 `delivery_result` 条目输出最终交付结果
- `delivery_result` 不替代 `Delivery Integration` 阶段
- 本阶段必须支持 `demo_delivery` 与 `git_auto_delivery` 两类交付模式
- `demo_delivery` 只生成展示型交付结果，不执行真实 Git 写操作
- `git_auto_delivery` 执行真实分支、提交与 MR/PR 交付动作
- 交付配置与凭据的阻塞性检查必须在进入本阶段前完成；本阶段只负责读取已固化的交付快照并输出结果展示
- 若本阶段失败，不生成 `delivery_result`，并由 run 尾部 `system_status` 终态条目表达失败结果
- 本阶段完成后，会话进入交付完成状态

## 10. 运行时控制点与人工介入边界

功能一必须严格区分 `业务阶段` 与 `运行时控制点`。

运行时控制点用于表达主链执行中的中断、等待、人工介入与重新推进，不替代正式业务阶段。

### 10.1 需求澄清边界

需求澄清必须满足以下规则：
- 只发生在 `Requirement Analysis` 阶段内部
- 表现为系统提问与用户补充信息的连续对话
- 使用会话统一输入入口，不使用 `Approve / Reject`
- 只有在明确等待用户补充时，当前 run 才进入 `waiting_clarification`
- 用户补充提交后，当前 run 必须恢复执行并继续同一个 `Requirement Analysis` 结点分析；若仍需继续澄清，可在同一结点内再次提出问题
- 不创建独立审批页面，不创建独立审批卡，不创建独立审批工作流

### 10.2 人工审批边界

功能一第一阶段只保留两个正式人工审批检查点：

1. `方案设计审批`
- 位于 `Solution Design` 阶段内部方案校验通过之后
- 审批对象为方案说明、影响范围、关键设计决策与相关风险
- `Reject` 后回到 `Solution Design`

2. `代码评审审批`
- 位于 `Code Review` 产出稳定评审结果之后
- 审批对象为代码变更内容、测试结果、测试缺口分析与评审结论
- 当当前项目 `delivery_mode = git_auto_delivery` 时，系统必须在该审批阶段检查交付配置与凭据是否就绪；未就绪时阻塞审批通过并提示用户完成配置
- 当当前项目 `delivery_mode = demo_delivery` 时，不因交付配置或凭据缺失额外阻塞该审批
- `Reject` 后默认回到 `Code Generation`

除以上两处外，功能一不再定义其他人工审批检查点。

### 10.3 暂停、恢复与终止边界

运行控制必须满足以下规则：
- `pause`、`resume` 与 `terminate` 属于主链运行控制动作，不是新的业务阶段
- `pause` 可发生在运行中的正式业务阶段、等待澄清状态或等待审批状态
- 若 `pause` 发生在 `waiting_approval`，该审批必须转为不可提交态；用户只有在 `resume` 之后，才能继续处理同一个待审批检查点
- `resume` 只允许继续同一个 `PipelineRun`
- `terminate` 只终止当前活动 run，不删除已产生的历史记录、审批记录与产物记录

### 10.4 回退与重新尝试边界

运行控制必须满足以下规则：
- `rollback` 属于当前活动 run 内的控制动作，用于在保留既有执行历史、产物、问题记录与审批记录的前提下，把链路退回到既定业务阶段并开始订正性重执行
- `retry` 在当前 run 内用于自动回归或阶段内再次尝试
- `重新尝试` 是在当前活动 run 已 `failed` 或 `terminated` 后创建新的 `PipelineRun`
- 回退与重试不构成新的正式业务阶段

- `structured_requirement`

- `clarification_record`
- `solution_design`
- `solution_validation_report`
- `code_changeset`
- `test_execution_result`

- 可在历史会话中回看

## 12. 功能一验收标准

功能一第一阶段至少满足以下产品级验收标准：

1. 支持在项目上下文中创建需求会话并启动一次完整执行。
2. 已加载且未移除的项目和未删除的历史会话在系统重启后仍可从前端左栏继续访问，不需要重新导入。
3. 用户能够从左栏重命名会话展示名，重命名后历史链路、审批记录、产物和 run 归属保持不变。
4. 用户能够从左栏删除没有活动 run 的会话；删除后该会话不再出现在历史会话列表中，也不能再作为历史会话打开。
5. 用户能够移除已加载的非默认项目；移除时该项目下用于产品回看的会话历史必须从常规产品查询与回看入口中消失，且不删除本地项目文件夹、目标仓库文件、远端仓库、远端分支、提交或代码评审请求。
6. 支持从需求输入到交付结果的完整链路运行。
7. 支持在会话启动前选择模板并编辑允许的运行配置；V1 允许字段固定为角色槽位绑定、`system_prompt`、`Provider` 绑定与自动回归配置，不包含 `AgentRole` 名称修改；运行启动后配置固化为快照。
8. 同一需求链路支持在同一会话下形成多个 `PipelineRun` 执行尝试，并通过独立 `run` 切换机制查看。
9. 需求澄清直接发生在 `Requirement Analysis` 阶段内部，不使用审批动作。
10. 人工审批只保留在 `Solution Design` 与 `Code Review` 两个检查点。
11. 人工审批必须支持 `Approve` 与 `Reject`，并要求拒绝理由进入后续上下文。
12. 用户可以在单一控制台中完成输入、澄清补充、审批、暂停恢复、终止与结果查看。
13. 阶段产物与控制产物可被查询、展示、回看与追溯。
14. 最终交付结果能够关联需求、方案、代码、测试与评审结论。
15. 系统支持 `demo_delivery` 与 `git_auto_delivery` 两类交付模式，并能对平台自身仓库完成至少一次真实端到端演示。
16. 支持在项目级配置默认 `DeliveryChannel`，并在最终人工审批通过后、进入 `Delivery Integration` 前固化为交付快照；正式快照字段与时间字段命名以后端规格为准，必须覆盖 `delivery_mode`，并在 `delivery_mode = git_auto_delivery` 时覆盖托管平台类型、仓库标识、默认分支、代码评审请求类型、`credential_ref`、`credential_status` 与配置就绪状态。
17. 所有核心命令与查询通过 REST API 暴露，会话级实时增量更新通过 SSE 提供，并提供完整 OpenAPI 文档。
18. 系统具备平台级日志审计基础能力，能通过本地日志文件、轻量索引、审计记录与关联标识支撑运行排障、审计复盘、安全追踪与后续运维分析，且不把日志审计替代为 Narrative Feed、Inspector 或领域事件。
19. 系统具备基于 `ToolProtocol` 的受控工作区工具最小面，正式工具契约名为 `bash`、`read_file`、`edit_file`、`write_file`、`glob`、`grep`，并能通过审计、工作区隔离和 `ChangeSet` 构建记录工具副作用。
20. 系统具备环境变量与可热重载配置设置的分层边界，能够支撑本地启动、前后端连接、数据落点、工作区隔离、日志审计落点、凭据引用解析、运行上限管理、Provider 管理、交付配置管理和日志策略管理，且不把业务配置放入环境变量矩阵。
21. 正式运行使用 LangGraph runtime；`deterministic test runtime` 只用于测试、联调、端到端验证和回归验证，不作为用户可选择的运行模式或平台配置项。
22. 功能二所需的核心扩展对象在后端规格层具备明确预留边界。

## 13. 风险与约束

功能一第一阶段必须控制以下风险：
- 不得把需求澄清重新建模为人工审批
- 不得把运行时控制点误写为正式业务阶段
- 不得把功能一退化为单轮聊天生成工具
- 不得把前端主链路拆成多个互相跳转的审批或澄清页面
- 不得让后续阶段绕过结构化需求、澄清结论与方案结论直接推进
- 不得让模型绕过 `ToolProtocol`、阶段 `allowed_tools`、工作区边界或审计策略直接读取文件、编辑文件、搜索内容、执行命令或构建变更集
- 不得在交付阶段重新引入新的人工审批关口
- 不得把日志审计能力弱化为前端展示日志，或把日志记录反向作为产品状态、阶段产物与交付结论的真源
- 不得在功能一 V1 中把完整日志检索台、日志全文搜索或运维可视化面板作为主线交付前提
- 不得把环境变量扩展为业务配置中心，不得把 Provider、DeliveryChannel、模板运行配置、Agent 运行上限或日志策略下沉为环境变量
- 不得让可热重载配置设置改变已经启动 run 的模板快照、Provider 与模型绑定快照、运行上限快照或交付通道快照
- 不得要求用户或前端设置逐个配置系统内部多 SQLite 职责库路径
- 不得把 `deterministic test runtime` 暴露为用户可选运行模式、前端设置项或平台环境变量
- 不得为长期扩展性引入与当前版本价值不匹配的复杂基础设施

## 14. 结论

功能一第一阶段应被正式理解为：

`一个在项目上下文中运行、以需求会话为载体、由 AI 推动从需求分析到交付集成、并在方案设计与代码评审两个质量关口保留人工介入的研发流程平台`

功能一的产品边界已经明确为：
- 六个正式业务阶段构成固定研发主链
- 需求澄清属于 `Requirement Analysis` 内部连续对话
- 人工审批只保留在 `Solution Design` 与 `Code Review`
- 暂停、恢复、终止、回退与重试属于运行时控制点
- 交付集成负责输出结果，不承担额外审批职责
